# DADS参照にもとづくアプリケーションUIデザインルール

この文書は、デジタル庁デザインシステム β版の公開ページだけを参照し、ウェブアプリケーション、オンラインサービス、業務システムで使いやすいように内容を一つの文書へ並べ直したものです。  
2026年4月11日時点で参照できた記述をもとにしています。  
DADS に記載のない独自ルールは加えず、ガイドラインが準備中の項目は、その状態だけを明記します。

## 1. この文書の使い方

DADS は、特定の組織のブランドをそのまま表現するための固定デザインではなく、個別サービスごとに調整して使うための土台です。したがって、この文書は完成済みの見た目集ではなく、個別サービスのスタイルガイドを起こすための基準として扱います。

最初に決める対象は、少なくとも次のとおりです。

- カラースキーム
- 文字サイズと行間の対応
- 余白の段階
- 角の形状の段階
- エレベーションの段階
- アイコンの役割
- ボタンの優先度と並び順
- ブレークポイントとレイアウトの切り替え方

## 2. 最上位の原則

### 2.1 アクセシビリティを前提にする

DADS は、色とコントラスト、文字サイズ、キーボード操作とフォーカス表示、リンクやボタンの押しやすさ、操作に応じた見え方、動きの扱い、内容量の増減に耐える構造に特に配慮して組まれています。  
ただし、実際のサービスで使う文章、画像、手続き、入力導線そのもののアクセシビリティは、各サービス側で別に確保しなければなりません。

### 2.2 サービス内で一貫性を保つ

色、リンク状態、ボタンの優先度、余白、角丸、エレベーション、アイコンの意味づけは、画面ごとに変えず、サービス全体で揃えます。  
DADS でも、色や状態表現、ボタン配置、スタイルの一貫性を繰り返し求めています。

### 2.3 重要な情報は隠さない

アコーディオンやディスクロージャーは、表示を小さくまとめるためだけに使ってはいけません。  
多くの利用者にとって重要な情報は、最初から見える場所に置きます。

### 2.4 画面単体ではなく、流れの中で判断する

ボタンや入力部品は、その部品だけの見た目で決めません。  
その画面で何を最優先に完了させたいか、利用者にどの選択肢を先に見せたいか、前後の流れの中で決めます。

## 3. 基本デザイン

### 3.1 カラー

カラーは、ブランドを見せるためだけではなく、部品の意味や役割を揃えて伝えるために使います。  
サービスでは、少なくとも次の系統を決めます。

| 系統 | 主な役割 | 守ること |
|---|---|---|
| プライマリーカラー | ブランドを最も強く表す要素、主要な CTA、アクティブ状態 | 主要背景とのコントラスト比を 4.5:1 以上にする |
| セカンダリーカラー | 主役ではない動線、選択肢、状態表示 | 非テキストでは 3:1 以上、文字色として使うときは 4.5:1 以上 |
| ターシャリーカラー | セカンダリーと対になる補助色 | 非テキストでは 3:1 以上、文字色では 4.5:1 以上 |
| バックグラウンドカラー | 白や黒以外の背景 | その背景の上に置く文字と非文字のコントラストを都度見直す |
| 共通カラー | 本文、枠線、区切り線、背景の基本色 | 文字は常に 4.5:1 以上、枠線や区切り線は 3:1 以上 |
| アクセントカラー | 視線を集める強調 | 多用しない。機能を持つ部品に使う場合は文字 4.5:1 以上、非テキスト 3:1 以上 |
| セマンティックカラー | 成功、失敗、警告などの意味 | 成功は緑、エラーは赤、警告は黄または橙の意味を保つ。明度や彩度は調整してよいが、意味を逆転させない |

#### リンク色と状態色

リンク色は、未訪問と訪問済みを見分けやすくしながら、文字コントラスト 4.5:1 以上を満たします。  
DADS は、未訪問を青、訪問済みを紫系とする慣習を基本にしつつ、色覚多様性のために明度差を意識することを求めています。  
また、ボタンなど状態を持つ UI は、デフォルト、ホバー、アクティブの見え方をサービス内で揃えます。

### 3.2 タイポグラフィ

文字設計は、読みやすさと見た目の印象を決める土台です。  
本文や通常の UI では、原則として 16 CSS px 以上を基準にします。14 CSS px は、領域制約が強い場面だけに限り、14 未満は原則使いません。

本文の行間は、少なくとも文字サイズの 1.5 倍を保ちます。  
ただし、見出しはやや狭くてもよく、管理画面や業務システムのように情報量を優先する画面では、さらに狭い行間を使うことがあります。

DADS の文字スタイルは次のように整理できます。

| 文字スタイル | 主な使い方 | ルールの要点 |
|---|---|---|
| 強い見出しや冒頭メッセージ用の Display | 冒頭コピー、強い見出し | 行高 140%、字間 0 |
| 見出しと本文の基本形である Standard | 一般的な見出しと本文 | 16 CSS px 以上を基本にし、本文との差が明確になるよう見出しを定義する |
| 情報量を優先する Dense | 管理画面、業務システム | 行高 120% または 130% を中心に使う |
| 1 行前提の UI テキスト用の Oneline | ボタンなど 1 行で収める UI | 行高 100%、字間 2% |
| コード表示用の Mono | コードや等幅表示 | 等幅フォント、行高 150%、字間 0 |

太さは、主に通常の太さである 400 と、太字である 700 を使います。  
文字サイズの大きさと見出しレベルは別に定義し、サービスのスタイルガイドで対応関係を決めます。

### 3.3 アイコン

アイコンは一つの種類で済ませず、文書構造の中でどこに置くかに応じて役割を分けます。  
DADS では次の 4 種類を定義しています。

- 先頭外側に置くフロントアイコン
- 行内先頭に置くリードアイコン
- 行内末尾に置くテールアイコン
- 末尾外側に置くエンドアイコン

同じ見た目のアイコンでも、役割が違うなら使い分けます。

### 3.4 レイアウト

レイアウトは、グリッドを基準に決めます。  
グリッドシステムの基本要素は、マージン、カラム、ガターです。

- カラム幅は本文文字サイズの整数倍で決める
- ガターは原則として本文文字サイズの 2 倍以上を確保する
- 左サイドにメニューやナビゲーションを置くシステムでは、その右端からグリッドを組む
- 12 カラムは一般的な基準として使いやすい

DADS は、まず 1 つのブレークポイントを置く考え方を採ります。

| 区分 | ビューポート |
|---|---|
| モバイル・タブレット | 768px 未満 |
| デスクトップ | 768px 以上 |

画面構成は、1 カラム、2 カラム、3 カラム、4 カラム、カラムオフセットを基本候補にします。  
2 カラムでは 9:3、3:9、8:4、4:8、6:6 のような割り方、3 カラムでは 3:6:3 または 4:4:4、4 カラムでは 3:3:3:3 が例示されています。

### 3.5 余白

余白は、読ませ方と理解しやすさを作る重要な要素です。  
要素内の余白であるパディングと、要素間の余白であるマージンを使い分けます。

余白設計では、次を守ります。

- 同種の要素には同じ間隔を使う
- 規則のある段階を作る
- 階層が高い情報ほど大きな余白を与える
- 詰め込みすぎない
- 画面幅が変わっても見た目のバランスを崩さない

基準単位としては 8 CSS px が一般的な候補とされ、スタイルガイドで使う段階は 3 から 5 程度が多いとされています。  
DADS の作例では、8、24、64 CSS px の 3 段階が示されています。

### 3.6 リンクテキスト

リンクは、通常の本文と明確に見分けられなければなりません。  
色だけではなく、下線など別の手掛かりも組み合わせます。

- 外部リンクや新規タブを開くリンクでは、末尾アイコンを付けてもよい
- その末尾アイコンは下線を引かなくてもよいが、クリック可能範囲には含める
- 未訪問、訪問済み、フォーカス、ホバー、アクティブの状態を定義する
- ホバーで文字サイズや太さを変えてレイアウトががたつく表現は、他要素へ影響しない場合だけに限る
- PDF やダウンロードファイルへのリンクは、ファイル形式や容量を文中に明記する
- 新規タブやファイル種別を示すアイコンには代替テキストを付ける

### 3.7 角の形状

角の形状は、部品の理解や見つけやすさに影響します。  
DADS は、角丸なし、スモール、ミディアム、ラージ、フルの 5 段階を基本の目安にしています。

ただし、同じ半径をそのまま当てると、小さい部品ほど丸く見え、大きい部品ほど角張って見えます。  
そのため、角丸値は部品ごとに調整し、見た目の一貫性を保ちます。  
必要であれば、四隅すべてではなく、特定の角だけに角丸を与えてもかまいません。

### 3.8 エレベーション

エレベーションは、画面上の高さ関係を表すための設計です。  
デフォルトは高さレベル 0 で、重なって表示する部品はそれより高いレベルに置きます。

- 表現はドロップシャドウとオーバーレイシェードで行う
- ドロップシャドウだけでは 3:1 のコントラストを満たせないので、浮いている要素には背景との 3:1 を満たすボーダーを持たせる
- ホバーで高さを変える場合は、少なくとも 1 レベル以上上げる
- オーバーレイ部品は、直下にある最も高い要素より少なくとも 2 段階上に置く
- オーバーレイシェードは、モーダル時に親画面への視線と操作を遮断するために使う
- オーバーレイシェード自体は原則クリック可能にしない。閉じるなどの解除操作はオーバーレイ内に置く
- エレベーションは多用しない。高低差をつけると理解しやすくなる場面だけで使う

## 4. フォーム設計の共通ルール

フォームは、入力しやすさと誤入力からの立ち直りやすさを優先します。  
見た目を整えるために、利用者へ余計な負担をかけてはいけません。

| 項目 | ルール |
|---|---|
| ラベル | 項目ラベルは必須。必要に応じて `※必須` または `※任意` を付ける |
| サポートテキスト | 入力条件や例を、簡潔かつ具体的に書く。プレースホルダーテキストで代用しない |
| エラーテキスト | 何が悪いか、どう直すかを具体的に書く。先頭に `＊` を付ける |
| 並べ方 | ラベルとサポートテキスト、入力欄とエラーは上下に積むのを基本にする。入力項目が多い場合は横並びでもよいが、面を揃えて余白を一定にする |
| 穴埋め文章 | 文章の途中に入力欄を埋め込まず、項目ごとに独立した入力欄にする |
| 文字種制限 | 原則として文字種を限定しない。制限が必要な場合は、自動変換支援か送信時の明確なエラーで対応する |
| プレースホルダー | 原則使わない。入力中に消え、比較もしにくく、支援技術でも不利になるため |
| disabled | 原則使わない。押せない理由が伝わらず、操作を止めやすい |
| readonly | 原則避ける。編集できない値は地の文で見せるほうが分かりやすい |
| 最大長 | `maxlength` 属性を使わない。上限はサポートテキスト等で示し、超過時はバリデーション結果を返す |
| コピー・ペースト | 禁止しない |
| メールアドレス入力 | `@` の前後で分割しない |
| 入力時の変化 | 値変更だけで突然ダイアログを出したり画面遷移したりしない。必要な変化はボタンやリンクをきっかけにする |
| エラー通知 | エラーテキストは静的に表示し、読み上げ強制のために `aria-live` や `role="alert"` を使わない |

## 5. 入力部品の選び方と使い方

### 5.1 インプットテキスト

インプットテキストは、氏名や電話番号など、1 行以内の文字入力に使います。  
構成要素は、入力欄、項目ラベル、要否ラベル、サポートテキスト、エラーテキストです。

- サイズは Large 56px、Medium 48px、Small 36px の高さが示されている
- 横幅は均等にせず、入力内容に見合う長さにする
- 多くの場合、縦積み配置が推奨される。モバイルでも崩れにくいため
- サポートテキストは、条件や例を先に示す
- エラーは、曖昧な表現ではなく、原因と訂正方法を具体的に書く
- 文字種をどうしても制限する場合は、送信時または確認画面表示時に自動変換するか、そのタイミングでエラーを返す
- 入力直後やフォーカスアウト時の自動変換や即時エラーは、混乱や煩わしさを招きやすいので避ける

### 5.2 テキストエリア

テキストエリアは自由記述に使います。  
ただし、自由記述は利用者の負担が大きいため、先に質問を分けて、必要な人だけに書いてもらう形を優先します。

- ラベルは端的に書く
- 文字数制限がある場合は、ラベルの中で上限を具体的に示す
- おおむね 20 文字を超える上限、またはスクロールが必要になる上限なら、文字数カウンターを常時表示する
- 上限を超えたときは、超過したことが分かるようにする

### 5.3 単一選択と複数選択

選択肢の見せ方は、件数と選択数で決めます。

| ケース | 使う部品 | ルール |
|---|---|---|
| 単一選択で、選択肢が 5 件以下 | ラジオボタン | 全体が見えるので選びやすい |
| 単一選択で、選択肢が長い、または 6 件以上 | セレクトボックス | 事前の質問で候補数を減らせないか検討する |
| 複数選択 | チェックボックス | チェックボックスはテキストの左側に置く |

ラジオボタンは、一度選ぶと無選択に戻せません。  
そのため、任意回答では、あてはまるものはないのような選択肢を用意します。

セレクトボックスのオプションリストは、独自の見た目を規定せず、OS 標準の UI を採用します。

### 5.4 日付入力

日付入力は、年月日を直接入れる欄とカレンダーを組み合わせた部品です。  
見た目は、年月日を一続きに見せる Consolidated タイプと、年・月・日を別欄に分ける Separated タイプから選びます。  
要件に応じて、年を省く、日を省く構成も可能です。

#### カレンダーを置いてよい場面

カレンダーは、前後の日付を見比べながら近い期間の中から選ぶ場面に向きます。  
たとえば、イベント日、宿泊期間、予定入力です。

#### カレンダーを置かない場面

既知の日付を入力させる場面では、直接入力のほうが適しています。  
たとえば、生年月日、経歴年月、クレジットカード有効期限です。  
このような入力では、カレンダーボタンを置きません。

#### 日付入力の振る舞い

- 要否ラベルは `※必須` または `※任意` を使う
- サポートテキストで、西暦であることと具体例を示す
- プレースホルダーテキストは使わない
- 年月日を入力しただけで自動的に次欄へフォーカスを移さない
- IME を強制制御しない
- 全角数字は半角数字へ正規化する
- ゼロ埋めされていない数字は正規化する
- `disabled` を付けない

Consolidated タイプでは、矢印キーで年・月・日フィールド間を移動できるようにする実装が推奨されています。  
なお、和暦対応の日付ピッカーは、参照時点の DADS では未提供です。

### 5.5 ファイルアップロード

ファイルアップロードは、単体または複数ファイルを送るための部品です。  
通常時は、ファイル選択ボタン、必要に応じたドロップ領域拡大チェックボックス、ドロップエリア、ラベル、要否ラベル、サポートテキスト、ファイル未選択メッセージで構成します。

- ファイル選択ボタンとドロップ領域拡大チェックボックスの間は、少なくとも 56 CSS px 空ける
- ドロップエリアは、または、このエリア内にドラッグ＆ドロップという補助文を伴う
- 選択後は、選択ファイルの要約、個別ファイル、エラーを表示できるようにする
- ドラッグ操作に不慣れな利用者のため、ドロップ範囲をブラウザ全体へ広げるかどうかを利用者が選べる構造にする

## 6. ボタン

ボタンは、実行または遷移のきっかけです。  
DADS は、見た目よりも、優先度と並び順の一貫性を重視しています。

### 6.1 サイズと押しやすさ

| サイズ | 最小幅 | 高さ |
|---|---:|---:|
| Large | 136px | 56px |
| Medium | 96px | 48px |
| Small | 80px | 36px |
| X-Small | 72px | 28px |

- 幅はラベル長に合わせて伸ばす
- ボタンの押下対象領域は 44 CSS px 以上を確保する
- Small と X-Small は高さが 44px に満たないため、上下余白で対象領域を補う

### 6.2 優先度によるスタイル分け

DADS の基本スタイルは 3 種類です。

| スタイル | 役割 | 典型的な意味づけ |
|---|---|---|
| 塗りボタン | 最も強い行動 | プライマリー |
| アウトラインボタン | 次点の選択肢 | セカンダリー |
| テキストボタン | 中断、中止、補助 | ターシャリー |

次のルールで使います。

- プライマリーは 1 画面 1 個を原則とする
- セカンダリーは同じ文脈で同時に 3 個までを原則とする
- ターシャリーは 1 画面 1 個を原則とする
- 色の違いだけで階層を表してはいけない。必要ならボーダー差なども加える
- 塗りボタンが強すぎる場面では、アウトラインとテキストの組み合わせで階層を作ってよい

### 6.3 並び順と配置

DADS は、左から右へ読む言語を前提に、進む操作を右、中止や戻る操作を左に置く考え方を採ります。

#### デスクトップ

- `OK`、`完了`、`次へ` は右寄せ
- `キャンセル`、`中止`、`戻る` は左寄せ
- 右側のボタン群では、右端ほど重要度を高くする

#### モバイル

- デスクトップと同じ意味順を保ったまま、画面幅に合わせて縦積みにする
- 2 種類の縦積みでは、上にプライマリー、下にセカンダリー
- 3 種類の縦積みでは、セカンダリー群、プライマリー、ターシャリーの順が基本例として示されている
- ただし、モバイル側だけで別順序を全面禁止しているわけではなく、サービス内で一貫させることが重要

#### 特別な場面

- ボタンが 1 個だけなら、通常はプライマリーを使う
- フロー画面やノティフィケーションバナーで 1 個だけ置くときは右端
- モーダルダイアログや緊急時バナーで 1 個だけ置くときは中央

### 6.4 無効化を安易に使わない

ボタンを押せない状態にすると、なぜ押せないのか分からず、利用者の手が止まりやすくなります。  
DADS は、まず無効化しない設計を勧めています。

- 先にやるべき作業を画面内で明示する
- 未入力のまま押されたら、必要な作業をエラーで案内する
- どうしても無効化するなら、直近に理由と有効化条件を具体的に書く
- `disabled` 属性のボタンにはアクセシビリティ上の問題がある

## 7. 情報提示と構造化の部品

### 7.1 見出し

見出しは、ページやセクションの主題を示すための部品です。

- HTML の `<h1>` から `<h6>` に対応させる
- ページ主題は `<title>` と対応する `<h1>` にする
- レベル 2 以降は、各セクションの主題を書く
- 見出しレベルと文字サイズは別に定義し、サービスのスタイルガイドで対応づける
- ショルダーやサブタイトルは `<p>` 要素として書き、`<hgroup>` で見出しとまとめる
- ショルダーやサブタイトル自体を `<h1>` から `<h6>` にしてはいけない
- リードアイコン、チップ、飾り罫は補助表現として使える

### 7.2 テーブル

テーブルは、行と列で構造化された情報を見せるために使います。

- 構造化データの表示に使う
- 比較に使う
- 大量データの一覧表示と、選択、編集、並び替えに使う

次の使い方は避けます。

- 単純な名前と値の組は、説明リストを使う
- レイアウト目的で使わない

### 7.3 アコーディオンとディスクロージャー

両者は似ていますが、折りたたむ範囲が違います。

| 部品 | 使う場面 | 使ってはいけない場面 |
|---|---|---|
| アコーディオン | 同種のセクションが連続し、見出しだけで概要が分かるとき | 表示を小さくするだけの目的、重要情報を隠す使い方、独立した単一セクション |
| ディスクロージャー | セクション内の補足情報だけを折りたたむとき | 展開しないと理解できない重要情報、セクション全体の折りたたみ |

### 7.4 ノティフィケーションバナー

ノティフィケーションバナーは、サービス全体、ページ、要素単位で重要度の高い情報を、利用者の操作に関係なく提示する部品です。  
利用者に対応を求めることもできます。

#### タイプ

- サクセス  
  完了や成功を伝える
- エラー  
  操作が完了しなかったことを伝える
- 警告  
  注意喚起や警戒情報を伝える
- 情報提示  
  成功、失敗、警告には当てはまらないが、重要度が高い情報を伝える

#### 配置

- 原則はページ上部のファーストビューに置く
- 特定コンテンツに関する通知なら、そのセクション内に置いてよい
- `position: fixed` や `position: sticky` で追従表示するなら、必ず閉じるボタンを付ける

#### リンクとアクション

- 目的は明確な告知であり、複雑な誘導は避ける
- 閉じる操作をアクションボタンで作らず、専用の閉じるパーツを使う
- 詳細ページへの単一リンクを持つ場合は、単一のプライマリーボタンで明示する
- 選択肢を与える場合は、ボタン群だけをクリック対象にする
- 閉じる操作を提供するなら、閉じた状態を記憶し、必要なら再表示手段も用意する
- モバイルではできるだけ簡素な構成にする

バナー外枠、アイコン、文字、ボタン文字にも、それぞれ必要なコントラストを確保します。

## 8. ナビゲーション

### 8.1 パンくずリスト

パンくずリストは、サイト階層内での現在位置を示します。

- ヘッダーとページ見出しの間に置く
- 階層の深さやページ名の長さを制御できないので、改行前提で設計する
- モバイルでは、改行方式か、横スクロール方式を採る

### 8.2 モバイルメニュー

モバイルメニューは、ハンバーガーメニューボタンとドロワーと組み合わせて使う重要部品です。  
モバイルでは本文側に関連リンクが省略され、モバイルメニューからしか行けないページが生じうるためです。

#### 形式

- スライド型とアコーディオン型のどちらかを選ぶ
- 両者を混在させない
- 情報設計として、2 段階を超える階層を作らない

#### アイコンと挙動の見せ方

- 通常リンクを強調したいときは、エンドアイコンを使う
- ページ遷移リンクと、サブメニュー開閉のような異なる挙動を混在させる場合は、明確に違うアイコンを使う
- 同じサービス内では、挙動ごとのアイコン仕様を揃える
- 単にシェブロンの向きだけで別機能を表現する方法は避ける

#### セクションの見せ方

- カテゴリータイトルでまとまりを示す
- 必要ならディバイダーを使う
- さらに明確にしたいときだけ、セクションアイコンを補助的に使う

### 8.3 ボトムナビゲーション

ボトムナビゲーションは、DADS では非推奨です。  
アクセシビリティまたはユーザビリティの観点から、現在は使用が勧められていません。  
どうしても使う場合は、不利益を受ける利用者がいることを前提に、慎重に扱います。
