# 担当領域

技術スタック・アーキテクチャ

# 前提理解

NextPatch は、開発中アプリや GitHub リポジトリ単位で、課題、バグ、実装予定、構想、技術メモ、参考サービス、ChatGPT との壁打ちメモを整理し、「今やるべきこと」をすぐ把握するための開発管理 Web アプリである。

本検討では、要件定義書・検討メモの全文はこの会話内で参照できないため、ユーザーが提示した説明、NextPatch プロジェクト文脈、DADS UI ルール、公式ドキュメントを前提に判断する。

UI に関わる判断では、DADS のアクセシビリティ、一貫性、重要情報を隠さない、フォームのラベル必須、安易な disabled 回避、768px を境にしたモバイル・デスクトップ切替、ボタン優先度の一貫性を優先する。テーブルコントロール、モーダル、タブ、グローバルメニュー等は DADS 側で未整備とされているため、採用する場合は NextPatch 独自仕様ではなく「暫定案」として扱う。

# 決定すべき論点

最初に決めるべき論点は、次のとおり。

1. アプリ形態を、通常の Web アプリ、PWA、ローカルファースト、デスクトップ/ローカルアプリのどれに寄せるか。
2. フロントエンドを Next.js にするか、Vite/React SPA にするか。
3. バックエンドを Next.js 内に持たせるか、BaaS/Functions に寄せるか。
4. DB を PostgreSQL、Supabase PostgreSQL、Firestore、SQLite 系、IndexedDB 中心のどれにするか。
5. ORM を使うか、DB 生成型 + SQL/クライアント SDK 中心にするか。
6. 認証を Supabase Auth、Auth.js、自前実装、Firebase Auth のどれにするか。
7. GitHub API / OpenAI API の秘密鍵やトークンを、どの層で保持・実行するか。
8. MVP でオフライン登録・同期まで入れるか、オンライン前提 + ローカル下書きに留めるか。
9. バックアップを、DB 基盤のバックアップだけに任せるか、ユーザー操作可能な JSON/CSV エクスポートを入れるか。
10. 型定義、バリデーション、ログ、エラーハンドリングをどこに集約するか。

| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| Next.js + PostgreSQL | フルスタックを TypeScript でまとめやすい。RDB なのでリポジトリ、課題、タグ、メモ、状態履歴、外部連携の関係を表現しやすい。Next.js App Router は Server Components 等を前提とした構成で、Route Handlers も app ディレクトリ内で実装できる。([Next.js][1]) | 認証、権限、ストレージ、バックアップ、管理画面、RLS 相当の認可を自前で決める必要がある。MVP では周辺実装が重い。                                 | 中長期では有力。ただし MVP では Supabase なしの素の PostgreSQL より、Supabase 採用のほうが開発開始が速い。 |
| Next.js + Supabase   | PostgreSQL、Auth、RLS、Storage、TypeScript 型生成、SQL migration、バックアップを同じ基盤で扱える。Supabase Auth は SSR と互換があり、セッションを Cookie に保存する構成が案内されている。([Supabase][2])                        | Supabase の設計思想に寄る。RLS 設計を誤ると権限バグが出る。料金・運用条件はプラン依存。                                           | **MVP 推奨。** NextPatch の関係データ、認証、バックアップ、将来連携のバランスが最もよい。                  |
| Vite/React + BaaS    | SPA として軽く始めやすい。Vite は高速なフロントエンドビルドツールとして公式に位置付けられている。([vitejs][3])                                                                                                       | GitHub Webhook、OpenAI API、バックアップ、認証済み SSR、サーバー側秘密鍵管理のために別バックエンドが必要になりやすい。                    | 小規模な閲覧中心アプリなら可。ただし NextPatch は将来連携が重要なので、MVP の本命にはしない。                  |

| SQLite 系構成           | Cloudflare D1 は SQLite セマンティクスを持つマネージド serverless DB、Turso は同じ SDK でローカル SQLite、remote、embedded replica 等に接続できる。([Cloudflare Docs][5])                                   | 認証、RLS、ファイル、バックアップ、将来の連携ジョブを別途組む必要がある。チーム/ワークスペース権限設計は PostgreSQL + RLS より自前寄り。              | 単一ユーザー・ローカル重視なら候補。NextPatch MVP では優先しない。                                |
| Firebase 系構成         | Firebase/Firestore は Auth、Hosting、Functions、オフライン同期が強い。Firestore は Web/Mobile でオフライン永続化をサポートし、復帰時に同期する。([Firebase][6])                                                   | Firestore は NoSQL であり、リポジトリ、課題、タグ、状態履歴、横断検索、集計、バックアップ/エクスポートの設計が NextPatch の管理データとやや噛み合いにくい。 | オフライン・リアルタイム最優先なら選ぶ。NextPatch MVP では PostgreSQL 系を優先する。                 |

# 推奨案

**MVP の推奨構成は、Next.js App Router + TypeScript + Supabase（PostgreSQL / Auth / Storage / RLS）+ Vercel Hosting + Zod + Supabase 生成型である。ORM は MVP では採用しない。**

理由は明確で、NextPatch は「開発リポジトリ単位の構造化メモ管理」であり、データは NoSQL より RDB に向く。さらに、初期段階から認証、権限、バックアップ、将来の GitHub / OpenAI 連携を見据える必要があるため、素の PostgreSQL よりも Supabase を使うほうが、MVP の開発範囲を小さくできる。

Next.js は画面、フォーム、サーバー処理、API エンドポイントを同一リポジトリにまとめやすい。App Router は Server Components 等を前提にしており、Route Handlers は Web Request/Response API によるカスタムリクエスト処理を app ディレクトリ内に持てる。フォーム更新には Server Actions、外部 API / Webhook / Export には Route Handlers を使い分けるのがよい。([Next.js][1])

Supabase は PostgreSQL、Auth、RLS、TypeScript 型生成、SQL migration、Storage、Edge Functions、バックアップを同じ方向性で扱える。RLS は Supabase Auth と組み合わせてブラウザから DB までの認可防御に使えるため、ワークスペース単位・リポジトリ単位のアクセス制御に向く。([Supabase][7])

ホスティングは Vercel を第一候補にする。Vercel は Next.js のデプロイに対してゼロコンフィグに近い体験と Preview URL を提供し、環境変数も Production / Preview / Development で分けられる。GitHub 連携や OpenAI 連携に必要な秘密情報は、クライアントに出さずサーバー側環境変数で扱う。([Vercel][8])

ORM は MVP では使わない。Supabase の SQL migration をスキーマの正とし、`supabase gen types` で DB 型を生成し、アプリ側では repository/service 層で DB 型を DTO に変換する。複雑な集計や検索が増えた段階で、Drizzle ORM を server-only の補助として導入する余地を残す。Drizzle は database-first と codebase-first の両方に対応できるが、MVP 初期から入れると Supabase 型生成・RLS・SQL migration との責務が重複しやすい。([Supabase][9])

条件別の代替案は次のとおり。

* **Supabase ロックインを強く避ける条件**なら、Next.js + PostgreSQL + Drizzle + Auth.js を選ぶ。
* **完全オフライン入力が最重要条件**なら、Firebase / Firestore か、IndexedDB + 同期基盤を再検討する。
* **単一ユーザー・ローカル保存が最重要条件**なら、SQLite / Turso / D1 系を再検討する。
* **将来のセルフホストが必須条件**なら、Supabase 依存を薄くし、DB schema、repository 層、export/import を早期に分離する。

# MVPに含めるべき内容

MVP に含めるべき技術構成は次のとおり。

| 項目       | MVP 採用内容                                                                                                                                             |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| アプリ形態    | レスポンシブ Web アプリ。PWA は「軽量対応」に留める。                                                                                                                      |
| フロントエンド  | Next.js App Router、React、TypeScript。Server Components 基本、必要箇所のみ Client Components。                                                                   |
| UI 実装    | DADS 方針に沿った独自コンポーネント群。Tailwind CSS を使う場合も、DADS 由来の色、余白、文字、ボタン優先度を design token 化する。                                                                  |
| バックエンド   | Next.js Server Actions + Route Handlers。Server Actions は内部フォーム更新、Route Handlers は API、export、webhook、将来連携に使う。                                        |
| DB       | Supabase PostgreSQL。                                                                                                                                 |
| 認証       | Supabase Auth。MVP はメールログインを基本にし、GitHub ログインは「認証」と「GitHub リポジトリ連携」を混同しない形で後から追加可能にする。                                                                 |
| 認可       | Supabase RLS。workspace_members を基準に、本人が参加している workspace のみ参照・更新可能にする。                                                                                |
| ORM      | MVP では不採用。SQL migration + Supabase generated types + repository 層。                                                                                   |
| バリデーション  | Zod。フォーム、Server Action、Route Handler の入口で同じ schema を使う。Zod は TypeScript-first の schema validation ライブラリとして、schema 定義と parse による検証を提供している。([Zod][10]) |
| バックアップ   | Supabase 側バックアップに加え、MVP から workspace 単位の JSON export を入れる。                                                                                           |
| ログ       | 構造化ログ。requestId、userId、workspaceId、action、entityType、entityId、result を出す。本文メモや API key は記録しない。                                                       |
| エラー監視    | MVP 後半または公開前に Sentry を導入候補。Sentry は Next.js App Router 向けの error/log/trace 取得手順を提供している。([Sentry Docs][11])                                           |
| PWA      | manifest、アイコン、モバイルホーム画面起動を先に入れる。Service Worker による本格オフライン同期は入れない。                                                                                    |
| オフライン    | MVP では「オンライン保存」前提。ただしクイック登録フォームの未送信下書きだけ localStorage または IndexedDB に保存する。                                                                           |
| 将来連携の受け皿 | `integrations`、`external_accounts`、`sync_jobs`、`integration_events`、`ai_runs` 相当のテーブル枠を用意する。                                                         |

# MVPでは後回しにする内容

MVP では次を後回しにする。

* 完全なオフラインファースト同期。
* IndexedDB を主 DB とする構成。
* 複数端末間の競合解決。
* GitHub App の本格実装。
* GitHub Issue / PR / commit の双方向同期。
* ChatGPT / OpenAI API による自動分類、自動要約、自動優先度付け。
* ベクトル検索、RAG、類似メモ検索。
* 複雑なロール権限。
* 組織向け監査ログの完全対応。
* import 復元機能の完全実装。
* Supabase Edge Functions によるジョブワーカーの本格運用。
* タブ、モーダル、テーブルコントロール等の細かい独自 UI 確定。DADS 未整備領域は暫定案に留める。

# データ設計への影響

推奨構成では、データモデルは最初から PostgreSQL の正規化モデルで設計する。

MVP の主要テーブルは次を推奨する。

| テーブル                        | 役割                                                     |
| --------------------------- | ------------------------------------------------------ |
| `profiles`                  | Supabase Auth のユーザーに紐づくアプリ内プロフィール。                     |
| `workspaces`                | 個人またはチームの作業単位。MVP は個人 workspace だけでもよい。                |
| `workspace_members`         | workspace 参加者とロール。RLS の基準。                             |
| `repositories`              | GitHub リポジトリまたは任意の開発プロジェクト。                            |
| `items`                     | 課題、バグ、実装予定、将来構想、技術メモ、参考サービス、新規アイデア、壁打ちメモを統合的に扱う中核テーブル。 |
| `item_notes`                | item に紐づく追記メモ、履歴、壁打ち結果。                                |
| `labels`                    | バグ、UI、API、緊急、調査中などのラベル。                                |
| `item_labels`               | item と labels の多対多。                                    |
| `item_links`                | item 同士の関連、ブロック、重複、参考リンク。                              |
| `views` または `saved_filters` | 「今やるべきこと」表示の条件保存。                                      |
| `attachments`               | 将来の画像、ログ、資料添付。Supabase Storage のパスを保持。                 |
| `exports`                   | バックアップ export の履歴。                                     |
| `integration_accounts`      | 将来の GitHub App installation、OAuth、外部連携アカウント。           |
| `integration_events`        | GitHub Webhook 等の受信イベント。冪等処理用。                         |
| `sync_jobs`                 | 将来の同期ジョブ、再試行、エラー管理。                                    |
| `ai_runs`                   | 将来の OpenAI API 実行履歴、prompt version、入力参照、出力参照、コスト概算。    |

`items` は `type` を持たせ、`task`、`bug`、`plan`、`idea`、`memo`、`tech`、`reference`、`future` などを扱えるようにする。ただし、型を増やしすぎると UI が複雑になるため、MVP では「課題」「バグ」「メモ」「アイデア」「参考」の 5 種度に絞るのがよい。

RLS は workspace 単位で設計する。すべての主要テーブルに `workspace_id` を持たせ、`workspace_members` に存在する user のみ select / insert / update / delete 可能にする。Supabase RLS は Postgres の policy としてテーブルアクセスごとに適用されるため、この構成と相性がよい。([Supabase][7])

バックアップは、DB 基盤のバックアップとアプリ内 export を分ける。Supabase の PITR は任意時点への復元を可能にするが有料プラン・追加条件があるため、MVP ではユーザーが手元に保管できる JSON export を別途用意する。([Supabase][12])

# 画面設計への影響

画面設計は、スマホでのクイック登録と PC での一覧管理を分けて最適化する。

MVP の主要画面は次を推奨する。

| 画面      | 役割                               | 技術・UI上の注意                                                        |
| ------- | -------------------------------- | ---------------------------------------------------------------- |
| ダッシュボード | 「今やるべきこと」を表示                     | Server Component で初期表示を速くする。重要情報を折りたたまない。                        |
| クイック登録  | スマホから 3 操作以内でメモ登録                | Client Component + Server Action。タイトル、種類、リポジトリ、本文だけに絞る。          |
| リポジトリ一覧 | PC で横断管理                         | テーブルまたはカード。DADS ではテーブルは構造化データ向きだが、テーブルコントロールは未整備なのでフィルタ UI は暫定案。 |
| item 詳細 | 課題、バグ、メモの詳細                      | 重要状態、次アクション、履歴を上部に出す。補足だけディスクロージャー可。                             |
| 検索・フィルタ | status、type、label、repository で絞る | URL query に状態を持たせ、共有・復元しやすくする。                                   |
| 設定      | workspace、export、将来連携設定          | GitHub / OpenAI は MVP では「準備中」でもよい。                               |

DADS に合わせ、フォームではプレースホルダーをラベル代替にしない。未入力時はボタンを disabled にするより、送信後に具体的なエラーテキストを表示する。スマホではボタンを縦積みにし、主要操作を上、戻る/キャンセルを下に置く方針で統一する。

PWA は MVP では「インストールしやすい Web アプリ」程度に留める。manifest はアプリ名、アイコン、テーマ色などをブラウザに伝えるために使えるが、オフライン操作には Service Worker や Cache、IndexedDB 等の追加設計が必要になる。([MDNウェブドキュメント][13])

# API・機能設計への影響

API は単純 CRUD ではなく、ユースケース単位に寄せる。

MVP の内部操作は Server Actions を中心にする。

* `quickCreateItem`
* `updateItemStatus`
* `updateItemPriority`
* `archiveItem`
* `addItemNote`
* `createRepository`
* `updateRepository`
* `exportWorkspace`

外部公開または将来連携に関わる処理は Route Handlers にする。

* `GET /api/export/workspaces/:id`
* `POST /api/import/preview`、将来対応
* `POST /api/integrations/github/webhook`、将来対応
* `POST /api/integrations/github/sync`、将来対応
* `POST /api/ai/suggest-next-actions`、将来対応
* `POST /api/ai/summarize-thread`、将来対応

GitHub 連携は、将来 GitHub App 方式を前提にした構造にする。GitHub は Webhook により push や pull request などのイベント通知を受け取れるため、ポーリングではなく webhook + job queue を基本にする。GitHub 公式も REST API 利用時は polling を避け、webhook events を購読することを推奨している。([GitHub Docs][14])

GitHub API はレート制限を前提に設計する。未認証 REST API は 60 req/hour、認証済みユーザーは通常 5,000 req/hour、GitHub App installation は最低 5,000 req/hour から条件により増える。さらに secondary rate limit もあるため、`sync_jobs` に `retry_after`、`rate_limit_remaining`、`last_error_code` を保持する。([GitHub Docs][15])

OpenAI / ChatGPT 連携は、必ずサーバー側 API として設計する。OpenAI は API key をブラウザやモバイルアプリなどの client-side 環境に置かず、バックエンド経由にすることを推奨している。将来の AI 機能では、prompt version、入力 item、出力 JSON、使用モデル、実行者、コスト概算、失敗理由を `ai_runs` に保存する。([OpenAI Help Center][16])

# リスクと対策

| リスク             | 内容                                       | 対策                                                                         |
| --------------- | ---------------------------------------- | -------------------------------------------------------------------------- |
| Supabase ロックイン  | Auth、RLS、Storage、generated types に依存する   | DB schema、repository 層、DTO、export/import を分離し、DB 自体は標準 PostgreSQL に寄せる。    |
| RLS 設計ミス        | workspace 越境参照が起こる可能性                    | すべての主要テーブルに `workspace_id` を持たせ、RLS policy のテストを必須にする。                     |
| ORM 不採用によるクエリ散在 | Supabase client 呼び出しが画面に散らばる             | DB access は `src/server/repositories/*` に閉じる。UI から直接 Supabase query を呼ばない。 |
| オフライン未対応への不満    | 移動中のスマホ登録で保存失敗する可能性                      | MVP では下書きを localStorage / IndexedDB に保存し、送信失敗時に復元できるようにする。完全同期は後回し。        |
| GitHub 連携の複雑化   | Webhook、認証、レート制限、再試行が絡む                  | MVP ではテーブルと抽象 interface だけ準備し、実連携は後続。GitHub App、webhook、job queue 前提で設計する。 |
| OpenAI コスト増     | 自動要約・分類の多用で費用が読みにくい                      | `ai_runs` に usage/cost を保存し、手動実行から開始。API key は server-side のみ。             |
| UI コンポーネントの独自化  | DADS 未整備部分を勝手に固定仕様にする危険                  | テーブル操作、モーダル、タブ等は「暫定案」と明記し、DADS 更新時に見直す。                                    |
| バックアップ過信        | Supabase 側 backup だけではユーザーが自分で保全できない     | MVP から workspace JSON export を入れる。PITR は本番プランで検討する。                        |
| ログの情報漏洩         | メモ本文、API key、GitHub token、AI 入出力をログに出す危険 | ログは metadata のみ。本文・秘密情報は出さない。                                              |

# 未決定事項

開発着手前に、次を統括担当で確定する必要がある。

1. MVP は個人利用前提か、最初から workspace / team を入れるか。
2. GitHub ログインを MVP 認証に含めるか。ただし、GitHub ログインと GitHub App 連携は別機能として扱うべき。
3. バックアップ export の形式を JSON のみにするか、CSV も必要か。
4. item type の初期種類をいくつに絞るか。
5. 「今やるべきこと」の算出ロジックを、単純な手動 priority にするか、status / due / priority / blocked を組み合わせるか。
6. 添付ファイルを MVP に含めるか。含める場合は Supabase Storage の private bucket と RLS 方針が必要。
7. エラー監視を MVP 初期から Sentry まで入れるか、Vercel logs + structured console に留めるか。
8. PWA の Service Worker を MVP で入れるか。推奨は manifest 中心で、Service Worker は静的 asset cache 程度または後回し。
9. Tailwind CSS を使うか、CSS Modules / CSS variables 中心にするか。推奨は Tailwind + DADS design token 化。
10. 将来 AI 機能で保存する情報の範囲。AI 入出力全文を保存するか、要約・参照 ID のみにするか。

# 統括担当へ渡す要点

MVP の技術構成は、**Next.js App Router + TypeScript + Supabase + Vercel** を推奨する。

DB は Supabase PostgreSQL、認証は Supabase Auth、認可は Supabase RLS、型は Supabase generated types、バリデーションは Zod、ホスティングは Vercel とする。ORM は MVP では使わず、SQL migration と repository 層で責務を分ける。

PWA は MVP で「軽量対応」に留める。manifest、アイコン、スマホ起動導線は入れるが、完全なオフライン同期は後回しにする。スマホのクイック登録は、オンライン保存 + ローカル下書き復元で対応する。

GitHub 連携は、MVP では実装しなくてもよいが、DB と API の設計では GitHub App、webhook、rate limit、job queue を前提にしておく。OpenAI / ChatGPT 連携も、API key を client-side に出さない server-side adapter と `ai_runs` テーブルを前提にする。

バックアップは、Supabase の platform backup だけに依存せず、MVP から workspace 単位の JSON export を入れるべきである。Supabase PITR は本番プランで検討する。

UI では DADS を優先し、フォームラベル、エラー表示、ボタン優先度、disabled 回避、768px breakpoint、重要情報を隠さない設計を守る。DADS 未整備のテーブルコントロール、モーダル、タブ等は、NextPatch 独自確定仕様ではなく暫定案として扱う。
