# 担当領域

ChatGPT連携・AI取り込み。

対象は、スマホでChatGPTと壁打ちした内容をNextPatchへ取り込み、課題・進捗・バグ・実装予定・将来構想・採用技術候補・参考サービス・新規アイデア・未整理メモへ整理する前段の設計判断である。実装コードは扱わない。

# 前提理解

NextPatchは、リポジトリ単位・開発中アプリ単位で「今やるべきこと」をすぐ把握するための開発管理アプリである。ChatGPT連携は、最初から完全自動化するよりも、壁打ちで得た内容を失わず保存し、あとから安全に分類できることを優先すべき領域である。

確認できた前提資料は、会話内で与えられたNextPatchの説明と、`dads_app_ui_design_rules_20260411.md`である。NextPatch要件定義書および「開発着手前に決定すべき内容の検討メモ」の全文は、この環境では確認できなかったため、本回答は会話内の要件を前提にし、統括担当側で全文照合する前提で整理する。

UIではDADS参照ルールを優先する。特に、フォームはラベル・サポートテキスト・静的なエラーテキストを持たせ、プレースホルダーで説明を代用せず、コピー・ペーストを禁止しない方針が重要である。クイック登録は長文貼り付けが中心なので、テキストエリアを使い、サポートテキストで形式例を明示する。 また、ボタンは1画面1プライマリーを原則とし、確認画面では「選択した候補を登録」を主ボタンにする。

DADS上でモーダルダイアログ、ステップナビゲーション、タブ、テーブルコントロール等はガイドライン準備中とされているため、分類確認画面でモーダルやステップUIを使う案は「暫定案」として扱う。

Web参照元、確認日はいずれも2026年4月20日。OpenAI公式ドキュメントでは、新規プロジェクトにはResponses APIが推奨され、Structured OutputsはJSON modeよりもスキーマ準拠性を担保できると説明されている。将来のAI分類をAPI化する場合は、Responses APIとStructured Outputsを前提候補にするのが妥当である。([OpenAI開発者][1]) OpenAI APIのデータはデフォルトではモデル学習に使われず、API利用時のabuse monitoring logsは原則最大30日保持されると説明されているため、プライバシー設計では「送信前の明示確認」「`store: false`」「送信禁止情報の定義」が必要である。([OpenAI開発者][2]) ChatGPT側ではTemporary Chatが履歴に残らず、メモリを使わず、モデル学習にも使われないと説明されているが、それでも機密情報を貼らない運用ルールは必要である。([OpenAI Help Center][3])

# 決定すべき論点

1. MVPで、NextPatch本体がAI分類を行うか、手動貼り付けと未整理メモ保存に限定するか。
2. ChatGPTから貼り付ける形式を、自由文、Markdown、JSON、またはJSON＋Markdownの併用にするか。
3. 貼り付けた内容を、いきなりタスク等に分解するか、必ず未整理メモとして原文保存するか。
4. 分類先の初期カテゴリをどこまで持つか。最低限は、タスク、バグ、アイデア、参考サービス、採用候補、未整理メモ。
5. AI分類を入れる場合、分類結果を自動登録するか、候補として確認画面に出すだけにするか。
6. AIに送ってよい情報、送らない情報、送信前の警告条件をどう定義するか。
7. JSON取り込みのスキーマをMVPから固定するか、将来拡張可能なバージョン付き形式にするか。
8. Markdown取り込みを正式対応にするか、MVPでは「読める保存形式＋簡易解析」に留めるか。
9. 将来のChatGPT Actions、Apps SDK、MCP、OpenAI API連携を見据えて、内部APIをどう分離するか。
10. スマホ操作で、クイック登録から保存まで3操作以内に収める画面導線にするか。

# 選択肢比較

| 案                                            | 内容                                        | 長所                                            | 短所                              | MVP適性   |
| -------------------------------------------- | ----------------------------------------- | --------------------------------------------- | ------------------------------- | ------- |
| A. 自由文を未整理メモとして保存のみ                          | クイック登録に貼り付け、原文だけ保存                        | 実装が最小。誤分類なし。プライバシーリスクが低い                      | あとで整理する負担が大きい。「今やるべきこと」へ変換されにくい | 低〜中     |
| B. 手動貼り付け＋未整理メモ保存＋JSON/Markdown簡易解析          | 原文は必ず保存し、JSONがあれば候補を表示。Markdownは見出しだけ簡易解析 | 安全性と実用性のバランスがよい。AI APIなしでも始められる。将来AI分類へ拡張しやすい | ChatGPTに標準プロンプトを使わせる必要がある       | 高       |
| C. MVPからNextPatch内でAI分類                      | 貼り付け後、NextPatchがAI APIで分類候補を作る            | ユーザー負担が少ない。分類体験を早く検証できる                       | API費用、誤分類、機密送信、プロンプト管理、監査ログが必要  | 低〜中     |
| D. ChatGPT Actions / Apps SDKで直接NextPatchへ登録 | ChatGPTからNextPatch APIを呼ぶ                 | 体験は最短。将来の理想形になり得る                             | 認証、公開API、権限、悪用対策、審査・運用が重い       | MVPでは不適 |

推奨はBである。MVPでは「貼り付けた内容を必ず未整理メモとして残し、構造化データが含まれている場合だけ分類候補を表示する」。NextPatch本体から外部AIへ送信する機能はMVPには入れない。

理由は、ChatGPT壁打ちの主価値が「忘れないこと」と「あとから整理できること」にあり、初期からAI分類の精度・費用・プライバシー・誤分類UXを同時に抱えるとMVPが膨らむためである。将来AI分類を入れる場合でも、OpenAI公式ではResponses APIが新規プロジェクト向けに推奨され、Structured Outputsでスキーマ準拠の出力を扱えるため、MVPのJSONスキーマをそのままAI出力スキーマへ転用できる。([OpenAI開発者][1])

# 推奨案

## 1. MVPでの取り込み方式

MVPでは、**手動貼り付け方式**を採用する。

具体的には、リポジトリ詳細またはプロジェクト詳細に「クイック登録」ボタンを置き、ユーザーがChatGPTの出力をテキストエリアへ貼り付ける。保存時には、内容がJSONでもMarkdownでも自由文でも、まず必ず`未整理メモ`として原文を保存する。JSONブロックが含まれている場合だけ、NextPatchがローカルで構造を解析し、「分類候補」として確認画面に表示する。

操作は3操作以内を目標にする。

1. リポジトリ画面で「クイック登録」を押す。
2. ChatGPTの出力を貼り付ける。
3. 「保存」を押す。

保存後に分類まで進める場合は、「保存しました」バナー内に「分類候補を確認」を出す。ただし、重要情報を隠さないというDADS原則を踏まえ、分類が必要な状態は未整理メモ一覧にも明示する。

## 2. 将来のAI分類方式

将来は、未整理メモに対してユーザーが明示的に「AIで分類候補を作る」を押した場合のみAI分類を行う。

分類先は次を基本とする。

| 分類先               | 用途                          |
| ----------------- | --------------------------- |
| task              | 実装予定、調査、修正、確認など、実行可能な作業     |
| bug               | 再現条件、期待結果、実際結果を持つ不具合        |
| idea              | まだ実装確定していない改善案・新機能案         |
| reference_service | 参考サービス、競合、UI/UX参考、料金・機能比較対象 |
| tech_candidate    | 採用候補技術、ライブラリ、API、アーキテクチャ案   |
| future_plan       | MVP後の将来構想                   |
| note              | 分類不能または判断保留のメモ              |

AI分類は、必ず「候補」として保存し、タスク・バグ等の本体には自動登録しない。ユーザーが確認画面で選択・修正・除外してから登録する。

将来APIはOpenAI Responses APIを第一候補にし、Structured Outputsで`nextpatch.import.v1`と互換のJSON Schemaを指定する。OpenAI公式ではStructured OutputsがJSON modeよりスキーマ準拠に向くと説明されているため、分類結果の安定性を確保しやすい。([OpenAI開発者][4]) ただし、AI API利用時は、APIデータがデフォルトで学習に使われないこととは別に、abuse monitoring logsやアプリケーション状態の保持条件を理解し、送信前確認・`store: false`・機密情報除外を設計に含める。([OpenAI開発者][2])

## 3. 未整理メモのデータ構造

未整理メモは、分類前の原本であり、あとから分類・再分類・監査できるようにする。MVPから必ず持つ。

最小構造は次。

| フィールド                   |        型 | 必須 | 説明                                                                              |
| ----------------------- | -------: | -: | ------------------------------------------------------------------------------- |
| id                      |     UUID | 必須 | 未整理メモID                                                                         |
| workspace_id            |     UUID | 必須 | ワークスペース                                                                         |
| project_id              |     UUID | 任意 | 紐づく開発中アプリ                                                                       |
| repository_id           |     UUID | 任意 | 紐づくGitHubリポジトリ                                                                  |
| title                   |   string | 任意 | 自動抽出または手入力の短い題名                                                                 |
| raw_content             |     text | 必須 | 貼り付け原文                                                                          |
| source_type             |     enum | 必須 | `manual_paste`, `chatgpt_paste`, `api_future`, `import_file_future`             |
| source_label            |   string | 任意 | 例: ChatGPT, ChatGPT Temporary Chat, GPT-5.4                                     |
| format_detected         |     enum | 必須 | `json`, `markdown`, `plain_text`, `mixed`, `invalid_json`                       |
| schema_version          |   string | 任意 | 例: `nextpatch.import.v1`                                                        |
| status                  |     enum | 必須 | `unreviewed`, `parsed`, `reviewing`, `partially_applied`, `applied`, `archived` |
| privacy_level           |     enum | 必須 | `normal`, `confidential`, `secret`, `no_ai`                                     |
| ai_processing_allowed   |  boolean | 必須 | AI送信可否                                                                          |
| parse_errors            |     json | 任意 | JSON不正、必須項目不足など                                                                 |
| extracted_summary       |     text | 任意 | 人間向け要約                                                                          |
| created_at / updated_at | datetime | 必須 | 作成・更新時刻                                                                         |
| created_by              |     UUID | 必須 | 登録者                                                                             |
| checksum                |   string | 任意 | 重複検出用                                                                           |
| applied_item_count      |   number | 必須 | タスク等に反映済みの件数                                                                    |

将来AI分類のために、`classification_candidates`を別テーブルにする。

| フィールド           | 説明                                                                                  |
| --------------- | ----------------------------------------------------------------------------------- |
| id              | 分類候補ID                                                                              |
| memo_id         | 元の未整理メモ                                                                             |
| target_type     | `task`, `bug`, `idea`, `reference_service`, `tech_candidate`, `future_plan`, `note` |
| proposed_title  | 候補タイトル                                                                              |
| proposed_body   | 候補本文                                                                                |
| proposed_fields | 優先度、ステータス、技術名、URL、再現手順など                                                            |
| confidence      | `low`, `medium`, `high`                                                             |
| evidence_quote  | 原文中の根拠抜粋                                                                            |
| needs_review    | true固定に近い扱い                                                                         |
| user_decision   | `pending`, `accepted`, `edited`, `rejected`, `merged`                               |
| generated_by    | `parser`, `chatgpt_prompt`, `openai_api_future`, `manual`                           |
| model_info      | 将来AI利用時のみ                                                                           |
| prompt_version  | 将来AI利用時のみ                                                                           |

## 4. ChatGPTからNextPatchへ渡す推奨フォーマット

推奨は、**人間が読める短い要約＋機械が読めるJSONブロック**である。

NextPatch側の正規取り込み形式はJSONにする。Markdownは補助形式とし、MVPでは「Markdownをきれいに保存し、決められた見出しだけ簡易解析する」扱いに留める。

推奨JSONの骨子は次。

```json
{
  "schema_version": "nextpatch.import.v1",
  "source": {
    "tool": "ChatGPT",
    "conversation_title": "",
    "captured_at": "YYYY-MM-DD",
    "privacy_note": "機密情報は含めていない"
  },
  "repository_hint": {
    "owner": "",
    "name": "",
    "branch": "",
    "area": ""
  },
  "summary": "",
  "items": [
    {
      "kind": "task",
      "title": "",
      "body": "",
      "priority": "medium",
      "labels": [],
      "evidence": "",
      "confidence": "medium",
      "needs_review": true
    }
  ],
  "unclassified_notes": [
    {
      "title": "",
      "body": "",
      "reason": "分類保留"
    }
  ]
}
```

Markdown補助形式は次の見出しだけを正式認識する。

```markdown
# NextPatch Import

Repository: owner/repo
Captured: YYYY-MM-DD

## Summary
...

## Tasks
- ...

## Bugs
- ...

## Ideas
- ...

## Reference Services
- ...

## Tech Candidates
- ...

## Future Plans
- ...

## Unclassified Notes
- ...
```

# MVPに含めるべき内容

MVPに含めるべき内容は次で確定する。

1. **クイック登録画面**
   リポジトリまたはプロジェクトから開ける貼り付け専用画面。テキストエリア、任意のタイトル、関連リポジトリ、保存ボタンを持つ。フォームではラベルとサポートテキストを明示し、プレースホルダーに説明を依存させない。

2. **未整理メモ保存**
   どの形式でも必ず原文保存する。JSON解析に失敗しても保存できる。保存失敗時だけ静的エラーを表示する。

3. **JSON取り込みパーサー**
   `schema_version: nextpatch.import.v1`を検出し、`items`を分類候補に変換する。スキーマ外の値は破棄せず、`parse_errors`または`unclassified_notes`に残す。

4. **Markdown取り込み補助**
   正式見出しに一致する範囲だけ候補化する。それ以外は未整理メモ本文として保存する。

5. **分類候補確認画面**
   JSONまたはMarkdownから候補が作れた場合だけ表示する。候補は即登録しない。

6. **誤分類を防ぐUX**
   すべての候補に、分類先、タイトル、本文、根拠、信頼度、登録チェックを表示する。初期状態では`needs_review: true`扱い。

7. **プライバシー注意表示**
   クイック登録画面に「秘密鍵、トークン、個人情報、顧客情報、未公開の機密仕様は貼り付けない」旨を表示する。AI分類はMVPでは実行しないが、将来のために`ai_processing_allowed`を保存する。

8. **ChatGPT用標準プロンプト**
   ユーザーがChatGPT側で壁打ち後に使うプロンプトをNextPatch内に表示・コピーできるようにする。

# MVPでは後回しにする内容

MVPでは次を後回しにする。

* NextPatch本体からOpenAI APIへ送信するAI分類。
* AI分類結果の自動登録。
* GPT Actions、Apps SDK、MCPによるChatGPTからの直接登録。
* ChatGPTエクスポートファイルの一括取り込み。
* 画像、音声、PDF、スクリーンショットの取り込み。
* 埋め込み検索、類似メモ検索、重複候補AI判定。
* GitHub Issues、Linear、Notion、Slack等との双方向同期。
* 組織向け承認フロー、監査証跡の詳細レポート。
* 複雑なステップナビゲーションやモーダル中心の導線。DADS上でステップナビゲーションやモーダルダイアログは未整備のため、使う場合は暫定案として扱う。

# データ設計への影響

最重要は、**原文保存と分類結果を分離すること**である。

未整理メモは「原本」、分類候補は「変換候補」、タスク・バグ等は「確定済みデータ」として別エンティティにする。これにより、誤分類しても原文から再分類できる。

データ設計上の決定は次。

| 項目     | 決定                                                                 |
| ------ | ------------------------------------------------------------------ |
| 原文保存   | 必須。分類済みになっても削除しない                                                  |
| 分類候補   | 未整理メモとは別管理                                                         |
| 確定データ  | task, bug, idea, reference_service, tech_candidate等の既存または専用テーブルへ登録 |
| AI処理可否 | 未整理メモ単位で保持                                                         |
| スキーマ   | `nextpatch.import.v1`から開始し、破壊的変更は`v2`にする                           |
| 証跡     | どのメモからどのタスクが作られたかリンクを残す                                            |
| 取り込み失敗 | 失敗ではなく「未整理メモとして保存＋解析エラー表示」にする                                      |
| 機密度    | `normal`, `confidential`, `secret`, `no_ai`をMVPから持つ                |

分類先ごとの最低フィールドは次。

| 分類先               | 最低フィールド                                                                              |
| ----------------- | ------------------------------------------------------------------------------------ |
| task              | title, body, priority, status, source_memo_id                                        |
| bug               | title, observed, expected, steps_to_reproduce, environment, priority, source_memo_id |
| idea              | title, body, benefit, status, source_memo_id                                         |
| reference_service | name, url, reason, useful_points, source_memo_id                                     |
| tech_candidate    | name, category, reason, concerns, decision_status, source_memo_id                    |
| future_plan       | title, body, target_phase, source_memo_id                                            |

# 画面設計への影響

## クイック登録画面

画面名は「クイック登録」。モバイルでも1カラムで使える構成にする。

配置は次。

1. 見出し: クイック登録
2. 関連先: プロジェクト / リポジトリ
3. タイトル: 任意
4. 貼り付け本文: 必須テキストエリア
5. 機密度: 通常 / 機密 / AI送信禁止
6. アクション: 「保存」プライマリー、「キャンセル」テキスト

テキストエリアのサポートテキスト例は、「ChatGPTの回答、Markdown、NextPatch Import JSONを貼り付けできます。秘密鍵、APIキー、個人情報は貼り付けないでください。」とする。

DADSでは長文自由記述にはテキストエリアを使い、文字数上限がある場合はラベルや常時表示の文字数カウンターで示す考え方が示されている。

## 未整理メモ一覧

リポジトリごとに「未整理メモ」セクションを置く。表示項目は、タイトル、作成日、形式、機密度、分類状態、候補件数。

「未整理メモがあるが未分類」の状態は、ダッシュボードの「今やるべきこと」付近にも小さく出す。ただし重要情報をアコーディオンに隠さない。

## 5. 分類時の確認画面案

分類確認はモーダルではなく、原則として**専用ページ**にする。モーダル案はDADS未整備のため暫定案扱いに留める。

画面構成は次。

* 上部: 「分類候補の確認」
* 情報バナー: 「候補はまだ登録されていません。内容を確認してから登録してください。」
* 左または上段: 元の未整理メモの要約と原文リンク
* 中央: 分類候補カード一覧
* 各候補カード:

  * 登録チェックボックス
  * 分類先
  * タイトル
  * 本文
  * 根拠引用
  * 信頼度
  * 「編集」「除外」「未整理に戻す」
* 下部アクション:

  * プライマリー: 選択した候補を登録
  * セカンダリー: 未整理のまま保存
  * テキスト: キャンセル

分類先が5種類以内に絞られる候補編集ではラジオボタンを使える。6種類以上を扱う場合はセレクトボックスを使う。DADSでは、単一選択で5件以下ならラジオボタン、6件以上または長い選択肢ならセレクトボックスが整理されている。

# API・機能設計への影響

MVPでは外部AI APIを呼ばないが、将来呼べるように内部境界を分ける。

必要な内部機能は次。

| 機能                             |     MVP |                             将来 |
| ------------------------------ | ------: | -----------------------------: |
| quickCapture.create            |      必須 |                             継続 |
| importParser.parseJson         |      必須 |                             継続 |
| importParser.parseMarkdown     |      簡易 |                             強化 |
| classificationCandidate.create |      必須 |                      AI分類結果も保存 |
| classificationCandidate.apply  |      必須 |                             継続 |
| aiClassifier.classify          |      なし |                             追加 |
| privacyScanner.scan            | 簡易キーワード |                             強化 |
| providerAdapter.openai         |      なし |                             追加 |
| directImportApi                |      なし | ChatGPT Actions / Apps SDK用に追加 |

将来のAI分類APIは、`AiClassificationProvider`のような抽象層を置き、OpenAI API、ローカルLLM、別プロバイダを差し替え可能にする。OpenAI公式ではResponses APIが新規プロジェクトに推奨され、Structured Outputs、Function Calling、Web search、File search等の機能を持つとされているため、将来の分類・補完・重複検出の基盤候補になる。([OpenAI開発者][1])

ChatGPTからNextPatchへ直接登録する将来案では、GPT ActionsまたはApps SDKを検討する。GPT ActionsはChatGPTから外部REST APIを呼び、自然言語をAPI用JSONに変換して実行する仕組みであり、認証にはNone、API Key、OAuthの選択肢がある。([OpenAI開発者][5]) Apps SDKを使う場合はMCPサーバーが必要で、ChatGPTにツールを公開できる。([OpenAI開発者][6]) ただし、Apps SDKの公式セキュリティ指針では、最小権限、明示的同意、サーバー側検証、人間の確認、ログのPII削除が求められているため、MVPで扱うには重い。([OpenAI開発者][7])

# リスクと対策

| リスク           | 内容                          | 対策                                                                         |
| ------------- | --------------------------- | -------------------------------------------------------------------------- |
| AI誤分類         | アイデアがタスク化される、バグが要望扱いになる     | AI結果は候補に限定。自動登録しない。根拠引用と信頼度を必須表示                                           |
| 機密情報送信        | APIキー、トークン、顧客情報、未公開仕様をAIに送る | MVPではAI送信なし。将来は送信前確認、`no_ai`、キーワード警告、赤字注意                                  |
| 原文喪失          | 分類時に元メモが消える                 | 原文を未整理メモとして必ず保存し、確定データから参照リンクを残す                                           |
| JSON不正        | ChatGPT出力が壊れている             | 解析失敗しても保存。エラーは静的に表示し、修正方法を示す                                               |
| 過剰な自動化        | ユーザーがAI候補を無確認で信じる           | 登録前確認を必須化。低信頼度は初期チェックを外す                                                   |
| 重複タスク増加       | 同じ壁打ちから似たタスクが複数できる          | MVPでは警告なしでも可。将来、タイトル・checksum・類似度で重複候補                                     |
| プロンプトインジェクション | 貼り付け本文内に「前の指示を無視して」等が含まれる   | 貼り付け内容は命令ではなくデータとして扱う。将来AI分類ではサーバー側でスキーマ検証                                 |
| API保持・監査      | 外部AIへ送った情報の扱いが不透明になる        | OpenAI APIのデータ保持仕様を踏まえ、`store: false`、送信ログ最小化、AI利用設定を明示する。([OpenAI開発者][2]) |
| スマホで確認が面倒     | 候補確認が長くなる                   | 候補カードを短くし、詳細は展開。重要な判断項目は初期表示                                               |
| UI独自仕様化       | モーダルやタブでDADS未整備領域に入りすぎる     | MVPは専用ページ中心。未整備コンポーネントを使う場合は暫定案扱い                                          |

## 6. 誤分類を防ぐUX

誤分類を防ぐUXは、次を必須にする。

* 「AIが登録しました」ではなく「分類候補を作りました」と表現する。
* 候補ごとに根拠引用を表示する。
* `confidence: low`は初期状態で登録チェックを外す。
* 分類先はユーザーが変更できる。
* 「タスクにする」「バグにする」「未整理に戻す」を候補単位で選べる。
* 分類後も元の未整理メモへ戻れる。
* 保存ボタンをdisabledにしない。未選択で押された場合は、何を選べばよいか静的エラーで示す。DADSではdisabledを安易に使わず、押された後に必要な作業を案内する考え方が示されている。

## 7. プライバシー上の注意点

AIに送ってよい情報は、原則として次に限定する。

* 公開リポジトリに既に含まれる情報。
* 抽象化した仕様メモ。
* 個人名・顧客名・秘密情報を除いた課題説明。
* ユーザーが明示的に送信許可した未整理メモ。
* 送信前にNextPatchが表示した内容。

AIに送らない方がよい情報は次。

* APIキー、アクセストークン、秘密鍵、Cookie、認証ヘッダー。
* 顧客情報、患者情報、個人情報、メールアドレス一覧。
* 非公開リポジトリの機密コード全文。
* セキュリティ脆弱性の詳細な攻撃手順。
* 未公開の事業計画、契約情報、価格交渉情報。
* 社内URL、内部IP、インフラ構成、DB接続情報。
* ライセンス上外部送信できないログやドキュメント。

ユーザー向け注意文は、クイック登録画面に常時表示する。

> ChatGPTの内容を貼り付ける前に、APIキー、トークン、個人情報、顧客情報、未公開の機密仕様が含まれていないか確認してください。AI分類を使う場合、選択した内容が外部AIサービスへ送信されることがあります。

# 未決定事項

* MVPの分類先を5分類に絞るか、`future_plan`と`note`を初期から含めるか。推奨は初期から含める。理由は、将来構想と分類不能メモを無理にタスク化しないため。
* `privacy_level`の初期値を`normal`にするか、非公開リポジトリでは`confidential`にするか。推奨は、非公開リポジトリでは`confidential`初期値。
* ChatGPT標準プロンプトをNextPatch内のヘルプに置くか、クイック登録画面からコピーできるようにするか。推奨は後者。
* 将来AI分類で使うモデル、費用上限、1メモあたりの最大文字数。
* AI分類のログをどこまで残すか。推奨は、送信本文そのものではなく、memo_id、schema_version、model、実行時刻、結果IDを残す。
* API連携をGPT Actionsで始めるか、Apps SDK/MCPまで含めるか。推奨は、まずNextPatch側のREST APIを整え、後からGPT Actionsを検討する。
* チーム利用時に、AI送信をワークスペース管理者が禁止できるようにするか。推奨は、管理者による一括禁止を可能にする。

# 統括担当へ渡す要点

MVPでは、NextPatch本体にAI分類を入れず、**手動貼り付け＋未整理メモ原文保存＋JSON/Markdown解析＋分類候補確認**で進めるべきである。

正規フォーマットは`nextpatch.import.v1`のJSONとし、Markdownは補助形式にする。ChatGPTには標準プロンプトでこのJSONを出させる。

未整理メモは原本として必ず保存し、分類候補・確定タスク等とは分離する。これにより、誤分類、解析失敗、将来AI分類の再実行に耐えられる。

将来AI分類は、ユーザーの明示操作、送信前プライバシー確認、Structured OutputsによるJSON Schema、確認画面での人間承認を必須条件にする。OpenAI APIを使う場合はResponses APIを第一候補にする。([OpenAI開発者][1])

分類確認画面は、DADS未整備のモーダルやステップUIに依存せず、専用ページを基本にする。モーダル案は暫定案扱い。

最後に、標準プロンプト案をそのままNextPatchのヘルプまたはクイック登録画面の「ChatGPTに貼るプロンプト」として組み込む。

# ChatGPTで壁打ちした内容をNextPatchに貼り付けるための標準プロンプト案

以下をChatGPTでの壁打ち後に貼り付けて使う。

````text
ここまでの会話内容を、開発管理アプリ NextPatch に貼り付けるために整理してください。

目的:
- 開発中アプリまたはGitHubリポジトリの管理に使います。
- タスク、バグ、アイデア、参考サービス、採用候補技術、将来構想、未整理メモに分けたいです。
- 判断できないものは無理に分類せず、未整理メモに入れてください。
- 内容を捏造しないでください。会話内に根拠がない項目は作らないでください。
- APIキー、トークン、秘密鍵、個人情報、顧客情報、社内機密に見える内容は [REDACTED] に置き換えてください。
- 各項目には、可能な範囲で根拠となる短い引用または要約を入れてください。
- すべての項目は NextPatch 側で人間が確認する前提なので、needs_review は true にしてください。

出力形式:
1. 最初に短いMarkdown要約を書いてください。
2. その後に、NextPatchが読み取れるJSONを ```json のコードブロックで出してください。
3. JSON以外のコードブロックは使わないでください。
4. JSONは次の schema_version を使ってください: nextpatch.import.v1

分類先:
- task: 実行すべき作業、調査、実装、修正
- bug: 不具合、期待結果と実際結果の差、再現手順があるもの
- idea: 新機能案、改善案、まだ実装確定でないもの
- reference_service: 参考サービス、競合、UI/UX参考、料金や機能の参考
- tech_candidate: 採用候補の技術、ライブラリ、API、設計方式
- future_plan: MVP後または将来構想
- note: 分類不能、文脈メモ、判断保留

JSONの形:
{
  "schema_version": "nextpatch.import.v1",
  "source": {
    "tool": "ChatGPT",
    "conversation_title": "",
    "captured_at": "YYYY-MM-DD",
    "privacy_note": "機密情報は含めない。不明な機密情報は[REDACTED]に置換。"
  },
  "repository_hint": {
    "owner": "",
    "name": "",
    "branch": "",
    "area": ""
  },
  "summary": "",
  "items": [
    {
      "kind": "task | bug | idea | reference_service | tech_candidate | future_plan | note",
      "title": "",
      "body": "",
      "priority": "low | medium | high | unknown",
      "labels": [],
      "evidence": "",
      "confidence": "low | medium | high",
      "needs_review": true
    }
  ],
  "unclassified_notes": [
    {
      "title": "",
      "body": "",
      "reason": "分類保留の理由"
    }
  ]
}

追加ルール:
- 同じ内容を重複して登録しないでください。
- タスク化できるほど具体的でないものは idea または note にしてください。
- バグは、再現条件・期待結果・実際結果が不足している場合、bug ではなく note にして不足情報を書いてください。
- 採用候補技術は、利点だけでなく懸念点も body に含めてください。
- 参考サービスは、何を参考にするのかを body に含めてください。
- confidence が low の項目は、title の先頭に「要確認: 」を付けてください。
````
