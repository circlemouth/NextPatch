# 担当領域

状態遷移・ダッシュボードロジック。

NextPatch 内で扱う「タスク」「バグ」「アイデア」「採用候補」「参考サービス」「未整理メモ」を、種別ごとに自然な状態名で扱いつつ、ダッシュボードでは横断的に「未完了」「完了済み」「今やるべき」を判断できるようにするための設計判断を整理する。

# 前提理解

NextPatch は、GitHub リポジトリや開発中アプリ単位で、課題、進捗、バグ、実装予定、将来構想、採用技術、参考サービス、ChatGPT との壁打ちメモを整理し、「今やるべきこと」をすぐ把握するためのアプリである。

状態名は種別ごとに自然な言葉を使うべきである。一方、ダッシュボードや集計では、全種別を共通の内部判定に変換する必要がある。そのため、MVP では次の二層構造を推奨する。

| 層           | 役割                                                 |
| ----------- | -------------------------------------------------- |
| 表示用状態       | 種別ごとの自然な状態名。例：タスクの「作業中」、バグの「再現済み」、アイデアの「採用済み」      |
| 内部ライフサイクル判定 | `open` / `closed` / `completed` / `archived` の横断判定 |

UI では、DADS 参照ルールに従い、重要情報を折りたたみで隠さず、サービス内で状態色・ボタン優先度・余白・見出し構造を一貫させる。タブ、テーブルコントロール、ページナビゲーション、モーダル等は DADS 側で詳細ガイドライン未整備のため、使う場合は「暫定案」として扱う。

GitHub 連携を想定する場合、GitHub Issues は `open` / `closed` の状態を持ち、close 理由として `completed`、`not_planned`、`duplicate`、`reopened` などを扱う。NextPatch 側でも「完了」と「対応しない終了」を分けるべきである。確認日：2026-04-20。([GitHub Docs][1])

GitHub Projects は Issue、Pull Request、メモ的なアイデアを追跡し、テーブル、ボード、ロードマップ、カスタムフィールドで柔軟に管理できる。NextPatch も GitHub の状態をそのまま固定採用するのではなく、NextPatch 固有のメタデータを持つ前提にする。確認日：2026-04-20。([GitHub Docs][2])

# 決定すべき論点

1. 種別ごとの表示状態をどこまで分けるか。
2. どの状態を `open`、`closed`、`completed`、`archived` と判定するか。
3. `completedAt` を自動設定する状態をどれにするか。
4. `completedAt` をクリアする条件をどう定義するか。
5. 「保留」「却下」「中止」「再現不能」を完了扱いにするか。
6. 優先度とバグ重要度を分けるか。
7. 「今やるべき度」を手動指定にするか、ルールベースにするか。
8. ダッシュボードを単一ランキングにするか、意味別セクションに分けるか。
9. 実稼働中リポジトリのバグをどの程度優先するか。
10. 次バージョン対象タスクをどの程度優先するか。
11. 未整理メモをダッシュボードに常時出すか、条件付きにするか。
12. 最近完了した項目をどの期間まで表示するか。

# 選択肢比較

## 状態設計の選択肢

| 案                   | 内容                        | 長所         | 短所                  | 判断  |
| ------------------- | ------------------------- | ---------- | ------------------- | --- |
| A. 全種別で共通状態に統一      | すべてを「未着手 / 作業中 / 完了」などで統一 | 実装が単純      | バグ、アイデア、採用候補の意味が潰れる | 不採用 |
| B. 種別ごとに完全独立        | 種別ごとに状態名も判定も個別実装          | 表現は自然      | ダッシュボード横断表示が複雑化     | 不採用 |
| C. 表示状態は種別別、内部判定は共通 | 表示は自然名、内部で `open` 等へ変換    | 自然さと集計性を両立 | 状態マッピング表が必要         | 採用  |

## 完了判定の選択肢

| 案                        | 内容                  | 長所         | 短所                        | 判断  |
| ------------------------ | ------------------- | ---------- | ------------------------- | --- |
| A. closed はすべて completed | 却下・中止・再現不能も完了扱い     | 集計が簡単      | 「やり切ったこと」と「やらない判断」が混ざる    | 不採用 |
| B. completed は成功完了のみ     | タスク完了、バグ解決、採用決定などのみ | 達成量が正しく見える | closed と completed の区別が必要 | 採用  |
| C. completedAt は手動入力のみ   | 自動設定しない             | ユーザー制御が強い  | 入力漏れが起きる                  | 不採用 |

## ダッシュボード構成の選択肢

| 案                | 内容                       | 長所               | 短所                         | 判断        |
| ---------------- | ------------------------ | ---------------- | -------------------------- | --------- |
| A. 全項目を単一ランキング表示 | スコア順に一覧                  | 実装が単純            | なぜ表示されたか分かりにくい             | 不採用       |
| B. 種別ごとにタブ表示     | タスク、バグ、メモ等で切替            | 整理しやすい           | タブは DADS で詳細未整備、重要情報が隠れやすい | MVP では不採用 |
| C. 意味別セクション表示    | 今やるべき、重大バグ、次バージョン、未整理メモ等 | 判断しやすく、重要情報を隠さない | セクション設計が必要                 | 採用        |

# 推奨案

推奨案は、**「種別別の表示状態」＋「共通ライフサイクル判定」＋「ルールベースのダッシュボード」**である。

内部的には次の 4 判定を持たせる。

| 判定          | 意味               |
| ----------- | ---------------- |
| `open`      | まだ判断・作業・整理が残っている |
| `closed`    | 現時点で追加作業が不要      |
| `completed` | 目的を達成して肯定的に完了した  |
| `archived`  | 通常画面から隠す対象       |

重要なのは、`closed` と `completed` を分けることである。
「却下」「中止」「再現不能」「対応不要」「重複」は `closed` ではあるが、`completed` ではない。これにより、「最近完了したこと」と「対応しないと決めたこと」が混ざらない。

`archived` は状態そのものではなく、原則として `archivedAt` による別フラグにする。理由は、完了済みタスクをアーカイブしても「完了済み」という事実は残すべきだからである。

実稼働中リポジトリの重大バグは、MVP でも明確に優先する。
次バージョン対象タスクも優先するが、実稼働中リポジトリの重大バグより上には置かない。

優先順位は次の順を推奨する。

1. 実稼働中リポジトリの重大バグ
2. 手動で P0 指定された項目
3. 次バージョン対象で未完了のタスク・バグ
4. 期限が近い項目
5. 未整理で放置されているメモ
6. 通常の未完了項目

# MVPに含めるべき内容

## 1. 種別ごとの状態表

### タスク

| 状態     | 説明                     |
| ------ | ---------------------- |
| 未整理    | まだ作業内容・粒度・優先度が整理されていない |
| 着手待ち   | 作業可能だが未着手              |
| 作業中    | 実装・調査・文書化などを進めている      |
| レビュー待ち | 確認、レビュー、動作確認待ち         |
| ブロック中  | 依存作業や外部要因で進められない       |
| 保留     | 今は進めないが、将来再開の可能性がある    |
| 完了     | 目的を達成した                |
| 中止     | 途中でやめる判断をした            |
| 重複     | 他項目に統合された              |

### バグ

| 状態        | 説明                |
| --------- | ----------------- |
| 未確認       | 報告されたが再現・影響確認が未完了 |
| 再現確認待ち    | 再現手順や環境情報が不足している  |
| 再現済み      | 不具合として確認済み        |
| 調査中       | 原因調査中             |
| 修正中       | 修正作業中             |
| 修正済み・確認待ち | 修正は入ったが検証が未完了     |
| 解決済み      | 修正確認まで完了          |
| 再現不能      | 再現できず、現時点では追跡終了   |
| 仕様通り      | 不具合ではなく仕様として扱う    |
| 対応不要      | 影響や優先度から対応しない     |
| 重複        | 他のバグ項目に統合         |
| 保留        | 今は対応しないが、再開可能性がある |

### アイデア

| 状態     | 説明                |
| ------ | ----------------- |
| 未整理    | 思いつき、壁打ちメモの段階     |
| 検討中    | 価値、対象ユーザー、実現性を検討中 |
| 有望     | 将来実装候補として残す価値がある  |
| タスク化済み | 具体的なタスクへ変換済み      |
| 採用済み   | 方針として採用決定済み       |
| 保留     | 今は判断しない           |
| 却下     | 採用しない             |
| 重複     | 既存アイデアに統合         |

### 採用候補

| 状態   | 説明                       |
| ---- | ------------------------ |
| 未調査  | 技術・ライブラリ・サービス候補として記録しただけ |
| 調査中  | 公式情報、制約、料金、互換性などを確認中     |
| 比較中  | 他候補と比較中                  |
| PoC中 | 小さく試している                 |
| 採用決定 | 採用する判断が完了                |
| 不採用  | 採用しない判断が完了               |
| 保留   | 今は判断しない                  |
| 置換済み | 以前採用したが、別候補へ移った          |

### 参考サービス

| 状態      | 説明                    |
| ------- | --------------------- |
| 未確認     | URL や名前を保存しただけ        |
| 調査中     | UI、機能、価格、情報設計などを確認中   |
| 要点整理済み  | 参考にする観点を整理済み          |
| アイデア化済み | NextPatch 内のアイデアへ変換済み |
| 採用候補化済み | 技術・機能候補へ変換済み          |
| 見送り     | 参考にしない判断をした           |
| リンク切れ   | 参照不能になった              |
| 保留      | 後で見る                  |

### 未整理メモ

| 状態      | 説明                       |
| ------- | ------------------------ |
| 未整理     | ChatGPT 壁打ち、思いつき、断片メモのまま |
| 整理中     | 関連項目への分解・分類中             |
| アイテム化済み | タスク、バグ、アイデア等へ変換済み        |
| 記録のみ    | 今後の作業対象ではなく、参考記録として残す    |
| 保留      | 後で整理する                   |
| 破棄      | 残す必要がない                  |
| 重複      | 他メモや項目に統合                |

| 任意の open 状態 | リンク切れ   | 参照不能        |
| 任意の open 状態 | 保留      | 後で見る        |

### 未整理メモ遷移

| 遷移元         | 遷移先     | 主な条件            |
| ----------- | ------- | --------------- |
| 未整理         | 整理中     | 分類・分解開始         |
| 整理中         | アイテム化済み | タスク、バグ、アイデア等へ変換 |
| 整理中         | 記録のみ    | 作業対象にしないが保存     |
| 未整理 / 整理中   | 保留      | 後で整理            |
| 未整理 / 整理中   | 破棄      | 不要              |
| 任意の open 状態 | 重複      | 他メモへ統合          |

## 4. completedAt 自動設定ルール

### 自動設定する条件

`completedAt` は、次の条件をすべて満たすときに自動設定する。

| 条件 | 内容                       |
| -- | ------------------------ |
| 1  | 遷移前の `completed` が false |
| 2  | 遷移後の `completed` が true  |
| 3  | `completedAt` が未設定       |
| 4  | アーカイブ操作ではなく、状態変更による完了である |

対象状態は次の通り。

| 種別     | completedAt を設定する状態    |
| ------ | ---------------------- |
| タスク    | 完了                     |
| バグ     | 解決済み                   |
| アイデア   | タスク化済み、採用済み            |
| 採用候補   | 採用決定                   |
| 参考サービス | 要点整理済み、アイデア化済み、採用候補化済み |
| 未整理メモ  | アイテム化済み、記録のみ           |

GitHub から取り込む場合、GitHub Issue が `closed` かつ close 理由が `completed` 相当なら、GitHub 側の close 時刻を `completedAt` に入れる。`not_planned`、`duplicate` 相当なら `closedAt` は入れてよいが、`completedAt` は入れない。

### 自動設定しない条件

| 状態    | 理由                         |
| ----- | -------------------------- |
| 保留    | まだ open であり、作業・判断が残っている    |
| 却下    | 肯定的な完了ではない                 |
| 中止    | 目的達成ではない                   |
| 再現不能  | 解決ではなく追跡終了                 |
| 対応不要  | 作業完了ではなく判断終了               |
| 重複    | 別項目へ統合されたため、この項目自体は完了扱いしない |
| リンク切れ | 参照不能による終了であり、達成ではない        |
| 破棄    | メモ処理としては閉じたが、成果ではない        |

### completedAt をクリアする条件

| 条件                             | クリアするか | 理由              |
| ------------------------------ | -----: | --------------- |
| 完了状態から open 状態へ戻す              |  クリアする | 再オープン後は未完了として扱う |
| 解決済みバグを再現済みに戻す                 |  クリアする | 解決済みではなくなる      |
| 採用済みアイデアを検討中へ戻す                |  クリアする | 採用判断が未確定になる     |
| 完了状態のままアーカイブ                   | クリアしない | 完了の事実は残す        |
| アーカイブ解除                        | クリアしない | 表示状態の変更に過ぎない    |
| completed=false の closed 状態へ変更 |  クリアする | 完了ではなくなる        |
| completedAt をユーザーが手動修正         |   上書き可 | 履歴には残す          |

MVP では、`completedAt` とは別に `closedAt` を持つことを推奨する。
これにより、「完了した日」と「対応しないと決めた日」を分離できる。

# MVPでは後回しにする内容

次は MVP では後回しにする。

| 項目                | 理由                                |
| ----------------- | --------------------------------- |
| AI による自動優先度判定     | 説明可能性が低く、初期実装の検証が難しい              |
| 複雑なスコアリングのユーザー調整  | まず固定ルールで運用感を確認すべき                 |
| 種別ごとの完全カスタム状態定義   | 柔軟だが、初期設計が崩れやすい                   |
| スイムレーン型ボード        | DADS でボード専用ルールがないため、まず一覧・セクションで十分 |
| タブによるダッシュボード切替    | DADS でタブが未整備、重要情報が隠れやすい           |
| 高度なフィルタビルダー       | テーブルコントロールが DADS 未整備のため暫定扱い       |
| 自然言語からの状態推定       | 誤判定がダッシュボードの信頼性を下げる               |
| 複数リポジトリ横断の依存関係グラフ | 価値は高いが MVP には重い                   |
| 完了率・ベロシティ分析       | 状態運用が安定してからでよい                    |

## 優先度の定義

優先度は、種別横断の「処理順」を表す。
バグの被害規模とは分ける。

| 優先度 | 表示名 | 定義                              |
| --- | --- | ------------------------------- |
| P0  | 今すぐ | 今日扱うべき。放置するとリリース、実稼働、重要作業に支障が出る |
| P1  | 高   | 次バージョンまたは直近の重要作業に入れる            |
| P2  | 通常  | 余裕があれば進める通常項目                   |
| P3  | 低   | いつか、参考、改善余地                     |

## バグ重要度の定義

バグ重要度は、バグの影響範囲・被害の大きさを表す。
優先度とは別に持つ。

| 重要度 | 表示名 | 定義                        |
| --- | --- | ------------------------- |
| S0  | 致命的 | データ消失、セキュリティ、実稼働停止、主要操作不能 |
| S1  | 重大  | 中核機能が壊れており、回避策がないか困難      |
| S2  | 中   | 主要機能に支障があるが回避策がある         |
| S3  | 軽微  | 一部表示、補助機能、限定条件の問題         |
| S4  | 微小  | 文言、見た目、ほぼ影響のない問題          |

# 画面設計への影響

ダッシュボードは、MVP ではタブではなく、見出しつきセクションで構成する。
DADS では重要情報を隠さないことが重視されるため、最初の画面で「今やるべき」「重大バグ」「次バージョン対象」は見えるようにする。

## 5. ダッシュボード表示セクション案

推奨順は次の通り。

| 表示順 | セクション          | 表示条件                                      |             件数目安 |
| --: | -------------- | ----------------------------------------- | ---------------: |
|   1 | 今やるべき          | open、非アーカイブ、保留以外、今やるべき度が高い                |             5〜10 |
|   2 | 実稼働中リポジトリの重大バグ | repo が active_production、bug、open、S0〜S2   |                5 |
|   3 | 次バージョン対象       | targetVersionId が repo.nextVersionId、open |             5〜10 |
|   4 | ブロック中・確認待ち     | ブロック中、再現確認待ち、レビュー待ち                       |             5〜10 |
|   5 | 未整理メモ          | memo、未整理、条件を満たすもの                         |                5 |
|   6 | 最近完了した項目       | completedAt が直近 7 日以内                     |             5〜10 |
|   7 | 保留中            | 保留状態のうち更新が古いもの                            | 任意、初期は折りたたまず下部表示 |

## 未整理メモをダッシュボードに出す条件

未整理メモは、すべてを「今やるべき」に出すとノイズが増える。
MVP では専用セクションに出し、「今やるべき」への混入は条件付きにする。

| 表示場所       | 条件                                                            |
| ---------- | ------------------------------------------------------------- |
| 未整理メモセクション | `type = memo`、`status = 未整理`、`archivedAt = null`              |
| 今やるべきセクション | 上記に加えて、`isPinned = true`、または作成から 3 日以上経過、または次バージョン対象リポジトリに紐づく |
| 表示しない      | `status = 保留 / 記録のみ / 破棄 / 重複`、または `archivedAt != null`       |

## 最近完了した項目の表示条件

| 条件          | 内容                    |
| ----------- | --------------------- |
| completed   | true                  |
| completedAt | 現在日時から 7 日以内          |
| archived    | false                 |
| 表示件数        | 最大 10 件               |
| 並び順         | completedAt の降順       |
| 除外          | 却下、中止、再現不能、対応不要、重複、破棄 |

「最近完了した項目」は達成感と進捗確認のために有効だが、最上位には置かない。
最上位はあくまで「今やるべき」にする。

## UI コンポーネント上の注意

状態バッジ、優先度バッジ、重要度バッジは DADS に専用ルールがないため、MVP では「暫定案」とする。
ただし、色だけで意味を伝えず、ラベル文字を必ず併記する。コントラストも DADS ルールに従う。

テーブル表示は構造化データの表示として使えるが、並び替え・フィルタ・ページングなどのテーブルコントロールは DADS 未整備のため、MVP では簡素なセレクトボックス、チェックボックス、検索入力で代替する。

## 状態変更 API の責務

`PATCH /items/{id}/status` は、単に状態文字列を保存するだけではなく、次を実行する。

1. 種別に対して有効な状態か確認する。
2. 許可された遷移か確認する。
3. 遷移前後の `completed` 判定を比較する。
4. 必要に応じて `completedAt` を設定またはクリアする。
5. `closedAt` を設定またはクリアする。
6. 状態変更履歴を保存する。
7. ダッシュボードキャッシュがある場合は無効化する。

# リスクと対策

| リスク                           | 内容                             | 対策                                     |
| ----------------------------- | ------------------------------ | -------------------------------------- |
| completed と closed が混同される     | 却下や再現不能が完了扱いされる                | UI 上も「完了済み」と「終了済み」を分ける                 |
| 状態が多すぎて入力が面倒                  | ユーザーが状態更新しなくなる                 | MVP では状態数を抑え、よく使う遷移をボタン化               |
| ダッシュボードがノイズ化する                | 未整理メモや低優先度項目で埋まる               | セクション分け、件数上限、保留の抑制                     |
| 実稼働バグが常に最上位を占有する              | 他の重要作業が見えなくなる                  | 「重大バグ」専用セクションと「今やるべき」を分ける              |
| 次バージョン対象が多すぎる                 | どれからやるか分からない                   | priority、dueDate、status で二次ソート         |
| GitHub の状態と NextPatch の状態がずれる | GitHub は open/closed が中心で粒度が粗い | NextPatch 側を主状態、GitHub は同期元情報として扱う     |
| アーカイブで履歴が失われる                 | 完了実績が見えなくなる                    | `archivedAt` は表示制御のみ。`completedAt` は残す |
| スコアがブラックボックス化する               | なぜ「今やるべき」か分からない                | MVP はルールベース、将来も理由ラベルを返す                |

# 未決定事項

1. `closedAt` を MVP から正式採用するか。
   推奨は採用。`completedAt` だけでは却下・中止・再現不能の終了日を表せない。

2. 「記録のみ」メモを `completed=true` とするか。
   推奨は true。メモ整理というワークフローは完了しているため。

3. 「タスク化済み」アイデアを `completed=true` とするか。
   推奨は true。ただし、実装完了ではなく「アイデア処理の完了」と明示する。

4. 「参考サービス」の `要点整理済み` を completed にするか。
   推奨は true。参考調査としては完了しているため。そこから派生したアイデアや採用候補は別項目で管理する。

5. 保留項目を dashboard 下部に出すか。
   推奨は MVP では下部に少数表示。完全非表示にすると忘れられる。

6. 優先度の初期値をどうするか。
   推奨は P2。バグで S0/S1 の場合のみ P1 相当の表示優先を自動付与してもよいが、`priority` 自体は勝手に書き換えない。

7. GitHub Issue close reason との同期方向。
   推奨は、GitHub → NextPatch 取込時は変換、NextPatch → GitHub 更新時はユーザー確認後に反映。

# 統括担当へ渡す要点

* 状態名は種別ごとに自然なものを使う。
* ただし内部判定は `open` / `closed` / `completed` / `archived` に正規化する。
* `closed` と `completed` は分ける。
* `completedAt` は、肯定的な完了状態に入ったときだけ自動設定する。
* 却下、中止、再現不能、対応不要、重複、破棄は `closed=true` だが `completed=false`。
* アーカイブは状態ではなく `archivedAt` による表示制御とする。
* 優先度 P0〜P3 とバグ重要度 S0〜S4 は別フィールドにする。
* MVP の「今やるべき度」は AI 判定ではなく、説明可能な固定ルールにする。
* 実稼働中リポジトリの重大バグは、次バージョン対象タスクより優先する。
* 未整理メモは専用セクションに出し、「今やるべき」には条件付きで出す。
* 最近完了した項目は `completedAt` 直近 7 日以内、最大 10 件。
* UI はタブ依存にせず、重要セクションを初期表示する。
* DADS 未整備の UI コンポーネント、特にタブ、テーブルコントロール、状態バッジは暫定案として扱う。

## 6. 今やるべき度の MVP ルール

MVP では数値スコアではなく、まず `tier` で分類する。
分類理由を UI に出せるようにする。

| Tier | 表示上の意味 | 条件                               |
| ---: | ------ | -------------------------------- |
|    0 | 固定表示   | `isPinned = true` かつ open        |
|    1 | 最優先    | 実稼働中リポジトリの S0/S1 バグ、または P0       |
|    2 | 高優先    | 次バージョン対象の P0/P1、実稼働中リポジトリの S2 バグ |
|    3 | 直近対応   | dueDate が 3 日以内、レビュー待ち、修正済み・確認待ち |
|    4 | 整理必要   | 未整理で 3 日以上経過したタスク、アイデア、メモ        |
|    5 | 通常     | その他の open 項目                     |
|   除外 | 表示しない  | closed、archived、保留、破棄、重複         |

同じ Tier 内の並び順は次を推奨する。

1. 実稼働中リポジトリか。
2. バグ重要度が高いか。
3. 優先度が高いか。
4. 次バージョン対象か。
5. 期限が近いか。
6. 更新日時が新しいか。
7. 作成日時が古いか。

## ダッシュボード表示ロジック疑似コード

```pseudo
function deriveFlags(item):
    def = STATUS_DEFINITIONS[item.type][item.status]

    return {
        open: def.isOpen and item.archivedAt == null,
        closed: def.isClosed,
        completed: def.isCompleted,
        archived: item.archivedAt != null
    }


function onStatusChange(item, nextStatus, now):
    prevFlags = deriveFlags(item)

    assert isValidStatus(item.type, nextStatus)
    assert isAllowedTransition(item.type, item.status, nextStatus)

    item.status = nextStatus
    item.statusChangedAt = now

    nextFlags = deriveFlags(item)

    if prevFlags.completed == false and nextFlags.completed == true:
        if item.completedAt == null:
            item.completedAt = now

    if prevFlags.completed == true and nextFlags.completed == false:
        item.completedAt = null

    if prevFlags.closed == false and nextFlags.closed == true:
        if item.closedAt == null:
            item.closedAt = now

    if prevFlags.closed == true and nextFlags.closed == false:
        item.closedAt = null

    item.updatedAt = now

    saveStatusHistory(item.id, prevStatus, nextStatus, now)
    save(item)


function isProductionRepo(repo):
    return repo.productionStatus == "active_production"


function isNextVersionItem(item, repo):
    return item.targetVersionId != null
        and repo.nextVersionId != null
        and item.targetVersionId == repo.nextVersionId


function isOnHold(item):
    return item.status in [
        "保留"
    ]


function isOpenActionable(item):
    flags = deriveFlags(item)

    if flags.archived:
        return false

    if flags.closed:
        return false

    if isOnHold(item):
        return false

    return flags.open


function getUrgencyTier(item, repo, now):
    flags = deriveFlags(item)

    if flags.archived or flags.closed:
        return null

    if isOnHold(item):
        return null

    if item.isPinned:
        return 0

    if item.type == "bug"
       and isProductionRepo(repo)
       and item.bugSeverity in ["S0", "S1"]:
        return 1

    if item.priority == "P0":
        return 1

    if item.type == "bug"
       and isProductionRepo(repo)
       and item.bugSeverity == "S2":
        return 2

    if isNextVersionItem(item, repo)
       and item.priority in ["P0", "P1"]:
        return 2

    if item.dueDate != null
       and item.dueDate <= now + 3 days:
        return 3

    if item.status in ["レビュー待ち", "修正済み・確認待ち", "再現確認待ち"]:
        return 3

    if item.status == "未整理"
       and item.createdAt <= now - 3 days:
        return 4

    return 5


function compareDashboardItems(a, b, repos, now):
    repoA = repos[a.repoId]
    repoB = repos[b.repoId]

    tierA = getUrgencyTier(a, repoA, now)
    tierB = getUrgencyTier(b, repoB, now)

    sort by tier ascending

    sort by isProductionRepo(repo) descending

    if both are bugs:
        sort by bugSeverity ascending
        // S0 before S1 before S2...

    sort by priority ascending
    // P0 before P1 before P2 before P3

    sort by isNextVersionItem descending

    sort by dueDate ascending, nulls last

    sort by updatedAt descending

    sort by createdAt ascending


function buildDashboard(items, repositories, now):
    openItems = []
    productionBugs = []
    nextVersionItems = []
    blockedOrWaiting = []
    unorganizedMemos = []
    recentCompleted = []
    onHoldItems = []

    for item in items:
        repo = repositories[item.repoId]
        flags = deriveFlags(item)

        if flags.archived:
            continue

        if flags.completed
           and item.completedAt != null
           and item.completedAt >= now - 7 days:
            recentCompleted.append(item)

        if flags.closed:
            continue

        if isOnHold(item):
            onHoldItems.append(item)
            continue

        if item.type == "bug"
           and isProductionRepo(repo)
           and item.bugSeverity in ["S0", "S1", "S2"]:
            productionBugs.append(item)

        if isNextVersionItem(item, repo):
            nextVersionItems.append(item)

        if item.status in [
            "ブロック中",
            "レビュー待ち",
            "再現確認待ち",
            "修正済み・確認待ち"
        ]:
            blockedOrWaiting.append(item)

        if item.type == "memo"
           and item.status == "未整理":
            unorganizedMemos.append(item)

        tier = getUrgencyTier(item, repo, now)

        if tier != null and tier <= 4:
            if item.type != "memo":
                openItems.append(item)
            else if item.isPinned
                 or item.createdAt <= now - 3 days
                 or isNextVersionItem(item, repo):
                openItems.append(item)

    sort openItems by compareDashboardItems
    sort productionBugs by severity, priority, updatedAt desc
    sort nextVersionItems by priority, status, dueDate, updatedAt desc
    sort blockedOrWaiting by updatedAt desc
    sort unorganizedMemos by createdAt asc
    sort recentCompleted by completedAt desc
    sort onHoldItems by updatedAt asc

    return {
        sections: [
            {
                id: "now",
                title: "今やるべき",
                items: first 10 of openItems
            },
            {
                id: "production_bugs",
                title: "実稼働中リポジトリの重大バグ",
                items: first 5 of productionBugs
            },
            {
                id: "next_version",
                title: "次バージョン対象",
                items: first 10 of nextVersionItems
            },
            {
                id: "blocked_or_waiting",
                title: "ブロック中・確認待ち",
                items: first 10 of blockedOrWaiting
            },
            {
                id: "unorganized_memos",
                title: "未整理メモ",
                items: first 5 of unorganizedMemos
            },
            {
                id: "recent_completed",
                title: "最近完了した項目",
                items: first 10 of recentCompleted
            },
            {
                id: "on_hold",
                title: "保留中",
                items: first 5 of onHoldItems
            }
        ]
    }
```
