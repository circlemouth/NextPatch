# 担当領域

GitHub連携・外部連携。

対象は、NextPatch が「リポジトリ単位で課題・進捗・バグ・実装予定・将来構想・採用技術・参考サービス・ChatGPTメモを整理し、今やるべきことを把握する」ために、GitHub とどこまで接続するかを決める領域です。実装コードではなく、MVP前の設計判断として整理します。

# 前提理解

NextPatch の主役は GitHub そのものの再実装ではなく、開発中のリポジトリごとに散らばる判断材料を集約し、「次に何をするか」を分かりやすくすることです。したがって、MVPで最初から GitHub API と深く同期するより、まずはリポジトリURLを起点に、手入力の WorkItem とメモを確実に管理できる状態を優先すべきです。

この回答では、会話内で提示された NextPatch の概要、DADS参照UIルール、GitHub公式ドキュメントを前提にしています。NextPatch要件定義書と「開発着手前に決定すべき内容の検討メモ」の本文は、このセッションで確認できた検索範囲では見つかりませんでした。そのため、それらについてはユーザーが提示した概要を前提情報として扱います。

UIについては DADS ルールを優先します。特に、入力フォームではラベル・サポートテキスト・具体的なエラー表示を使い、プレースホルダー依存や安易な disabled を避けること、重要情報を隠さないこと、ボタンの優先度を画面内で一貫させることを前提にします。DADSでガイドライン準備中とされているモーダル、タブ、テーブルコントロール等は、NextPatch内では「暫定案」として扱うべきです。   

Web参照は GitHub公式ドキュメントを確認しました。確認日は 2026年4月20日です。GitHub REST API には、認証ユーザーのリポジトリ一覧取得、Issue、Pull Request、Release、Tag 取得・管理に関するエンドポイントがあります。([GitHub Docs][1])

# 決定すべき論点

最初に決めるべき論点は、MVPの連携レベルをどこに置くかです。結論から言うと、MVPは **Level 1: URL解析** までにするのが妥当です。Level 0の単なるURL手入力だけでは将来のAPI連携に必要な `owner / repo` を構造化できず、後でデータ移行が発生します。一方で、Level 2以上のAPI読み取り連携は、OAuth、GitHub App、トークン保管、権限スコープ、レート制限、プライベートリポジトリ対応を同時に持ち込むため、MVPの主価値に対して重すぎます。

決定すべき主な論点は次のとおりです。

| 論点                | 決める内容                      | 推奨判断                                                    |
| ----------------- | -------------------------- | ------------------------------------------------------- |
| MVPのGitHub連携レベル   | 手入力のみか、URL解析まで行うか          | **Level 1まで採用**                                         |
| GitHub URLの必須性    | リポジトリ作成時に必須にするか            | GitHubリポジトリとして登録する場合は主要入力。将来の非GitHub対応を考え、DB上は nullable |
| owner / repo 抽出   | URLから構造化するか                | **必ず抽出**                                                |
| GitHub API読み取り    | MVPで入れるか                   | **MVPでは入れない**                                           |
| Issue作成           | WorkItemからGitHub Issueを作るか | MVPでは後回し                                                |
| 双方向同期             | Issue変更とWorkItem変更を同期するか   | 後期機能。最初から入れない                                           |
| GitHub Projects連携 | Projectsを取り込むか             | Level 5。MVP対象外                                          |
| OAuth方式           | OAuth AppかGitHub Appか      | 将来は **GitHub App優先**                                    |
| トークン保存            | どこまで保持するか                  | MVPでは保存しない。将来は暗号化・短命トークン・tokenRef管理                     |
| WorkItemとIssueの関係 | 1対1同期か、関連リンクか              | MVPは独立WorkItem＋任意外部リンク。将来は外部対応テーブルで対応                   |

# 選択肢比較

| 選択肢                            | 内容                                   | 利点                             | 欠点                                             | MVP適性      |
| ------------------------------ | ------------------------------------ | ------------------------------ | ---------------------------------------------- | ---------- |
| Level 0: GitHub URL 手入力        | リポジトリURLを文字列として保存するだけ                | 最小実装。トークン不要。安全                 | owner/repoを扱えず、後からAPI連携しづらい                    | △ 最低限としては可 |
| Level 1: URL解析                 | URLから `owner / repo` を抽出し、正規化URLも保持  | 低コストで将来API連携の土台になる。ユーザー入力ミスも減る | パーサ実装とエラー表示が必要                                 | ◎ MVP推奨    |
| Level 2: 読み取り連携                | リポジトリ一覧、Issue、PR、Release/Tagを取得      | 既存GitHub情報を取り込める               | 認証・権限・レート制限・プライバシー対応が必要                        | △ 条件付き後続   |
| Level 3: Issue作成               | NextPatch WorkItemからGitHub Issueを作成  | NextPatch起点でGitHub作業を開始できる     | 書き込み権限、重複作成、通知、失敗時の再実行が難しい                     | × MVP後     |
| Level 4: 双方向同期                 | GitHub IssueとNextPatch WorkItemを相互更新 | 情報の二重管理を減らせる                   | 競合解決、フィールド差分、webhook、監査ログが必要                   | × 後期       |
| Level 5: Projects / Release 連携 | Projectsのビュー・フィールド、Release/Tagと連携    | ロードマップ・リリース管理に発展できる            | GitHub Projectsは独自の計画管理機能で、NextPatchと責務が重なりやすい | × 最後       |

GitHub API面では、Issueは閲覧・管理対象として assignee、comment、label、milestone などを含みます。Pull RequestはREST APIで一覧・取得・編集・作成・マージが可能ですが、PRはIssueの一種でもあり、assigneeやlabelなど共通操作は Issues API 側で扱う必要があります。([GitHub Docs][2])

Issue作成は、GitHub上で「pull access」を持つユーザーが可能ですが、Issuesが無効なリポジトリでは `410 Gone` になり、作成が多すぎると二次レート制限の対象になります。また、fine-grained tokenでは `"Issues" repository permissions (write)` が必要です。([GitHub Docs][3])

Release一覧は通常のGit tagを含まず、通常のタグ一覧は Repository Tags API を使う必要があります。Release取得には公開リソースなら認証なしでも使えますが、draft releaseはpush権限のあるユーザーのみが取得できます。([GitHub Docs][4])

# 推奨案

**MVPでは「Level 1: URL解析」までを採用する。GitHub API読み取り、Issue作成、双方向同期、Projects連携は入れない。**

理由は明確です。NextPatch のMVPで検証すべき価値は、GitHub上の情報を完全同期することではなく、「リポジトリごとに散らばる課題・メモ・構想・判断材料を整理し、今やるべきことを把握できるか」です。GitHub API連携を最初から入れると、OAuth、権限スコープ、トークン保存、private repository、レート制限、同期競合の設計が主作業になり、MVPの検証軸がぶれます。

ただし、Level 0だけでは弱いです。GitHub URLを保存するだけだと、将来 `GET /repos/{owner}/{repo}/issues` や `GET /repos/{owner}/{repo}/pulls` へ自然に接続できません。したがって、MVP時点で `owner` と `repo` は抽出・保存するべきです。GitHubのIssue/Release/TagなどのREST APIは `owner` と `repo` をパスパラメータとして扱うため、ここを最初から構造化しておく価値が高いです。([GitHub Docs][4])

条件付きで別案を選ぶなら、次の判断になります。

| 条件                                                  | 採用してよい案                                          |
| --------------------------------------------------- | ------------------------------------------------ |
| 個人利用・手元の数リポジトリ管理が中心                                 | Level 1で十分                                       |
| 既存Issueが大量にあり、初回取り込みがMVP価値の中心                       | Level 2読み取り連携を検討。ただしOAuth/GitHub App設計をMVP範囲に含める |
| NextPatchからGitHub Issueを作る体験が主要価値                   | Level 3を検討。ただし書き込み権限、重複作成、通知、失敗時再試行を必須設計にする      |
| チームでGitHub Projectsを既に運用しており、NextPatchが補助ダッシュボードになる | Level 5を早めてもよいが、MVPではなく次フェーズ扱い                   |

GitHub認証を後から入れる場合は、OAuth Appより **GitHub App優先** とします。GitHub公式ドキュメントでは、GitHub AppsはOAuth Appsより一般に推奨され、fine-grained permissions、リポジトリ単位のアクセス制御、短命トークンを使えるため、資格情報漏えい時の被害を小さくできます。また、GitHub Appは中央集約されたwebhookを持てる一方、OAuth Appはリポジトリや組織ごとにwebhook設定が必要になります。([GitHub Docs][5])

# MVPに含めるべき内容

MVPに含めるべき GitHub連携は、次の範囲です。

| 項目                      | MVP採用 | 内容                                                     |
| ----------------------- | ----: | ------------------------------------------------------ |
| GitHub URL入力            |     ○ | リポジトリ登録時にURLを貼る                                        |
| URL形式の検証                |     ○ | `github.com/{owner}/{repo}` 系を受け付け、`.git` や末尾 `/` を正規化 |
| owner / repo 抽出         |     ○ | 将来API連携の主キー候補として保存                                     |
| 正規化URL保存                |     ○ | 表示・外部リンク用に `https://github.com/{owner}/{repo}` 形式を持つ   |
| 生URL保存                  |     ○ | ユーザーが入力した値も保持し、解析失敗時の復旧に使う                             |
| 手入力WorkItem             |     ○ | NextPatch内の課題・バグ・実装予定は独立データとして作成                       |
| GitHub Issue URLの手動関連付け |     ○ | APIなしでIssue/PR/Releaseリンクを貼れるようにする                     |
| GitHub API呼び出し          |     × | MVPでは実行しない                                             |
| OAuth / GitHub App接続    |     × | MVPでは実装しない                                             |
| トークン保存                  |     × | MVPでは保存しない                                             |
| Issue自動取得               |     × | 後回し                                                    |
| PR自動取得                  |     × | 後回し                                                    |
| Issue作成                 |     × | 後回し                                                    |
| 双方向同期                   |     × | 後回し                                                    |

URL解析のユーザー体験は、3操作以内を守るため、次の流れにします。

1. リポジトリ作成画面で GitHub URL を貼る。
2. 画面内で `owner / repo` の解析結果を確認する。
3. 「保存」を押す。

DADSに従い、GitHub URL入力欄には項目ラベルを置き、入力条件や例はプレースホルダーではなくサポートテキストで示します。解析失敗時は「何が悪いか・どう直すか」が分かる静的なエラーテキストを表示し、保存ボタンを安易に disabled にしない設計にします。

# MVPでは後回しにする内容

MVPでは、次の内容を明確に後回しにします。

| 後回し項目                       | 理由                                          |
| --------------------------- | ------------------------------------------- |
| GitHub OAuth / GitHub App接続 | トークン保存、権限説明、失効処理、組織ポリシー対応が必要                |
| リポジトリ一覧の取得                  | 認証ユーザーのリポジトリ一覧取得はAPIで可能だが、初期価値に対して認証コストが高い  |
| Issue取得                     | 取り込んだIssueとNextPatch WorkItemの重複・対応関係が難しい   |
| PR取得                        | PRはIssueの一種として扱う部分もあり、Issueとの統合設計が必要        |
| Issue作成                     | 書き込み権限、GitHub通知、二次レート制限、失敗時の再実行が絡む          |
| Release / Tag自動取得           | ロードマップ・リリース計画機能が固まってからでよい                   |
| 双方向同期                       | 競合解決、監査ログ、webhook、再試行キューが必要                 |
| GitHub Projects連携           | NextPatchの責務とGitHub Projectsの責務が重なるため、最後に扱う |

GitHub Projectsは、GitHub上のIssueやPull Requestをテーブル、ボード、ロードマップで管理し、カスタムフィールドや複数ビュー、チャート、自動化を持つ機能です。Projects内の変更はIssue/PRにも反映される双方向性を持つため、NextPatchが早期にProjects連携へ踏み込むと、GitHub Projectsの代替なのか、補助レイヤーなのかが曖昧になります。([GitHub Docs][6])

# 画面設計への影響

MVPの画面設計では、GitHub連携を大きく見せすぎないことが重要です。リポジトリ登録画面では「GitHub URLを貼るだけで開始できる」ことを主導線にし、API接続やOAuth設定は出さないほうがよいです。

## リポジトリ作成画面

構成案は次のとおりです。

| UI要素      | 内容                                                                       |
| --------- | ------------------------------------------------------------------------ |
| 見出し       | リポジトリを追加                                                                 |
| 入力欄       | GitHub URL                                                               |
| サポートテキスト  | `https://github.com/owner/repo` 形式のURLを入力してください、という説明                    |
| 解析結果      | `owner / repo` を画面内に表示                                                   |
| エラー       | `＊GitHubのリポジトリURLとして読み取れません。https://github.com/owner/repo の形式で入力してください。` |
| プライマリーボタン | 保存                                                                       |
| セカンダリー    | キャンセルまたは戻る                                                               |

DADSのフォームルールに従い、入力条件はプレースホルダーではなくサポートテキストで示します。保存不可の理由が分からない disabled ボタンは避け、押された後に具体的なエラーを表示する設計にします。

## リポジトリ詳細画面

リポジトリ詳細では、GitHub連携状態を次のように段階表示します。

| 状態           | 表示                      |
| ------------ | ----------------------- |
| Level 0      | GitHub URLのみ            |
| Level 1      | `owner/repo` 解析済み       |
| Level 2以降未接続 | 「GitHub API連携は未接続」情報バナー |
| 将来接続済み       | 最終取得日時、権限、取得対象          |

外部リンクは本文リンクと明確に区別し、新規タブや外部リンクであることをアイコンとテキストで補助します。DADSでは、リンクは色だけでなく下線などで本文と区別し、外部リンクや新規タブのアイコンには代替テキストを付けることが求められます。

## WorkItem詳細画面

WorkItemには「関連GitHubリンク」欄を設けます。MVPでは手動URL貼り付けだけにし、Issue/PR番号の自動取得はURL解析で補助する程度に留めます。

将来のIssue作成ボタンは、MVPでは表示しないか、「今後対応予定」の説明に留めます。未実装機能を disabled ボタンで見せるのは、利用者がなぜ押せないのか分かりにくいため避けます。

## 将来のGitHub接続画面

将来 Level 2 を入れる場合は、設定画面に「GitHubに接続」導線を置きます。ただし、DADSではモーダルダイアログやタブが準備中扱いなので、OAuth説明や権限説明をモーダル前提にしないほうが安全です。まずは通常ページ内で、接続前に取得対象・保存する情報・権限範囲を明示する設計がよいです。

# API・機能設計への影響

MVPでは GitHub API を呼びません。実装するのは、URL解析と外部リンク管理だけです。

## MVPで必要な機能

| 機能                    | 内容                                                              |
| --------------------- | --------------------------------------------------------------- |
| GitHub URL parser     | URLから `host / owner / repo / optional object type / number` を抽出 |
| URL normalizer        | `.git`、末尾スラッシュ、Issue/PR配下URLからリポジトリURLへ正規化                      |
| validation            | GitHub URLとして解釈できるかを判定                                          |
| external link manager | WorkItemにGitHub URLを関連付け                                        |
| canonical display     | `owner/repo` と外部リンクを表示                                          |

対応したいURL例は、最低限次の範囲です。

| 入力例                                                 | 抽出結果                               |
| --------------------------------------------------- | ---------------------------------- |
| `https://github.com/owner/repo`                     | repository                         |
| `https://github.com/owner/repo/`                    | repository                         |
| `https://github.com/owner/repo.git`                 | repository                         |
| `https://github.com/owner/repo/issues/123`          | repository + issue `123`           |
| `https://github.com/owner/repo/pull/45`             | repository + pull request `45`     |
| `https://github.com/owner/repo/releases/tag/v1.0.0` | repository + release/tag candidate |

## Level 2: 読み取り連携

Level 2では、GitHub APIからリポジトリ一覧、Issue、PR、Release、Tagを取得します。

リポジトリ一覧は、GitHub REST APIに「List repositories for the authenticated user」等のエンドポイントがあります。([GitHub Docs][1])
Issue一覧は repository issues endpoint で取得でき、`state`、`labels`、`sort`、`since`、`per_page` 等のパラメータを持ちます。([GitHub Docs][3])
PR一覧は Pull requests API で取得できますが、PRはIssueの一種であり、labelやassigneeなど共通管理はIssues API側に寄る点を設計に反映する必要があります。([GitHub Docs][7])
ReleaseとTagは分けて扱い、Release APIは通常のGit tagを含まないため、タグ一覧は Repository Tags APIを使います。([GitHub Docs][4])

Level 2の初期実装は「自動同期」ではなく「手動更新」から始めるべきです。理由は、GitHub公式のREST APIベストプラクティスで polling を避け、Webhookを使うことが推奨されている一方、Webhookを入れるには受信エンドポイント、署名検証、再配信処理、接続管理が必要になるためです。([GitHub Docs][8])

## Level 3: Issue作成

Issue作成は、WorkItemからGitHub Issueを一方向に作成する機能です。最初の実装では「NextPatch → GitHub」の単発作成に留め、GitHub側更新を自動でNextPatchへ反映しないほうが安全です。

作成時に保存すべき情報は次のとおりです。

| 保存情報                  | 用途                         |
| --------------------- | -------------------------- |
| `github_issue_number` | 表示・リンク                     |
| `external_id`         | REST APIの識別                |
| `external_node_id`    | GraphQL連携                  |
| `external_url`        | GitHub画面への遷移               |
| `pushed_at`           | 作成時刻                       |
| `pushed_by_user_id`   | 監査                         |
| `sync_state`          | pushed / failed / conflict |

Issue作成はGitHub通知を発生させ、短時間に多数作成すると二次レート制限の対象になるため、キュー処理、重複作成防止、再試行設計が必要です。([GitHub Docs][3])

## Level 4: 双方向同期

双方向同期では、最低限次の設計が必要です。

| 項目        | 必要な判断                                                           |
| --------- | --------------------------------------------------------------- |
| 同期対象フィールド | title / body / state / labels / assignees / comments のどこまで同期するか |
| 正本        | NextPatchとGitHubのどちらを優先するか                                      |
| 競合        | 両方で更新された場合の表示と解決方法                                              |
| 監査        | いつ、誰が、どの値を変えたか                                                  |
| 失敗時       | リトライ、停止、ユーザー通知                                                  |
| webhook   | GitHub側変更を受け取る方法                                                |
| セキュリティ    | webhook secret検証、署名検証、ログ秘匿                                      |

Webhookを使う場合は、GitHub側でsecretを設定でき、GitHub由来のリクエストに限定するため高エントロピーなsecretを使うことが推奨されています。([GitHub Docs][9])

## Level 5: Projects / Release 連携

GitHub Projectsは REST中心ではなく GraphQL API を使って管理する前提です。GitHub公式ドキュメントでは、Projects API利用例で、クエリには `read:project`、クエリとmutationには `project` scope が必要とされています。([GitHub Docs][10])

NextPatch側では、Projectsを直接置き換えるのではなく、次のように扱うのがよいです。

| GitHub Projects                       | NextPatchでの扱い                  |
| ------------------------------------- | ------------------------------ |
| Status / Priority / Iteration 等のフィールド | 必要に応じて参照・表示                    |
| Board / Table / Roadmap view          | NextPatch内で完全再現しない             |
| Draft issue                           | NextPatchのIdea/Planに近いが、同一視しない |
| Project item                          | ExternalObjectとして保持            |
| Status update                         | 将来、リポジトリ進捗メモと関連付け              |

# リスクと対策

| リスク                   | 内容                                                           | 対策                                                |
| --------------------- | ------------------------------------------------------------ | ------------------------------------------------- |
| MVPが重くなる              | API連携を入れると認証・権限・同期に工数が流れる                                    | MVPはLevel 1で固定                                    |
| URLだけでは将来移行しづらい       | raw URLだけ保存するとAPI接続時に再解析が必要                                  | MVPから `owner / repo / host / canonical_url` を保存   |
| OAuthスコープ過大           | OAuth Appの `repo` scopeはpublic/private repositoryへの広いアクセスを含む | 将来はGitHub App優先。OAuth採用時も段階的権限にする                 |
| トークン漏えい               | PATやOAuth tokenをDBに平文保存すると危険                                 | MVPでは保存しない。将来はsecret manager、暗号化、tokenRef、短命token |
| レート制限                 | 無認証は低い上限、認証済みでも大量取得に制限がある                                    | 手動更新、差分取得、キュー、キャッシュ、webhook利用                     |
| polling過多             | 定期的な全件取得はAPI制限に当たりやすい                                        | 双方向同期段階ではwebhook中心                                |
| WorkItemとIssueの意味差    | NextPatchのWorkItemはIssueより広い                                 | 1対1強制せず、外部リンク・外部オブジェクトで関連付け                       |
| GitHub Projectsとの責務重複 | Projectsはすでにテーブル/ボード/ロードマップ/カスタムフィールドを持つ                     | NextPatchは「開発判断・メモ・次アクション整理」の上位補助レイヤーと定義          |
| リポジトリ名変更・移管           | `owner/repo` が変わる可能性                                         | API接続後は `id` / `node_id` / `html_url` を保持         |
| UIが複雑化                | 連携状態、エラー、権限が増える                                              | DADSに従い、重要情報を隠さず、段階的に表示                           |

GitHubのレート制限では、無認証REST APIリクエストは公開データ取得時にIP単位で扱われ、primary rate limit は 60 requests/hour です。認証済みユーザーは通常 5,000 requests/hour の枠を持ちますが、二次レート制限もあるため、同期設計ではキュー化と抑制が必要です。([GitHub Docs][11])

# 未決定事項

未決定のままにしてよいものと、MVP前に決めるべきものを分けます。

## MVP前に決めるべき

| 項目                | 推奨                                                         |
| ----------------- | ---------------------------------------------------------- |
| MVP連携レベル          | Level 1                                                    |
| GitHub URL解析仕様    | `github.com/{owner}/{repo}` と Issue/PR/Release URLからrepo抽出 |
| GitHub URL入力の必須性  | GitHubリポジトリ作成フローでは主要入力。DB上はnullable                        |
| WorkItemとIssueの関係 | 同一視せず、外部リンクで対応                                             |
| 外部連携用テーブル         | MVPから軽量に用意                                                 |
| API連携UI           | MVPでは出さない                                                  |

## 後続フェーズで決める

| 項目                                       | 判断条件                                               |
| ---------------------------------------- | -------------------------------------------------- |
| GitHub AppかOAuth Appか                    | チーム・組織利用を想定するならGitHub App                          |
| Public-only無認証読み取りを入れるか                  | API接続なしで公開Issueを見たい需要が強い場合                         |
| private repository対応                     | ワークスペース権限とトークン管理が整ってから                             |
| Issue import時の重複判定                       | URL、number、node_id、タイトル類似度のどれを使うか                  |
| WorkItem status と GitHub Issue state の対応 | `open/closed` だけでは不足するため、NextPatch statusを正にするか要判断 |
| コメント同期                                   | 情報量・プライバシー・通知のリスクが大きいため後期判断                        |
| GitHub Projects連携範囲                      | 参照のみか、Project item更新まで行うか                          |
| GitHub Enterprise Server対応               | `github_host` を汎用化し、API base URLを設定可能にするか          |

# 統括担当へ渡す要点

MVPで採用する連携レベルは **Level 1: URL解析**。Level 0のGitHub URL手入力を含めつつ、URLから `owner / repo` を必ず抽出・保存する。GitHub API、OAuth、Issue作成、双方向同期、Projects連携はMVPには含めない。

ロードマップは次の順序で固定する。

|   Level | 内容                                  | 採用時期      |
| ------: | ----------------------------------- | --------- |
| Level 0 | GitHub URL 手入力                      | MVP内包     |
| Level 1 | URL解析、owner/repo保存、正規化URL保存         | **MVP採用** |
| Level 2 | リポジトリ一覧、Issue、PR、Release/Tagの読み取り連携 | MVP後      |
| Level 3 | NextPatch WorkItemからGitHub Issue作成  | 読み取り連携安定後 |
| Level 4 | 双方向同期、webhook、競合解決                  | 後期        |
| Level 5 | GitHub Projects / Release連携         | 最後        |

将来追加する際の設計上の備えとして、MVP時点で次を入れておく。

* `Repository` に `provider / github_host / github_owner / github_repo / canonical_url / github_url_raw` を持たせる。
* `WorkItem` は GitHub Issue と同一視せず、外部リンク・外部オブジェクトを通じて対応させる。
* `ExternalLink` または `ExternalObject` の軽量テーブルを用意し、Issue/PR/Release URLを手動で紐づけられるようにする。
* `external_id / external_node_id / sync_state / last_synced_at` は将来用のnullable項目として設計に含める。
* トークン保存はMVPでは行わない。
* 将来の認証方式はGitHub Appを第一候補にする。
* GitHub ProjectsはNextPatchの代替対象ではなく、後期の参照・補助連携として扱う。
