# 担当領域

UX・情報設計・導線設計。

対象は、NextPatch の画面構成、情報配置、PC / スマホの使い分け、主要タスクを3操作以内で達成する導線設計です。実装コード、DBマイグレーション、具体的なコンポーネント実装は対象外です。

# 前提理解

今回確認できた資料は `dads_app_ui_design_rules_20260411.md` です。NextPatch 要件定義書および「開発着手前に決定すべき内容の検討メモ」の本文は検索結果では確認できなかったため、NextPatch 固有の要件は、この依頼文に含まれる説明を前提として整理します。Web 情報は参照していません。DADS 参照資料は、2026年4月11日時点のデジタル庁デザインシステム β版公開ページを整理した資料です。

NextPatch は、GitHub リポジトリまたは開発中アプリ単位で、課題、進捗、バグ、実装予定、将来構想、採用技術、参考サービス、新規アイデア、ChatGPT との壁打ちメモを集約し、「今やるべきこと」をすぐ判断するための開発管理アプリです。

UX上の中心価値は、単なるメモ保存ではなく、散らばった情報を「次に取る行動」へ変換することです。したがって、画面設計では「情報を全部見せる」よりも、「今判断すべきものを先に見せる」ことを優先します。

DADS では、アクセシビリティ、一貫性、重要情報を隠さないこと、画面単体ではなく前後の流れでUIを判断することが最上位原則とされています。NextPatch では、未整理メモ数、未修正バグ、次アクション、進行中タスクなどをアコーディオン等で隠さず、ダッシュボードとリポジトリ詳細の上部に露出させるべきです。

PC / スマホの切り替えは、DADS の基準に合わせて 768px 未満をモバイル・タブレット、768px 以上をデスクトップとして扱います。PCでは複数カラムの情報整理、スマホでは1カラムの確認・登録・状態更新を主目的にします。

# 決定すべき論点

| 論点           | 決めるべき内容                        | 推奨方向               |
| ------------ | ------------------------------ | ------------------ |
| ダッシュボードの役割   | 全リポジトリ横断の統計画面にするか、今やる作業の入口にするか | 「今やること」中心          |
| リポジトリ詳細の役割   | 全情報を並べるだけか、作業判断の中核にするか         | リポジトリ単位の作業基地       |
| 未整理メモの扱い     | 通常メモと混ぜるか、専用の整理待ち入口を作るか        | 専用画面を作る            |
| クイック登録       | モーダル / ドロワー / 専用画面のどれにするか      | MVPは専用画面           |
| アイデアからタスク化   | アイデアを上書きするか、タスクを新規作成して関連付けるか   | 元アイデアを残してタスクを生成    |
| バグ修正済み導線     | 詳細画面必須か、一覧から直接更新できるか           | 一覧から1操作で修正済み       |
| タスク完了導線      | 詳細画面必須か、一覧から直接完了できるか           | 一覧から1操作で完了         |
| リポジトリ切り替え    | サイドバー中心か、ヘッダーのセレクタ中心か          | ヘッダー常設セレクタ         |
| PC / スマホの主用途 | 同じ情報密度にするか、用途を分けるか             | PCは整理、スマホは登録・確認・更新 |
| タブ / モーダル等   | DADS未整備部品をどう扱うか                | MVPでは避けるか暫定案扱い     |

# 選択肢比較

## 1. ダッシュボードの性格

| 選択肢        | 内容                      | 長所       | 短所           | 評価   |
| ---------- | ----------------------- | -------- | ------------ | ---- |
| A. 統計サマリー型 | リポジトリ数、タスク数、バグ数などを中心に表示 | 状況把握しやすい | 「次に何をするか」が弱い | 不採用  |
| B. 横断ToDo型 | 全リポジトリから今やるべき項目を抽出      | 行動に直結する  | 分類ルールが必要     | 採用   |
| C. 最近更新型   | 最近編集したリポジトリやメモを表示       | 実装しやすい   | 重要度と一致しない    | 補助扱い |

推奨は B です。NextPatch の価値は「情報保管」ではなく「今やるべきことの把握」なので、ダッシュボードは行動キューにします。

## 2. リポジトリ詳細の構造

| 選択肢             | 内容                                 | 長所      | 短所          | 評価       |
| --------------- | ---------------------------------- | ------- | ----------- | -------- |
| A. 種別ごとに別画面     | `/tasks`, `/bugs`, `/ideas` などを分ける | 画面が単純   | 行き来が増える     | MVPでは不採用 |
| B. 1画面に全種別を統合   | リポジトリ詳細に主要情報をまとめる                  | 文脈を失わない | 情報量が増える     | 採用       |
| C. ダッシュボードだけで処理 | 詳細画面を薄くする                          | 操作数は少ない | リポジトリの文脈が薄い | 不採用      |

推奨は B です。ただし、全情報を同じ重みで並べるのではなく、上から「次アクション」「未修正バグ」「進行中」「未整理メモ」「アイデア・構想」「技術・参考」の順に置きます。

## 3. クイック登録UI

| 選択肢              | 内容                 | 長所             | 短所               | 評価       |
| ---------------- | ------------------ | -------------- | ---------------- | -------- |
| A. モーダル          | どの画面からでも重ねて入力      | 画面遷移が少ない       | DADSではモーダル詳細が準備中 | MVPでは避ける |
| B. ドロワー / サイドシート | PCで文脈を保ったまま入力      | 使いやすい          | DADS上の独自仕様になりやすい | 暫定案      |
| C. 専用登録画面        | `/capture/new` に遷移 | アクセシブルで実装判断が明確 | 元画面から離れる         | 採用       |

MVPでは C を採用します。モーダルダイアログ、タブ、グローバルメニュー、ページナビゲーション、テーブルコントロールは DADS 側で詳細ガイドライン準備中とされているため、独自仕様を断定せず、使う場合は暫定案として扱います。

## 4. 未整理メモの置き場所

| 選択肢             | 内容              | 長所         | 短所             | 評価  |
| --------------- | --------------- | ---------- | -------------- | --- |
| A. 各リポジトリ詳細内だけ  | repoごとに未整理メモを管理 | 文脈が明確      | repo未確定メモを置けない | 不採用 |
| B. グローバル未整理メモ画面 | 全メモを一括整理        | 整理作業に集中できる | repo文脈が薄い      | 採用  |
| C. 通常メモ一覧に混在    | 実装が単純           | 未整理が埋もれる   | 不採用            |     |

推奨は B です。未整理メモは「保管場所」ではなく「処理待ちキュー」として扱います。repoが分かっているメモはリポジトリ詳細にも表示し、repo未確定メモは未整理メモ画面で分類できるようにします。

## 5. モバイルナビゲーション

| 選択肢              | 内容         | 長所                 | 短所        | 評価  |
| ---------------- | ---------- | ------------------ | --------- | --- |
| A. ボトムナビ         | 下部に主要メニュー  | タップしやすい            | DADSでは非推奨 | 不採用 |
| B. ハンバーガー + ドロワー | 上部メニューから遷移 | DADSのモバイルメニュー方針に合う | 1タップ増える   | 採用  |
| C. 画面上部リンクだけ     | 実装が単純      | ページが増えると破綻         | 補助        |     |

推奨は B です。DADSではモバイルメニューは2階層を超えないこと、ボトムナビゲーションは非推奨であることが示されています。NextPatch のモバイルでは、下部固定ナビではなく、上部のリポジトリ選択、クイック登録、ハンバーガーメニューで構成します。

# 推奨案

推奨する情報設計は、次の4つを主軸にします。

1. **ダッシュボード**
   全リポジトリ横断で「今やること」を表示する画面。統計ではなく、次アクション、未修正バグ、未整理メモ、停滞中タスクを優先表示する。

2. **リポジトリ詳細**
   1つのリポジトリに関する作業基地。課題、進捗、バグ、実装予定、将来構想、採用技術、参考サービス、アイデア、メモを文脈付きで整理する。

3. **未整理メモ**
   ChatGPT壁打ち、思いつき、参考URL、まだ分類できないメモを受け止める画面。ここからタスク、バグ、アイデア、実装予定、技術メモ、参考サービスへ変換する。

4. **クイック登録**
   どの画面からでも3操作以内でメモを保存する入口。MVPでは専用画面 `/capture/new` とし、PC用サイドシートやモーダルは暫定案として後回しにする。

PCでは、左側に主要ナビゲーション、上部にリポジトリ切り替えとクイック登録、中央に作業リスト、右側に補助情報を置く構成にします。スマホでは、1カラムで「登録」「確認」「状態更新」を優先し、大量整理や横断比較はPCに寄せます。

ボタンは DADS に合わせ、主要CTAを原則1画面1個にします。たとえばダッシュボードの主CTAは「クイック登録」、登録画面の主CTAは「保存」、タスク詳細の主CTAは「完了にする」です。ボタンの押下対象領域は44 CSS px以上を確保し、安易な disabled は避け、未入力で押された場合に具体的なエラーを出す設計にします。 

フォームは、DADSに合わせてラベル、要否、サポートテキスト、具体的なエラーテキストを持たせます。プレースホルダーで説明を代用しない方針とし、クイック登録でも「本文」「対象リポジトリ」「種類」などのラベルを明示します。

# MVPに含めるべき内容

| 項目         | MVPでの扱い                              |
| ---------- | ------------------------------------ |
| ダッシュボード    | 全リポジトリ横断の「今やること」一覧を表示                |
| リポジトリ一覧    | 登録済みリポジトリの一覧、最近開いたリポジトリを表示           |
| リポジトリ詳細    | 次アクション、タスク、バグ、アイデア、予定、技術・参考情報、メモを表示  |
| 未整理メモ画面    | 未分類メモを一覧し、分類・タスク化・アーカイブできる           |
| クイック登録     | どの画面からでもメモを登録できる                     |
| タスク完了      | 一覧または詳細から1操作で完了にできる                  |
| バグ修正済み     | 一覧または詳細から1操作で修正済みにできる                |
| アイデアからタスク化 | 元アイデアを残したまま、関連タスクを作成する               |
| リポジトリ切り替え  | ヘッダーから2操作以内で切り替え                     |
| レスポンシブ対応   | PCは2カラム、スマホは1カラム                     |
| 通知         | 保存、完了、修正済みなどをページ上部または関連セクション内のバナーで通知 |

操作完了後は、DADSのノティフィケーションバナー方針に合わせ、成功・エラー・警告・情報提示を明確に出します。状態変更後は「完了しました。元に戻す」のように、複雑な確認モーダルではなく、バナーで取り消し導線を出すのが適切です。

# MVPでは後回しにする内容

| 項目                    | 後回しにする理由                        |
| --------------------- | ------------------------------- |
| GitHub Issues との双方向同期 | データ仕様と競合解決が重く、MVPの情報整理価値と分離できる  |
| AIによる自動分類             | 便利だが、まず手動分類の情報設計を固めるべき          |
| カンバン表示                | 視覚的には有効だが、スマホ対応とDADS未整備要素が増える   |
| ロードマップ専用画面            | まずはリポジトリ詳細内の「実装予定・将来構想」で足りる     |
| 高度な検索・保存フィルタ          | MVP後に情報量が増えてから導入                |
| チーム権限 / RBAC          | 個人または小規模利用で先に価値検証               |
| 通知・リマインダー             | 「今やること」画面が成立してから追加              |
| 添付ファイル管理              | メモ・URL・テキスト中心で開始                |
| モーダル型クイック登録           | DADS未整備のため暫定案として後回し             |
| タブUI                  | DADS未整備のため、MVPでは見出し・フィルタ・リンクで代替 |

# データ設計への影響

MVPでは、タスク、バグ、アイデア、メモを完全に別テーブル化するより、共通の `Item` として扱う設計を推奨します。理由は、未整理メモからタスク化、アイデアからタスク化、バグから修正タスク作成など、種類をまたいだ変換が多いためです。

## 主要エンティティ案

| エンティティ         | 主な役割                          | 主要フィールド例                                                                                                 |
| -------------- | ----------------------------- | -------------------------------------------------------------------------------------------------------- |
| Repository     | 管理対象のアプリ / GitHub repo        | id, name, owner, url, description, currentFocus, archivedAt                                              |
| Item           | タスク、バグ、アイデア、メモ等の共通単位          | id, repoId, type, title, body, status, priority, isNextAction, source, parentItemId, convertedFromItemId |
| ItemRelation   | アイデア→タスク、メモ→バグなどの関連           | fromItemId, toItemId, relationType                                                                       |
| ItemEvent      | 完了、修正済み、分類、変換などの履歴            | itemId, eventType, before, after, createdAt                                                              |
| RepositoryNote | repo固有の技術・参考情報をItemで持たない場合の補助 | repoId, category, title, url, body                                                                       |
| UserPreference | 最近使ったrepo、既定repoなど            | lastRepositoryId, recentRepositoryIds                                                                    |

## Item type 案

| type        | 用途             | MVPでの状態例                                      |
| ----------- | -------------- | --------------------------------------------- |
| `memo`      | 未整理メモ、壁打ちメモ    | `untriaged`, `converted`, `archived`          |
| `task`      | 実装・修正・調査タスク    | `backlog`, `next`, `in_progress`, `done`      |
| `bug`       | 不具合            | `open`, `fixing`, `fixed`, `wontfix`          |
| `idea`      | 新規アイデア、将来構想    | `captured`, `candidate`, `tasked`, `archived` |
| `plan`      | 実装予定、ロードマップ的項目 | `planned`, `active`, `done`, `postponed`      |
| `tech`      | 採用技術、技術メモ      | `active`, `considering`, `rejected`           |
| `reference` | 参考サービス、参考URL   | `active`, `archived`                          |

「今やること」は自動スコアだけにせず、MVPでは `isNextAction` または `status = next` を明示的に持たせます。自動推定は後回しにし、利用者が自分で「次にやる」に入れられるほうが、挙動が分かりやすくなります。

# 画面設計への影響

## 画面構成

| 観点        | 推奨構成                                              |
| --------- | ------------------------------------------------- |
| 必須画面      | ダッシュボード、リポジトリ一覧、リポジトリ詳細、未整理メモ、クイック登録、アイテム詳細 / 編集  |
| 後回しにできる画面 | 高度検索、ロードマップ専用、設定詳細、GitHub連携設定、通知設定、分析画面           |
| 統合できる画面   | タスク一覧、バグ一覧、アイデア一覧、予定一覧はリポジトリ詳細内のセクションまたはフィルタで統合   |
| 分けるべき画面   | ダッシュボードとリポジトリ詳細、未整理メモと通常メモ、クイック登録と詳細編集            |
| PC表示      | 左ナビ + 上部バー + メイン領域 + 右補助領域。リポジトリ詳細は8:4または9:3の2カラム |
| スマホ表示     | 1カラム。上部に現在リポジトリ、クイック登録、メニュー。詳細編集は別ページ遷移           |

## PCでの主要利用シーン

PCでは、整理・判断・変換を主目的にします。

| 利用シーン          | 画面               | 設計方針                       |
| -------------- | ---------------- | -------------------------- |
| 今日やる作業を決める     | ダッシュボード          | 「今やること」を最上部に表示             |
| repoごとの状況を確認する | リポジトリ詳細          | 次アクション、未修正バグ、未整理メモを上部に表示   |
| ChatGPTメモを整理する | 未整理メモ            | メモごとに「タスク化」「バグ化」「アイデア化」を表示 |
| バグ修正状況を見る      | リポジトリ詳細          | 未修正バグを目立つ位置に置く             |
| 将来構想をタスク化する    | リポジトリ詳細 / アイテム詳細 | アイデアから関連タスクを生成             |

## スマホでの主要利用シーン

スマホでは、長時間の整理ではなく、短時間の登録・確認・状態更新に限定します。

| 利用シーン      | 画面                | 設計方針               |
| ---------- | ----------------- | ------------------ |
| 思いつきを登録する  | クイック登録            | 本文だけでも保存可能         |
| 今やることを見る   | ダッシュボード           | 1カラムで次アクションを最上部に表示 |
| タスクを完了にする  | ダッシュボード / リポジトリ詳細 | カード上に「完了にする」を表示    |
| バグを修正済みにする | ダッシュボード / リポジトリ詳細 | カード上に「修正済みにする」を表示  |
| repoを切り替える | ヘッダー              | 最近使ったrepoを先に表示     |

## 各画面の役割

| 画面      | 役割                   | 最上部に置く情報                 |
| ------- | -------------------- | ------------------------ |
| ダッシュボード | 全repo横断で「今やること」を判断する | 次アクション、未修正バグ、未整理メモ数      |
| リポジトリ詳細 | repo単位の文脈を把握し、作業を進める | repo名、現在の焦点、次アクション、未修正バグ |
| 未整理メモ   | メモを行動可能な情報へ変換する      | 未分類件数、分類待ちメモ             |
| クイック登録  | とにかく情報を失わず保存する       | 本文入力、repo選択、種類選択         |
| アイテム詳細  | 詳細確認・編集・関連付けを行う      | タイトル、種類、状態、次の操作          |

## 主要導線

ここでの「操作」は、画面遷移、ボタン押下、選択、保存などの主要UI操作を指します。文字入力そのものは、該当フォーム操作に含めます。

| 目的           | 開始画面              | 操作1                | 操作2             | 操作3     | 補足                             |
| ------------ | ----------------- | ------------------ | --------------- | ------- | ------------------------------ |
| 今やることを確認する   | ダッシュボード           | 「今やること」欄を見る        | 必要なら対象カードを開く    | —       | ダッシュボードのファーストビューに表示するため実質0〜1操作 |
| リポジトリを切り替える  | 任意の画面             | ヘッダーのリポジトリ選択を開く    | 対象リポジトリを選ぶ      | —       | 選択後はリポジトリ詳細へ遷移                 |
| クイック登録する     | 任意の画面             | 「クイック登録」を押す        | 本文・repo・種類を入力   | 「保存」を押す | 種類未選択なら `memo` として未整理に入れる      |
| 未整理メモを分類する   | 未整理メモ             | メモの「分類する」を押す       | 種類・repoを選ぶ      | 「保存」を押す | 分類後は元メモを converted にする         |
| 未整理メモをタスク化する | 未整理メモ             | メモの「タスク化」を押す       | タスク名・優先度を確認     | 「作成」を押す | 元メモと新タスクを関連付ける                 |
| アイデアをタスク化する  | リポジトリ詳細 / アイテム詳細  | アイデアの「タスク化」を押す     | タスク内容を確認・編集     | 「作成」を押す | アイデアは残し、状態を `tasked` にする       |
| バグを修正済みにする   | ダッシュボード / リポジトリ詳細 | バグカードの「修正済みにする」を押す | —               | —       | 成功バナーに「元に戻す」を出す                |
| タスクを完了にする    | ダッシュボード / リポジトリ詳細 | タスクカードの「完了にする」を押す  | —               | —       | 確認モーダルは使わず、取り消し可能にする           |
| repo詳細を開く    | ダッシュボード / リポジトリ一覧 | repoカードを押す         | —               | —       | 最近使ったrepoを上位表示                 |
| バグ一覧を見る      | リポジトリ詳細           | 種類フィルタで「バグ」を選ぶ     | —               | —       | 未修正バグは上部に常時表示                  |
| 採用技術を見る      | リポジトリ詳細           | 「技術・参考」セクションへ移動    | —               | —       | PCでは右補助領域、スマホでは下部セクション         |
| メモを後で整理に回す   | クイック登録            | 本文を入力              | 「未整理メモとして保存」を押す | —       | repo未確定でも保存可能                  |

## ダッシュボード画面

PC表示では次の順に置きます。

1. ページ見出し：「今やること」
2. グローバルCTA：「クイック登録」
3. 次アクション一覧
4. 未修正バグ
5. 未整理メモ
6. 最近更新されたリポジトリ
7. 右補助領域にリポジトリ切り替え、最近使ったrepo、件数サマリー

スマホ表示では次の順です。

1. ページ見出し
2. クイック登録ボタン
3. 今やること
4. 未修正バグ
5. 未整理メモ
6. 最近のrepo

統計カードは補助扱いにし、最上部には置きません。

## リポジトリ詳細画面

PC表示では、メイン8カラム、補助4カラムを基本にします。

メイン領域：

1. repo名、説明、現在の焦点
2. 次アクション
3. 未修正バグ
4. 進行中タスク
5. 実装予定
6. アイデア・将来構想
7. 関連メモ

補助領域：

1. 採用技術
2. 参考サービス / URL
3. 未整理メモ数
4. 最近の更新履歴
5. repo設定へのリンク

スマホ表示では、補助領域を下部へ回し、最初に「次アクション」「未修正バグ」「クイック登録」を出します。

## 未整理メモ画面

未整理メモ画面は、保存されたメモを読む画面ではなく、分類する画面です。

各メモカードには、次の操作を表示します。

| 操作     | 意味                 |
| ------ | ------------------ |
| タスク化   | 実装・調査・修正タスクにする     |
| バグ化    | 不具合として登録する         |
| アイデア化  | 将来構想・新規アイデアにする     |
| 技術メモ化  | 採用技術や設計判断として保存する   |
| 参考URL化 | 参考サービス・参考記事として保存する |
| アーカイブ  | 今は使わない情報として閉じる     |

重要な分類操作はメニューの奥に隠さず、カード上に表示します。ただし、スマホでは主要3つ「タスク化」「バグ化」「アイデア化」を先に出し、それ以外は「その他」にまとめる暫定案とします。

## クイック登録画面

MVPのクイック登録は専用画面にします。

必須項目：

| 項目      | 入力方針               |
| ------- | ------------------ |
| 本文      | 必須。ラベルとサポートテキストを表示 |
| リポジトリ   | 任意。直近repoを初期候補にする  |
| 種類      | 任意。未選択なら未整理メモ      |
| 今やるに入れる | 任意。チェック時のみ次アクション化  |

種類の選択肢が増えすぎる場合、DADSの選択部品方針に合わせ、5件以下ならラジオ、6件以上ならセレクトボックスを使うのが妥当です。

# API・機能設計への影響

## 必要な機能単位

| 機能                 | 目的                     |
| ------------------ | ---------------------- |
| Dashboard Query    | 今やること、未修正バグ、未整理メモを横断取得 |
| Repository CRUD    | repo登録、編集、アーカイブ        |
| Item CRUD          | タスク、バグ、アイデア、メモなどの作成・編集 |
| Quick Capture      | 最小入力でメモ保存              |
| Item Status Update | タスク完了、バグ修正済みなどの状態変更    |
| Item Conversion    | メモ→タスク、アイデア→タスクなどの変換   |
| Item Relation      | 変換元と生成先の関連保持           |
| Undo Event         | 完了・修正済みなどの取り消し         |
| Recent Repository  | 最近使ったrepoの記録           |

## API案

| API                                           | 用途              |
| --------------------------------------------- | --------------- |
| `GET /api/dashboard?repoId=`                  | ダッシュボード表示       |
| `GET /api/repositories`                       | repo一覧          |
| `POST /api/repositories`                      | repo作成          |
| `GET /api/repositories/:repoId`               | repo詳細          |
| `GET /api/items?repoId=&type=&status=&focus=` | アイテム一覧          |
| `POST /api/items`                             | アイテム作成          |
| `PATCH /api/items/:itemId`                    | アイテム編集          |
| `PATCH /api/items/:itemId/status`             | 完了、修正済みなどの状態変更  |
| `POST /api/items/:itemId/convert`             | メモやアイデアから別種別を生成 |
| `GET /api/items/:itemId/relations`            | 関連アイテム取得        |
| `POST /api/item-events/:eventId/undo`         | 直前操作の取り消し       |

完了・修正済みは削除ではなく状態変更として扱います。UIでは即時反映し、成功バナーで取り消しを提供します。

# リスクと対策

| リスク              | 内容                                         | 対策                         |
| ---------------- | ------------------------------------------ | -------------------------- |
| ダッシュボードが情報過多になる  | 全情報を載せると「今やること」が埋もれる                       | 表示対象を次アクション、未修正バグ、未整理メモに絞る |
| 未整理メモが溜まり続ける     | 保存は簡単だが整理されない                              | ダッシュボードに未整理件数を常時表示         |
| 種別が増えすぎる         | task, bug, idea, plan, tech, reference が混在 | 共通Itemモデル + typeで管理        |
| スマホが複雑になる        | PCと同じ機能量を載せると使いづらい                         | スマホは登録・確認・状態更新に限定          |
| タブ・モーダル依存        | DADS未整備コンポーネントの独自仕様化                       | MVPでは専用画面と見出し構成で代替         |
| 3操作以内と確認安全性が衝突   | 完了前確認を入れると操作が増える                           | 確認モーダルではなく取り消しバナー          |
| repo未確定メモが保存できない | 思いつきを逃す                                    | repo任意で保存し、未整理メモで後分類       |
| GitHub連携が肥大化する   | 認証、同期、競合解決が重い                              | MVPではURL保持または手動repo管理まで    |
| ボタンが多くなる         | 各カードに操作を並べすぎる                              | 主要操作を1〜2個に絞り、補助操作は詳細画面へ    |

# 未決定事項

| 未決定事項           | 決める必要がある内容                                                       |
| --------------- | ---------------------------------------------------------------- |
| 「今やること」の判定      | 明示フラグだけか、優先度・期限・状態から算出するか                                        |
| Item type の最終種類 | `memo`, `task`, `bug`, `idea`, `plan`, `tech`, `reference` で足りるか |
| バグの状態           | `open` / `fixed` だけか、`verified` まで持つか                            |
| タスクの状態          | `backlog`, `next`, `in_progress`, `done` で開始するか                  |
| repo未紐付けメモ      | 許可するか、必ずrepo選択させるか                                               |
| PC用サイドシート       | クイック登録を将来サイドシート化するか                                              |
| GitHub連携        | MVPでrepo URL保持のみか、Issue同期まで含めるか                                  |
| チーム利用           | 個人利用前提か、複数ユーザー・権限まで想定するか                                         |
| 検索              | MVPで簡易検索を入れるか、後回しにするか                                            |
| ソート順            | 優先度、更新日、手動順のどれを基本にするか                                            |

# 統括担当へ渡す要点

NextPatch のUXは、「メモを溜めるアプリ」ではなく、「未整理情報を次アクションへ変換するアプリ」として設計するべきです。

MVPでは、ダッシュボード、リポジトリ詳細、未整理メモ、クイック登録の4本柱に絞ります。クイック登録は専用画面、状態更新は一覧から1操作、分類・タスク化は3操作以内にします。

DADSとの関係では、モーダル、タブ、グローバルメニュー、ページナビゲーション、テーブルコントロールは詳細ガイドライン準備中のため、MVPでは断定仕様にしないでください。ナビゲーションはPCでは左ナビ暫定案、スマホではハンバーガーメニューを基本にし、ボトムナビは採用しません。
