# 担当領域

QA・テスト・アクセシビリティ。

NextPatch の MVP が、データ破壊、状態不整合、権限漏れ、スマホ表示崩れ、キーボード操作不能、DADS 参照ルール違反によって使えなくならないことを確認するための品質設計を担当します。

# 前提理解

NextPatch は、GitHub リポジトリや開発中アプリ単位で、課題、進捗、バグ、実装予定、将来構想、採用技術、参考サービス、新規アイデア、ChatGPT との壁打ちメモを整理し、「今やるべきこと」をすぐ把握するための開発管理アプリです。

この回答では、会話内で明示された NextPatch 概要、指定された検討論点、DADS 参照ルールを前提にします。NextPatch 要件定義書と「開発着手前に決定すべき内容の検討メモ」の全文はこの会話内では確認できないため、画面名・データ名は `Repository`、`Item`、`Memo`、`Dashboard` などの仮称として扱います。実際の要件定義書に正式名称がある場合は読み替えてください。

UI 品質では DADS 参照ルールを優先します。フォームではラベル必須、プレースホルダー代用禁止、具体的なエラー文、安易な disabled 回避、エラーの静的表示を確認対象にします。ボタンでは 1 画面 1 プライマリー原則、押下対象 44 CSS px 以上、モバイル時の縦積み順序を確認します。リンクは色だけでなく下線などで識別し、重要情報はアコーディオンやディスクロージャーに隠さない方針です。モバイルでは DADS 上、ボトムナビゲーションは非推奨であり、モバイルメニューは階層を深くしすぎないことを確認します。

アクセシビリティの外部基準としては、MVP の目標を **WCAG 2.2 AA 相当**に置くのが妥当です。WCAG 2.2 は W3C の Web Content Accessibility Guidelines として、Web コンテンツのアクセシビリティ推奨事項をまとめています。確認日: 2026-04-20。([W3C][1])

E2E と自動アクセシビリティ検査の候補としては Playwright と axe-core を推奨します。Playwright は E2E 自動化に加えてアクセシビリティ上の問題検出にも利用でき、axe-core は HTML ベース UI 向けの自動アクセシビリティ検査エンジンです。確認日: 2026-04-20。([Playwright][2])

パフォーマンスと品質回帰の補助指標としては Lighthouse / Lighthouse CI を使えます。Lighthouse はパフォーマンス、アクセシビリティ、PWA、SEO などの監査を行い、Lighthouse CI により回帰検知にも使えるとされています。確認日: 2026-04-20。([Chrome for Developers][3])

# 決定すべき論点

最初に決めるべき論点は、品質ゲートの対象範囲です。MVP では「全部を完璧にテストする」より、「壊れると致命的な箇所を自動テストで固定し、DADS とアクセシビリティをリリース前に必ず確認する」方針にします。

特に決めるべき点は次のとおりです。

| 論点                    | 決める内容                | 推奨判断                                                           |
| --------------------- | -------------------- | -------------------------------------------------------------- |
| テストピラミッド              | 単体・結合・E2E の比重        | 単体と結合を厚くし、E2E は主要導線に絞る                                         |
| 受け入れ条件                | MVP を出してよい最低品質       | P0/P1 バグなし、主要 E2E 全通過、重大な a11y 違反なし                            |
| completedAt           | 完了時刻の自動設定仕様          | サーバー側で状態遷移時に設定・解除する                                            |
| repositoryId nullable | リポジトリ未紐づけデータの扱い      | 明示的に許容し、検索・ダッシュボード・エクスポート対象に含める                                |
| 未整理メモ分類               | 自動分類の信頼性             | 元メモを破壊せず、分類候補・信頼度・未分類維持を許容する                                   |
| DADS 準拠               | どこまでを必須にするか          | フォーム、ボタン、リンク、重要情報、モバイルメニュー、色依存回避は必須                            |
| アクセシビリティ              | 自動・手動の役割             | axe / Lighthouse は補助、キーボードとフォーカスは手動必須                          |
| レスポンシブ                | 対象幅                  | DADS の 768px 境界に加え、375px / 390px / 768px / 1024px / 1440px を確認 |
| 認証・権限                 | データ分離確認              | UI と API の両方で他ユーザーのデータ参照・変更不可を確認                               |
| エクスポート・復元             | MVP に含める範囲           | JSON 全量エクスポートと全量復元を MVP に含め、復元はトランザクション必須                      |
| 品質ゲート                 | PR / RC / リリース前の停止条件 | 自動テスト失敗、重大 a11y、権限漏れ、復元失敗はリリース不可                               |

# 選択肢比較

| 項目                    | 案A             | 案B                | 比較                                                                   |
| --------------------- | -------------- | ----------------- | -------------------------------------------------------------------- |
| テスト戦略                 | E2E 中心         | 単体・結合中心 + 重要 E2E  | 案Bを推奨。状態遷移、completedAt、repositoryId nullable、分類ロジックは E2E だけでは原因特定が遅い |
| アクセシビリティ              | 自動検査のみ         | 自動検査 + キーボード + 目視 | 案Bを推奨。自動検査はラベル欠落やコントラスト検出に有効だが、導線の理解しやすさやフォーカス順は手動確認が必要              |
| レスポンシブ                | 768px 境界のみ     | 代表端末幅も確認          | 案Bを推奨。DADS の境界は 768px だが、実際のスマホ幅で横スクロールやボタン折返しが起きやすい                 |
| completedAt           | クライアントで設定      | サーバーで設定           | サーバー設定を推奨。改ざん、時刻差、二重送信時の不整合を避けやすい                                    |
| repositoryId nullable | nullable を例外扱い | 正規の状態として扱う        | 正規状態を推奨。リポジトリ未定のメモや横断アイデアが NextPatch の利用価値に直結する                      |
| 未整理メモ分類               | 自動で即分類・移動      | 元メモ保持 + 候補提示      | 候補提示を推奨。誤分類による情報喪失を防げる                                               |
| 復元方式                  | 差分マージ          | 全量置換または空環境への復元    | MVP では全量置換または空環境復元を推奨。差分マージは競合処理が増えるため後回し                            |
| モバイルナビ                | ボトムナビ          | ハンバーガー + ドロワー     | ハンバーガー + ドロワーを推奨。DADS でボトムナビは非推奨                                     |
| モーダル利用                | 多用             | 原則避ける             | 原則避ける。DADS でモーダルダイアログは詳細ガイドライン未整備のため、使う場合は暫定案として扱う                   |

# 推奨案

MVP の QA 方針は、**「データ整合性・権限・主要導線・DADS 準拠を品質ゲート化する」**です。

具体的には、次の構成を推奨します。

1. **単体テスト**
   状態遷移、バリデーション、検索・絞り込み、ダッシュボード集計、未整理メモ分類、エクスポート形式、権限判定など、UI に依存しないロジックを固定します。

2. **結合テスト**
   API、DB、認証、repositoryId nullable、completedAt 自動設定、復元トランザクション、他ユーザー ID を指定した不正アクセスを確認します。

3. **E2E テスト**
   ログインから「リポジトリ作成 → メモ登録 → 分類 → 今やること確認 → 状態変更 → 検索 → エクスポート」までの主要導線を確認します。E2E は増やしすぎず、MVP の価値が壊れていないかを確認する用途に絞ります。

4. **アクセシビリティ確認**
   自動検査で重大違反を拾い、手動でキーボード操作、フォーカス表示、フォームエラー、色だけに依存しない状態表示を確認します。

5. **レスポンシブ確認**
   DADS の 768px 境界を中心に、スマホ幅、タブレット境界、デスクトップで確認します。特にダッシュボード、一覧、詳細、フォーム、モバイルメニューを重点対象にします。

6. **DADS 準拠確認**
   フォーム、ボタン、リンク、重要情報、モバイルメニュー、ボトムナビ非採用をチェックリスト化します。DADS で未整備のモーダル、タブ、テーブルコントロール、グローバルメニュー等を使う場合は「暫定案」と明記し、MVP では依存度を下げます。

最も重要な判断は、**completedAt と repositoryId nullable を UI テストだけで済ませないこと**です。この2つはデータ設計とダッシュボード表示に直接影響するため、単体・結合・E2E の三層で確認します。

# MVPに含めるべき内容

## 1. MVP受け入れ条件

MVP をリリース可能と判断する条件は次のとおりです。

| 分類                    | 受け入れ条件                                                    |
| --------------------- | --------------------------------------------------------- |
| 基本導線                  | ログイン後、3操作以内で「今やるべきこと」に到達できる                               |
| データ登録                 | リポジトリ、課題、バグ、実装予定、メモを作成・編集・削除できる                           |
| 未整理メモ                 | 未整理メモを登録でき、分類候補を確認でき、元メモが失われない                            |
| ダッシュボード               | 未完了・進行中・完了済み・未整理・リポジトリ未紐づけの件数と優先項目が正しく表示される               |
| completedAt           | ステータスを完了にした時点で `completedAt` が自動設定され、未完了へ戻すと null になる     |
| repositoryId nullable | `repositoryId = null` の項目を作成・編集・検索・ダッシュボード表示・エクスポート・復元できる |
| 検索・絞り込み               | キーワード、種別、ステータス、リポジトリ、未紐づけで絞り込める                           |
| 認証・権限                 | 他ユーザーのリポジトリ・項目・メモを UI/API から参照・変更できない                     |
| エクスポート                | 自分の全データをスキーマバージョン付き JSON としてエクスポートできる                     |
| 復元                    | エクスポート済み JSON を空環境または全量置換モードで復元でき、失敗時に部分復元されない            |
| PC 表示                 | 1024px / 1440px で主要画面が破綻しない                               |
| スマホ表示                 | 375px / 390px で横スクロールなし、主要操作が可能                           |
| DADS フォーム             | ラベル、要否表示、サポートテキスト、具体的エラー文がある                              |
| DADS ボタン              | 主要 CTA が一貫し、押下対象 44px 以上、安易な disabled がない                 |
| DADS リンク              | リンクが色だけでなく下線等でも識別できる                                      |
| アクセシビリティ              | キーボードのみで主要導線を完了でき、フォーカスが見える                               |
| 色依存回避                 | 状態表示が色だけでなくテキスト・アイコン・ラベルで判別できる                            |
| エラー表示                 | 入力エラー、通信エラー、権限エラー、復元エラーが具体的に表示される                         |
| 品質ゲート                 | P0/P1 バグなし、主要 E2E 全通過、重大 a11y 違反なし                        |

P0 はデータ喪失、権限漏れ、ログイン不能、復元による破壊、主要導線不能です。P1 は完了状態不整合、ダッシュボード重大誤表示、スマホ主要操作不能、キーボード操作不能、重大なフォーム誤案内です。

## 2. テスト観点一覧

| テスト種別    | 対象                    | 必須観点                                                  |
| -------- | --------------------- | ----------------------------------------------------- |
| 単体テスト    | 状態遷移                  | `todo` / `inProgress` / `blocked` / `done` などの遷移、完了解除 |
| 単体テスト    | completedAt           | 完了化で設定、再保存で意図せず上書きしない、未完了化で null                      |
| 単体テスト    | バリデーション               | 必須、文字数、enum、不正 repositoryId、日付、URL                    |
| 単体テスト    | repositoryId nullable | null を正常値として扱う、undefined と null の扱いを統一                |
| 単体テスト    | 検索・絞り込み               | 日本語、英数字、大文字小文字、空白、複合条件                                |
| 単体テスト    | ダッシュボード集計             | 件数、優先順、未整理、期限切れ、未紐づけ                                  |
| 単体テスト    | 未整理メモ分類               | 入力文から種別候補を返す、低信頼時は未整理維持、元文保持                          |
| 単体テスト    | エクスポート                | JSON スキーマ、バージョン、nullable、日時、関連 ID                     |
| 単体テスト    | 復元                    | スキーマ検証、ID 衝突、欠損フィールド、ロールバック判定                         |
| 結合テスト    | API + DB              | 作成・更新・削除・一覧・詳細                                        |
| 結合テスト    | 認証・権限                 | 他ユーザー ID 指定時 403/404、エクスポート範囲                         |
| 結合テスト    | completedAt           | DB 保存値がサーバー時刻で一貫する                                    |
| 結合テスト    | repositoryId nullable | DB 制約、一覧、検索、復元で null が通る                              |
| 結合テスト    | リポジトリ削除               | 紐づく項目の扱いを確認。MVP では削除ブロックを推奨                           |
| 結合テスト    | エクスポート・復元             | round-trip、トランザクション、失敗時 rollback                      |
| E2E      | 主要導線                  | ログイン、作成、分類、ダッシュボード、検索、完了、エクスポート                       |
| E2E      | スマホ導線                 | モバイルメニュー、フォーム入力、一覧確認、状態変更                             |
| E2E      | エラー導線                 | 必須未入力、権限エラー、通信失敗、復元失敗                                 |
| アクセシビリティ | 自動検査                  | axe / Lighthouse で重大違反を検出                             |
| アクセシビリティ | 手動検査                  | キーボード、フォーカス、読み順、色依存、ラベル                               |
| レスポンシブ   | レイアウト                 | 375 / 390 / 768 / 1024 / 1440px                       |
| パフォーマンス  | 主要画面                  | 初期表示、検索応答、一覧件数増加、復元処理                                 |

## 3. E2Eシナリオ

MVP で必須にする E2E シナリオは次の 12 本に絞るのが適切です。

| No | シナリオ                           | 合格条件                                            |
| -: | ------------------------------ | ----------------------------------------------- |
|  1 | 新規ユーザーがログインし、空のダッシュボードを見る      | 空状態メッセージと次の行動が表示される                             |
|  2 | リポジトリを作成し、課題を1件追加する            | ダッシュボードとリポジトリ詳細に反映される                           |
|  3 | repositoryId なしの未整理メモを作成する     | 未紐づけとして表示され、検索・ダッシュボードに出る                       |
|  4 | 未整理メモを分類候補から「バグ」または「実装予定」に変換する | 元メモが保持され、分類後の項目が作られる                            |
|  5 | 項目を進行中から完了へ変更する                | `completedAt` が表示または内部反映され、完了一覧に移動する            |
|  6 | 完了項目を未完了へ戻す                    | `completedAt` が null になり、未完了一覧に戻る               |
|  7 | キーワードとステータスで検索・絞り込みする          | 条件に合う項目だけ表示され、解除で元に戻る                           |
|  8 | スマホ幅でモバイルメニューから主要画面へ移動する       | ボトムナビなしで主要画面へ到達できる                              |
|  9 | 必須項目を空で保存する                    | disabled ではなく、押下後に具体的エラーが静的表示される                |
| 10 | 別ユーザーの項目 ID に直接アクセスする          | UI/API ともに内容が見えず、403/404 相当になる                  |
| 11 | データをエクスポートし、空環境へ復元する           | 件数、状態、nullable、completedAt が一致する                |
| 12 | キーボードだけでログイン後にメモ登録・保存する        | Tab / Shift+Tab / Enter / Space で完了でき、フォーカスが見える |

| 自動検査        | axe で critical / serious 相当を 0 にする    | 必須     |
| 手動確認        | キーボードのみで主要 E2E 3 本を実行する               | 必須     |

## 5. レスポンシブチェックリスト

|        幅 | 確認対象               | 合格条件                                 |
| -------: | ------------------ | ------------------------------------ |
|    320px | 最小スマホ相当の耐性確認       | 横スクロールが発生しない。ただし主要サポート幅は 375px 以上でよい |
|    375px | 標準スマホ              | ダッシュボード、一覧、詳細、フォーム、メニューが操作可能         |
|    390px | 近年スマホ              | CTA、フィルタ、検索欄が破綻しない                   |
| 768px 未満 | DADS のモバイル・タブレット領域 | モバイルレイアウトに切り替わる                      |
|    768px | 境界値                | ナビ、グリッド、一覧が中途半端に崩れない                 |
|   1024px | 小型 PC / タブレット横     | 2カラム以上のレイアウトが成立する                    |
|   1440px | 標準デスクトップ           | 情報密度が高すぎず、空白が不自然でない                  |

重点画面は、ダッシュボード、リポジトリ一覧、リポジトリ詳細、未整理メモ一覧、項目作成フォーム、検索・絞り込み、エクスポート・復元画面、モバイルメニューです。

チェック項目は次です。

* 横スクロールが出ない。
* 主要 CTA が画面外に隠れない。
* フィルタ条件が折り返しても意味が分かる。
* 一覧カードのタイトル、状態、リポジトリ名、更新日が読める。
* 重要情報が折りたたみ内だけに隠れない。
* モバイルでボタン順序がサービス内で一貫する。
* モバイルメニューの階層が2段階を超えない。
* ボトムナビを使っていない。
* テーブルを使う場合、スマホではカード化または横スクロール領域を明示する。
* PC で情報が横に広がりすぎず、読みやすい最大幅がある。

## 6. リグレッションしやすい箇所

| 箇所                    | 起きやすい不具合                     | 対策                       |
| --------------------- | ---------------------------- | ------------------------ |
| completedAt           | 完了時に設定されない、再保存で上書き、未完了化しても残る | 単体・結合・E2E で固定            |
| repositoryId nullable | null が検索・復元・ダッシュボードから漏れる     | null 専用 fixture を常設      |
| 未整理メモ分類               | 元メモ喪失、誤分類、分類後に検索不能           | 元文保持と分類候補方式をテスト          |
| ダッシュボード               | 件数ズレ、優先順ズレ、完了項目混入            | seed データで集計 expected を固定 |
| 検索・絞り込み               | AND/OR 条件不一致、解除不能、日本語検索漏れ    | 単体 + E2E                 |
| 認証・権限                 | 他ユーザーデータ参照、ID 直打ち突破          | API 結合テスト必須              |
| エクスポート                | nullable や completedAt が欠落   | golden JSON 比較           |
| 復元                    | 部分復元、重複、関連 ID 壊れ             | トランザクションテスト              |
| フォーム                  | disabled 多用、ラベル欠落、エラーが不明     | DADS チェックリスト             |
| モバイル表示                | ボタンが画面外、フィルタが読めない            | viewport E2E             |
| モバイルメニュー              | キーボードで開閉不能、現在位置不明            | 手動 + E2E                 |
| 色状態                   | 完了/警告/エラーが色だけになる             | UI レビュー + a11y           |
| 暫定モーダル                | フォーカストラップ、閉じ方不明              | MVP では原則回避               |

# MVPでは後回しにする内容

MVP では、次は後回しにしてよいです。

* 複雑な差分復元・マージ復元。
* GitHub API と完全同期する高度なテスト。
* AI 分類の品質評価を大量データで行うこと。
* 全ブラウザ・全端末の網羅確認。
* 視覚回帰テストの全面導入。
* スクリーンリーダー全文読み上げの詳細検証。
* 高度なパフォーマンスチューニング。
* タブ、モーダル、テーブルコントロール、グローバルメニュー等の作り込み。DADS で詳細ガイドライン未整備の UI は、使う場合も暫定案として最小利用に留める。
* ボトムナビゲーション。DADS で非推奨のため MVP では採用しない。
* 高度な監査ログ、履歴復元、変更差分ビュー。
* 複数人同時編集の競合解決。

ただし、後回しにしても **completedAt、repositoryId nullable、権限、エクスポート・復元、キーボード操作、スマホ破綻防止**は外さないほうがよいです。ここを外すと、MVP の信頼性が大きく落ちます。

# データ設計への影響

QA 観点から、データ設計では次を明確にしておく必要があります。

| データ項目                      | 推奨仕様                           | テスト影響                   |
| -------------------------- | ------------------------------ | ----------------------- |
| `repositoryId`             | nullable。未紐づけ項目を正式に許容          | null fixture を全テスト層に含める |
| `completedAt`              | 完了状態なら日時、未完了なら null            | 状態遷移テスト必須               |
| `status`                   | enum として定義                     | 不正値拒否、状態別表示を確認          |
| `type`                     | 課題、バグ、実装予定、将来構想、技術、参考、アイデア、メモ等 | 分類・検索・集計の対象             |
| `sourceMemoId`             | 分類後項目が元メモを参照できる                | 元メモ保持テスト                |
| `classificationConfidence` | 自動分類を使う場合は信頼度を保持               | 低信頼時の未整理維持              |
| `schemaVersion`            | エクスポート JSON に必須                | 復元互換性テスト                |
| `ownerId`                  | 全主要データに必須                      | 認証・権限テスト                |
| `createdAt` / `updatedAt`  | サーバー側で設定                       | 並び順・復元の確認               |
| `deletedAt`                | MVP では論理削除を検討                  | 誤削除・復元対策                |

`completedAt` の推奨仕様は、次で固定するべきです。

* 未完了から完了へ遷移したとき、サーバー時刻で `completedAt` を設定する。
* 完了状態のままタイトル等を編集しても、`completedAt` は上書きしない。
* 完了から未完了へ戻したとき、`completedAt` を null にする。
* 再度完了したとき、新しい完了時刻を設定する。
* エクスポート・復元では `completedAt` を保持する。

リポジトリ削除時の挙動は、MVP では **削除ブロック**を推奨します。紐づく項目があるリポジトリを削除しようとしたら、「先に項目を移動または未紐づけにしてください」と表示します。自動で null にする仕様は便利ですが、意図しない情報散逸が起きやすいため、MVP 後に検討するのが安全です。

# 画面設計への影響

QA 観点から、画面設計では「確認しやすい構造」を最初から入れる必要があります。

ダッシュボードでは、「今やるべきこと」を最初から見える場所に置きます。重要なタスク、未整理メモ、ブロック中、期限が近いものをアコーディオン内だけに隠してはいけません。

フォームでは、DADS に沿って次を必須にします。

* ラベルを必ず表示する。
* 必須/任意を明示する。
* 入力条件や例はサポートテキストに書く。
* プレースホルダーを説明の代用にしない。
* 保存ボタンを安易に disabled にしない。
* 未入力で押されたら、具体的なエラーテキストを静的に表示する。
* エラー文は「タイトルを入力してください」のように修正方法まで書く。
* 文字数上限がある場合は常時分かるようにする。

ボタンでは、画面ごとの主操作を1つに絞ります。たとえば作成画面なら「保存」をプライマリー、キャンセルや戻るはテキストまたはセカンダリーにします。スマホでは縦積みでも意味順が変わらないようにします。

リンクでは、通常本文と明確に区別します。外部リンク、GitHub リポジトリリンク、エクスポートファイルのダウンロードリンクは、リンク先やファイル種別が分かるようにします。

モバイルでは、ボトムナビではなく、ハンバーガーメニュー + ドロワーを基本にします。メニュー内の階層は2段階以内にし、ページ遷移リンクとサブメニュー開閉をアイコンだけで曖昧に区別しない設計にします。

# API・機能設計への影響

API・機能設計では、UI からの入力を信頼せず、サーバー側で最終検証する必要があります。

| 機能                    | API・機能設計への要求                         |
| --------------------- | ------------------------------------ |
| 作成・更新                 | 必須、enum、文字数、repositoryId 所有権をサーバーで検証 |
| completedAt           | クライアントから直接任意設定させず、状態遷移からサーバーで決定      |
| repositoryId nullable | null は正常値として受け入れ、不正 ID は拒否           |
| 検索                    | ユーザー所有データに限定して検索                     |
| ダッシュボード               | 集計条件を API 側で安定化し、UI とロジックが分裂しないようにする |
| 未整理メモ分類               | 分類処理は元メモを破壊しない。失敗時も未整理のまま残す          |
| 認証                    | 未ログインはログインへ誘導、API は 401              |
| 権限                    | 他ユーザー所有データは 403 または 404。挙動は統一        |
| エクスポート                | 自分のデータのみ。schemaVersion を含める          |
| 復元                    | 事前検証、トランザクション、失敗時 rollback           |
| エラー                   | UI が具体的に表示できるエラーコードとメッセージを返す         |
| パフォーマンス               | 一覧・検索はページングまたは件数制限を用意                |

認証・権限テストでは、UI で見えないだけでは不十分です。API に他ユーザーの ID を直接渡した場合も拒否される必要があります。

# リスクと対策

| リスク                            | 影響                      | 対策                             |
| ------------------------------ | ----------------------- | ------------------------------ |
| completedAt の仕様が曖昧             | 完了履歴、ダッシュボード、エクスポートが不整合 | 状態遷移仕様を実装前に固定                  |
| repositoryId nullable が例外扱いになる | 未整理メモや横断アイデアが消える・検索漏れ   | nullable を正規仕様としてテスト fixture 化 |
| 未整理メモ分類が誤る                     | 情報が見つからない、信頼性低下         | 元メモ保持、候補提示、低信頼時は未整理            |
| E2E 過多                         | テストが遅く不安定になる            | E2E は主要導線、ロジックは単体・結合へ寄せる       |
| アクセシビリティを自動検査だけに頼る             | キーボード不能や導線不明を見逃す        | 手動キーボード確認を品質ゲート化               |
| DADS 未整備コンポーネントを多用             | 暫定 UI が増え一貫性が崩れる        | MVP ではモーダル・タブ等を最小化             |
| スマホ表示の後付け                      | 横スクロール、CTA 消失           | 375px から設計レビュー                 |
| disabled 多用                    | 利用者が次に何をすべきか分からない       | 押せる状態で具体的エラーを出す                |
| 色だけの状態表示                       | 色覚多様性や読み上げで状態不明         | テキスト・アイコン・ラベルを併用               |
| 復元処理の部分成功                      | データ破壊                   | トランザクション必須、dry-run 的検証         |
| 権限漏れ                           | 重大インシデント                | API 結合テストで他ユーザー ID を直接指定       |

# 未決定事項

開発着手前に統括担当が決めるべき未決定事項は次です。

1. 正式なデータ名
   `Item`、`Memo`、`Repository`、`Project` などの正式名称。

2. ステータス enum
   例: `todo`、`inProgress`、`blocked`、`done`、`archived`。

3. 完了解除時の `completedAt` 仕様
   本回答では「未完了へ戻したら null」を推奨。

4. リポジトリ削除時の挙動
   本回答では「紐づく項目があれば削除ブロック」を推奨。

5. 未整理メモ分類の方式
   ルールベース、LLM、手動タグ、ハイブリッドのどれにするか。

6. 復元方式
   MVP では「空環境復元」または「全量置換」を推奨。差分マージは後回し。

7. 権限モデル
   MVP が個人利用のみか、将来チーム共有を見越すか。

8. 対応ブラウザ
   MVP では Chromium 系 + Safari 系の最低確認を推奨。Firefox を含めるかは開発工数次第。

9. パフォーマンス seed 件数
   MVP の通常件数、重い件数を決める。例: 通常 100件、負荷確認 1,000件。

10. DADS 未整備コンポーネントの扱い
    モーダル、タブ、テーブルコントロールを使う場合の暫定ルール。

# 統括担当へ渡す要点

統括担当が作業計画書に入れるべき品質ゲートは次です。

## 品質ゲート案

| ゲート    | タイミング  | 必須条件                                             | リリース停止条件                          |
| ------ | ------ | ------------------------------------------------ | --------------------------------- |
| Gate 0 | 実装前    | 状態遷移、completedAt、repositoryId nullable、復元方式を仕様決定 | 仕様未決定                             |
| Gate 1 | 各 PR   | 単体テスト、型チェック、lint、主要ロジックテスト通過                     | completedAt / nullable / 権限のテスト欠落 |
| Gate 2 | 機能統合時  | API + DB 結合テスト通過                                 | 他ユーザーデータ参照可能、復元 rollback 不可       |
| Gate 3 | UI 完成時 | DADS チェック、フォーム/ボタン/リンク/モバイル確認                    | disabled 多用、ラベル欠落、色だけ状態表示         |
| Gate 4 | RC 前   | 主要 E2E 12本全通過                                    | ログイン後主要導線不能、スマホ主要操作不能             |
| Gate 5 | RC     | axe / Lighthouse 補助検査、手動キーボード確認                  | critical / serious 相当の a11y 問題    |
| Gate 6 | リリース前  | エクスポート・復元 round-trip、権限、データ整合性確認                 | P0/P1 残存                          |
| Gate 7 | リリース後  | 既知不具合、監視項目、復旧手順を記録                               | 復旧手順なし                            |

## 統括担当への短い結論

MVP では、**completedAt、repositoryId nullable、未整理メモ分類、ダッシュボード、検索、認証・権限、エクスポート・復元、DADS 準拠、キーボード操作、スマホ表示**を品質ゲートに入れるべきです。

特にリリース停止条件は明確にしてください。

* 他ユーザーのデータが見える。
* 完了状態と `completedAt` が不整合。
* `repositoryId = null` のデータが消える、検索できない、復元できない。
* 未整理メモ分類で元メモが失われる。
* エクスポート・復元で round-trip できない。
* スマホで主要操作ができない。
* キーボードのみで主要導線を完了できない。
* フォームにラベルや具体的エラーがない。
* 状態表示が色だけに依存している。
* ボトムナビを前提にしている。

この品質ゲートを満たせば、NextPatch の MVP は「小さくても壊れにくく、PC / スマホで使え、あとから機能追加しやすい」状態で開発を開始できます。
