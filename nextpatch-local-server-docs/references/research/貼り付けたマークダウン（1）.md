# 担当領域

プロダクト方針・MVP定義。

NextPatch を「普通の TODO アプリ」「Notion 風メモアプリ」「GitHub Issue 管理ツール」のいずれにも寄せすぎず、**開発中アプリ・GitHub リポジトリごとの散らばった判断材料を、次に進めるべき開発行動へ変換するアプリ**として定義する。

# 前提理解

本回答では、この会話で提示された NextPatch の概要と、確認できた `dads_app_ui_design_rules_20260411.md` を前提にする。NextPatch 要件定義書と「開発着手前に決定すべき内容の検討メモ」は本文を確認できなかったため、提示文に含まれる要件を上位前提として扱う。

DADS については、アクセシビリティ、一貫性、重要情報を隠さないこと、画面単体ではなく利用者の流れで判断することを優先する。UI 未整備コンポーネントは独自仕様として断定せず、暫定案として扱う。

GitHub Projects は、Issue、Pull Request、メモしたアイデアを table、kanban board、roadmap で扱える柔軟な作業追跡機能であり、フィルタ、ソート、グルーピング、チャート、カスタムフィールドも提供している。したがって NextPatch は GitHub Projects の代替ボードを作るのではなく、GitHub 外の検討メモや将来構想、ChatGPT との壁打ちメモを含めて「次に何をするか」を決める層に寄せるべきである。([GitHub Docs][1])

GitHub の REST API では Issue、担当者、コメント、ラベル、マイルストーンなどを扱えるため、GitHub 連携は将来実装可能だが、MVP で本格同期まで入れると認証、権限、同期差分、PR と Issue の扱いなどがスコープを大きくする。([GitHub Docs][2])

ChatGPT 側では GitHub リポジトリを接続してコードや README などを検索・引用できる機能があり、利用可能範囲はプランや体験によって異なる。NextPatch が ChatGPT 側のアプリやコネクタとして振る舞う場合、OpenAI Apps SDK / MCP の設計、ツール定義、認証、デプロイが必要になるため、MVP の中心価値検証とは分けるべきである。([OpenAI Help Center][3])

Web 参照元は GitHub Docs、OpenAI Help Center、OpenAI Developers。確認日は 2026-04-20。

# 決定すべき論点

NextPatch の中心価値は、**開発プロジェクトごとの未整理情報を、現在の開発判断に使える形へ圧縮し、「今やるべきこと」を明示すること**と定義する。

主ユーザーは、まず個人開発者、少人数開発者、複数リポジトリを同時に触る開発者、ChatGPT や Codex と壁打ちしながら開発するユーザーに置く。初期からチーム利用を主対象にすると、権限、通知、担当者、レビュー、共有ワークフローが先に肥大化し、NextPatch 独自の価値検証が遅れる。

PC 利用は、整理、比較、優先順位付け、リポジトリ別の全体把握、実装前の計画作成に使う。スマホ利用は、外出中のアイデア記録、今日やることの確認、簡単なステータス更新、未整理メモの投入に限定する。3 操作以内で目的へ到達する観点では、PC は「ホーム → リポジトリ → 次アクション」、スマホは「起動 → 今やる → 詳細」または「起動 → 追加 → 保存」を基本導線にする。

MVP で決めるべき最重要論点は、次の 7 点である。

| 論点         | 決める内容                                              |
| ---------- | -------------------------------------------------- |
| 何を中心単位にするか | リポジトリ / 開発中アプリを中心単位にする                             |
| 何を成果と呼ぶか   | タスク完了ではなく「次に着手すべき開発行動が明確になること」                     |
| GitHub 連携  | MVP では外部リンク保存まで。本格同期は P1 以降                        |
| ChatGPT 連携 | MVP ではコピペ取り込みと構造化入力まで。API / Apps SDK / MCP は P1 以降 |
| チーム利用      | MVP は個人利用。データモデルだけ将来チーム化に備える                       |
| メモとタスクの扱い  | 同じ箱に混ぜず、メモからアクション候補へ変換する流れを持たせる                    |
| UI 方針      | DADS 優先。重要な次アクションは隠さない。未整備部品は暫定案扱い                 |

# 選択肢比較

## 1. プロダクトの寄せ方

| 選択肢              | 内容                         | 利点               | 問題                                    |
| ---------------- | -------------------------- | ---------------- | ------------------------------------- |
| TODO アプリ型        | タスクを列挙し完了チェックする            | 作りやすい、理解されやすい    | 開発文脈、判断理由、将来構想、参考サービスが扱いにくい           |
| Notion 風メモ型      | 自由なページ、DB、メモ整理を中心にする       | 汎用性が高い           | 自由すぎて「今やるべきこと」が埋もれる                   |
| GitHub Issue 管理型 | Issue、PR、ラベル、マイルストーン中心     | 開発者に馴染みやすい       | GitHub Projects / Issues と競合し、独自価値が薄い |
| 開発判断ダッシュボード型     | リポジトリごとの課題・メモ・構想を次アクションへ変換 | NextPatch 独自性が出る | 設計を誤ると抽象的になりやすい                       |

推奨は **開発判断ダッシュボード型**。理由は、NextPatch の価値を「記録」ではなく「開発判断の圧縮」に置けるためである。

## 2. GitHub 連携を MVP に入れるか

| 選択肢        | 内容                                | 評価                            |
| ---------- | --------------------------------- | ----------------------------- |
| 入れない       | リポジトリ名、URL、関連 Issue / PR URL を手入力 | MVP では推奨。独自価値検証に集中できる         |
| 読み取り専用で入れる | Issue / PR を一覧取得                  | P1 候補。便利だが認証と同期設計が必要          |
| 双方向同期まで入れる | NextPatch から Issue 更新、ラベル更新       | P2。GitHub 管理ツール化しやすく、MVP には重い |

GitHub Projects は既に GitHub データと同期し、Issue / PR と連動する作業追跡を提供しているため、MVP で同等機能を作る必要はない。NextPatch は GitHub の外側にある「判断材料」を整理する層に置くべきである。([GitHub Docs][4])

## 3. ChatGPT 連携を MVP に入れるか

| 選択肢               | 内容                             | 評価                          |
| ----------------- | ------------------------------ | --------------------------- |
| 連携なし              | ユーザーが手入力する                     | 価値検証には不足                    |
| コピペ取り込み           | ChatGPT との会話メモを貼り付け、手動で分類する    | MVP では推奨                    |
| 構造化補助             | 貼り付けたメモを「課題、候補、決定、保留」に分ける入力 UI | MVP 後半または P1                |
| OpenAI API 連携     | 要約、分類、優先度案を生成                  | P1。費用、API 設計、プロンプト評価が必要     |
| ChatGPT App / MCP | ChatGPT から NextPatch を操作する     | P2。MCP サーバー、ツール定義、認証、UI が必要 |

OpenAI Apps SDK では、ChatGPT アプリを作るには MCP サーバーが必要で、ツールや入出力契約を設計する必要がある。ツール設計でも、1 ツール 1 ジョブ、明示的入力、予測可能な出力、読み取りと書き込みの分離が推奨されているため、MVP で入れるには重い。([OpenAI開発者][5])

## 4. 個人利用かチーム利用か

| 選択肢      | 内容                 | 評価         |
| -------- | ------------------ | ---------- |
| 個人利用特化   | 1 人の開発者が複数リポジトリを管理 | MVP 推奨     |
| 小規模チーム対応 | 共有、担当者、コメント、通知を持つ  | P2         |
| 最初から組織利用 | RBAC、監査、権限、連携管理    | 初期リリースでは過剰 |

推奨は、**MVP は個人利用。ただしデータモデルに workspace / owner を持たせ、後からチームへ拡張できる余地を残す**ことである。

# 推奨案

NextPatch は、**リポジトリ単位の開発判断 OS**として定義する。

中心価値は、次の 1 文に集約する。

**「NextPatch を開けば、各リポジトリで次に進めるべき 1〜3 個の開発行動と、その理由が分かる。」**

この価値を成立させるために、MVP では「全部を管理できる」よりも「迷いを減らす」を優先する。ホーム画面はタスク一覧ではなく、**Now / Inbox / Repos** の 3 領域を中心にする。

推奨するプロダクト方針は次の通り。

| 項目       | 推奨案                                  |
| -------- | ------------------------------------ |
| 中心単位     | Project / Repository                 |
| 中心画面     | Now 画面                               |
| 中心データ    | Item ではなく「判断材料つき Action」             |
| MVP の入力  | 手入力、URL、ChatGPT メモ貼り付け               |
| MVP の連携  | GitHub URL 保存まで。API 同期なし             |
| MVP の AI | なし、または手動整理を助けるテンプレートまで               |
| 初期ユーザー   | 個人開発者、少人数開発者                         |
| 将来像      | GitHub / ChatGPT / Codex とつながる開発判断ハブ |

「今やるべきこと」は、AI の推測だけで成立させず、**ユーザーが理解できるルールベースの表示**で成立させる。たとえば、Impact、Urgency、Effort、Confidence、Blocked、Freshness、Due date、Manual pin を使い、Now に出す理由をチップで見せる。

例：

| 表示                         | 理由チップ                          |
| -------------------------- | ------------------------------ |
| 認証エラーの再現条件を整理する            | Bug / 影響大 / Blocker なし / 30分以内 |
| README のセットアップ手順を直す        | Release前 / Effort小 / ユーザー影響あり  |
| GitHub Issue 連携仕様を後回し判断にする | Scope整理 / MVP判断待ち              |

この「理由が見える優先順位」が、NextPatch を TODO アプリやメモアプリから分ける核になる。

## 初期リリースの完成定義

初期リリースは、次のシナリオが迷わず完了できれば完成とする。

1. ユーザーが 3 つ以上のリポジトリを登録できる。
2. 各リポジトリに、バグ、実装予定、将来構想、技術メモ、参考 URL、ChatGPT メモを登録できる。
3. 未整理メモを Inbox に入れ、Triage で Action / Memo / Decision / Someday に分けられる。
4. 各リポジトリで Next Action を 1 件設定できる。
5. ホームの Now 画面で、全体から今やるべき候補が理由付きで表示される。
6. PC では詳細整理、スマホでは Now 確認とクイック追加ができる。
7. GitHub や ChatGPT の API 連携がなくても、NextPatch の価値が理解できる。

# MVPでは後回しにする内容

GitHub OAuth、Issue / PR 同期、Webhook、ラベル同期、GitHub Projects 連携は後回しにする。これらは便利だが、MVP の価値である「散らばった判断材料から次アクションを見つける」ことを検証する前に実装すると、GitHub 管理ツール化してしまう。

OpenAI API による自動分類、ChatGPT App、MCP、Codex 連携も後回しにする。Apps SDK / MCP を使う場合、MCP サーバー、ツール、認証、UI、デプロイ、ツール説明の設計が必要で、MVP の検証範囲を超える。([OpenAI開発者][6])

チーム機能も後回しにする。共有、コメント、担当者、通知、権限管理を入れると、NextPatch の中心価値よりも運用管理の設計が先行するためである。

また、ロードマップ、ガントチャート、複雑なカンバン、タブ UI、モーダル多用、ボトムナビゲーションは MVP では避ける。DADS ではボトムナビゲーションが非推奨であり、モーダル、タブ、ページナビゲーションなどは詳細ガイドライン未整備とされているため、使う場合は暫定案として扱う。

# データ設計への影響

MVP のデータ設計は、単なる `tasks` テーブルではなく、**リポジトリ、判断材料、次アクション、根拠**を分ける必要がある。

推奨する主要エンティティは次の通り。

| エンティティ               | 役割                                  |
| -------------------- | ----------------------------------- |
| Workspace            | 将来のチーム化に備えた単位。MVP では個人ワークスペースのみ     |
| User                 | 所有者。MVP では 1 人利用前提                  |
| Project / Repository | 開発中アプリまたは GitHub リポジトリ              |
| Item                 | 課題、バグ、実装予定、将来構想、技術メモ、参考、アイデア、決定事項   |
| Memo                 | ChatGPT メモ、壁打ちメモ、自由記述               |
| Source               | GitHub URL、Web URL、ChatGPT、手入力などの出所 |
| Link                 | Issue、PR、参考サービス、ドキュメントなどへの外部リンク     |
| NextAction           | 今やる候補。Item から派生してもよい                |
| Decision             | 採用技術、後回し判断、仕様判断など                   |
| ReviewState          | 最終確認日、放置期間、見直し対象                    |
| Tag                  | 任意分類                                |
| StatusHistory        | 状態変更履歴                              |

Item には最低限、次のフィールドを持たせる。

| フィールド            | 目的                                                                            |
| ---------------- | ----------------------------------------------------------------------------- |
| type             | bug / task / idea / tech / reference / future / decision / memo               |
| status           | inbox / triage / next / doing / blocked / waiting / done / someday / archived |
| impact           | 影響度                                                                           |
| urgency          | 緊急度                                                                           |
| effort           | 手間                                                                            |
| confidence       | 確信度                                                                           |
| blocked          | 着手可能か                                                                         |
| due_date         | 期限                                                                            |
| repo_id          | 対象リポジトリ                                                                       |
| next_action_text | 次にやる具体行動                                                                      |
| reason_summary   | Now に出る理由                                                                     |
| source_type      | manual / chatgpt / github_link / web                                          |
| source_url       | 外部参照 URL                                                                      |
| pinned           | 手動優先                                                                          |
| reviewed_at      | 最終見直し日                                                                        |

重要なのは、**Memo と Action を同一視しないこと**である。ChatGPT との壁打ち結果はまず Memo / Inbox に入り、Triage を通じて Action、Decision、Future、Reference に変換される。この変換構造が NextPatch の独自価値になる。

# 画面設計への影響

MVP の主要画面は、次の 5 画面に絞る。

| 画面                | 目的                      |
| ----------------- | ----------------------- |
| Now               | 全リポジトリ横断で今やることを表示       |
| Repositories      | リポジトリ一覧と状態確認            |
| Repository Detail | そのリポジトリの課題、メモ、構想、次アクション |
| Inbox / Triage    | 未整理メモを分類する              |
| Item Detail       | 根拠、リンク、判断、次アクションを編集する   |

PC では、Now 画面を 2〜3 カラム構成にする。左にリポジトリ一覧、中央に Now / Next Action、右に Inbox / 注意事項を置く構成が有力である。DADS ではグリッド、余白段階、ブレークポイントを定めることが求められており、デスクトップは 768px 以上、モバイル・タブレットは 768px 未満を境に扱う方針が示されているため、MVP でもこの切り替えを前提にする。

スマホでは、画面数を減らし、次の 2 目的に絞る。

| スマホ目的    | 導線           |
| -------- | ------------ |
| 今やることを見る | 起動 → Now     |
| 思いつきを入れる | 起動 → 追加 → 保存 |

モバイルでボトムナビゲーションを使う案は避ける。DADS で非推奨とされているため、MVP では上部の主要 CTA、モバイルメニュー、またはホーム内カードによる遷移を暫定案とする。

フォームでは、プレースホルダーに頼らず、ラベル、要否ラベル、サポートテキスト、具体的なエラーテキストを使う。ボタンは 1 画面 1 プライマリーを基本にし、Now 画面では最重要行動を強く出す。

# API・機能設計への影響

MVP の API / 機能は、外部連携よりも内部整理に寄せる。

P0 で必要な機能は次の通り。

| 機能                | 内容                                       |
| ----------------- | ---------------------------------------- |
| Repository CRUD   | リポジトリ登録、編集、一覧、詳細                         |
| Item CRUD         | 課題、バグ、予定、構想、メモ、決定事項の登録                   |
| Inbox Add         | 未整理メモの追加                                 |
| Triage Update     | Inbox から type / status / next action を設定 |
| NextAction Update | 各リポジトリの次アクション設定                          |
| Now Query         | 優先候補を横断取得                                |
| Reason Builder    | 表示理由チップを生成                               |
| Search / Filter   | 種別、状態、タグ、リポジトリで絞り込み                      |
| Link Management   | 外部 URL の保存                               |
| Review Mark       | 見直し済み、後回し、アーカイブ                          |

Now Query は、完全自動ではなく、次のような説明可能なルールにする。

```text
Now候補 =
  pinned が true
  または status が next / doing
  または blocked が false かつ impact または urgency が高い
  または due_date が近い
  または reviewed_at が古く、かつ status が next
```

表示順は、概念的には次のように決める。

```text
score =
  impact
+ urgency
+ confidence
+ pinned bonus
+ due soon bonus
- effort penalty
- blocked penalty
- stale uncertainty penalty
```

ただし MVP では数式を UI に出さず、ユーザーには「影響大」「期限近い」「手間小」「手動固定」「ブロックなし」のような理由チップで見せる。

GitHub API、OpenAI API、Apps SDK、MCP、Webhook、バックグラウンド同期は P0 から外す。将来 GitHub / ChatGPT とつなげるために、`external_source`、`external_id`、`source_url`、`source_type` は P0 から持たせる。

# リスクと対策

| リスク           | 内容                           | 対策                                                  |
| ------------- | ---------------------------- | --------------------------------------------------- |
| TODO アプリ化     | チェックリストだけになり独自価値が消える         | Item に source、reason、decision、next_action を必須級で持たせる |
| Notion 化      | 自由メモが増え、次アクションが埋もれる          | Inbox → Triage → Now の流れを必須導線にする                    |
| GitHub 管理ツール化 | Issue / PR 管理が中心になる          | MVP では GitHub URL 保存までに制限する                         |
| AI 依存         | AI がないと価値が出ない設計になる           | MVP は手動・ルールベースで価値を成立させる                             |
| 優先順位の不信       | なぜ Now に出たか分からない             | 理由チップと手動固定を用意する                                     |
| 情報過多          | 管理項目が多くなり使われなくなる             | Now は 1〜3 件、各リポジトリの Next Action は 1 件を基本にする         |
| スマホ UI の肥大化   | PC と同じ機能を詰め込む                | スマホは確認とクイック追加に限定する                                  |
| DADS 未整備部品の断定 | タブ、モーダル等の仕様を勝手に決める           | 未整備部品は暫定案として扱い、多用しない                                |
| チーム対応の先行      | 権限や通知が中心になり MVP が遅れる         | データモデルだけ workspace 対応、UI は個人向けにする                   |
| 連携スコープ肥大      | GitHub / ChatGPT 連携でリリースが遅れる | P0 は手入力、URL、コピペ。P1 以降で連携する                          |

# 未決定事項

次は統括側で最終判断が必要である。

| 未決定事項               | 推奨する初期判断                          |
| ------------------- | --------------------------------- |
| 認証方式                | 個人利用でも PC / スマホ同期のためログインありを推奨     |
| 保存先                 | クラウド保存を推奨。完全ローカルはスマホ連携と相性が悪い      |
| Now の上限             | 全体 3 件、リポジトリごと 1 件を推奨             |
| Item type の初期数      | 7 種類以内に抑える                        |
| Effort / Impact の段階 | 3 段階から始める                         |
| ChatGPT メモ取り込み      | MVP は貼り付け欄。自動要約は P1               |
| GitHub 連携の開始時期      | MVP 完成後、read-only Issue / PR 取得から |
| チーム対応               | P2。データモデルだけ先に備える                  |
| モバイルメニュー方式          | DADS に従い、ボトムナビゲーションは避ける           |
| タブ利用                | DADS 未整備のため、MVP では暫定案として必要最小限にする  |

# 統括担当へ渡す要点

NextPatch の MVP は、**GitHub Issue 管理でも、Notion 風メモでも、単純 TODO でもなく、リポジトリごとの開発判断を「今やること」に変換する個人向け開発管理アプリ**として定義する。

## MVPスコープ案

* 個人利用前提のワークスペース
* リポジトリ / 開発中アプリ登録
* Now 画面
* リポジトリ別ダッシュボード
* Inbox
* Triage
* Item 登録
* Item 種別：task、bug、idea、future、tech、reference、decision、memo
* Next Action 設定
* 優先判断フィールド：impact、urgency、effort、confidence、blocked、due、pinned
* Now 表示理由チップ
* GitHub / PR / Issue / 参考 URL の外部リンク保存
* ChatGPT メモの貼り付け保存
* 検索、フィルタ、タグ
* PC は整理・比較・優先判断中心
* スマホは Now 確認とクイック追加中心
* DADS 準拠の基本 UI
* DADS 未整備コンポーネントは暫定案扱い

## 非スコープ案

* GitHub OAuth
* GitHub Issue / PR 自動同期
* GitHub Projects 代替機能
* GitHub への書き戻し
* OpenAI API による自動分類
* ChatGPT App / MCP 連携
* Codex 連携
* チーム共有
* 担当者、コメント、通知、権限管理
* ガントチャート
* 汎用 Notion 風ページビルダー
* 汎用 TODO アプリ機能
* CI / CD 監視
* コードレビュー機能
* ボトムナビゲーション前提のスマホ UI
* タブ、モーダル、ページナビゲーションの独自仕様断定
