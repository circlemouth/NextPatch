# 担当領域

バックアップ・エクスポート・運用。

NextPatch に蓄積される「課題、進捗、バグ、実装予定、将来構想、採用技術、参考サービス、アイデア、ChatGPT 壁打ちメモ」を、障害・誤削除・サービス移行・ローカル保管・将来の Git 管理に耐える形で保全するための設計判断を担当する。

# 前提理解

この会話内で本文を確認できた前提資料は `dads_app_ui_design_rules_20260411.md` です。NextPatch 要件定義書と検討メモの本文は確認できなかったため、本回答ではユーザー提示の NextPatch 概要を要件側の前提として扱います。

UI に関わる判断では、DADS 資料の「アクセシビリティを前提にする」「サービス内で一貫性を保つ」「重要情報を隠さない」「ボタン優先度を揃える」「フォームではラベル・サポートテキスト・具体的エラーを用意する」「モーダルダイアログやプログレスインジケーター等は未整備項目として扱う」という方針を優先します。DADS 未整備の UI コンポーネントは、本回答ではすべて「暫定案」として扱います。

Web 参照元は以下です。確認日はいずれも **2026年4月20日** です。

* JSON 仕様: IETF RFC 8259。JSON は軽量・テキストベース・言語非依存の構造化データ交換形式として定義されています。([IETF Datatracker][1])
* CSV 仕様: RFC 4180。CSV の MIME type `text/csv`、ヘッダー行、レコード、カンマ区切り、引用符の扱いが整理されています。([RFC エディター][2])
* Markdown 仕様: CommonMark 0.31.2。Markdown を読みやすいプレーンテキスト形式として扱い、曖昧さを減らす仕様として参照します。([CommonMark仕様][3])
* 機密情報の Git 管理リスク: GitHub Docs。機密情報をリポジトリ履歴から削除するには履歴書き換え、クローン済み利用者との調整、シークレットの失効・ローテーションが必要になるため、バックアップの Git 保存は慎重に扱うべきです。([GitHub Docs][4])
* バックアップ運用の安全性: OWASP Secrets Management Cheat Sheet。自動バックアップ、復元手順のテスト、バックアップの暗号化・アクセス制限が推奨されています。([OWASP Cheat Sheet Series][5])
* DB バックアップ方式: SQLite は Online Backup API により稼働中 DB のスナップショット作成を想定でき、PostgreSQL は `pg_dump` / `pg_restore` がポータブルなバックアップ・移行手段として使えます。([SQLite][6])

# 決定すべき論点

最重要の決定は、**NextPatch のバックアップは JSON を唯一の可逆バックアップ形式にする**ことです。Markdown と CSV は便利だが、復元の正本にはしない。

決定すべき論点は次のとおりです。

| 論点              | 決定すべき内容                                         |
| --------------- | ----------------------------------------------- |
| バックアップの正本       | JSON、DB dump、Markdown、CSV のどれを復元可能な正本にするか       |
| エクスポート範囲        | ワークスペース全体、リポジトリ単位、選択アイテム単位のどこまで MVP に含めるか       |
| 復元方式            | 既存データ上書き、既存データへマージ、新規ワークスペース復元のどれを標準にするか        |
| 自動バックアップ        | 定期バックアップ、重要操作前バックアップ、DB レベルバックアップのどこまで MVP 化するか |
| 削除モデル           | アーカイブ、論理削除、物理削除をどう分けるか                          |
| 添付ファイル          | バイナリ添付を持つか、URL のみとするか                           |
| GitHub リポジトリ内保存 | 将来の Markdown / JSON 同期機能として扱うか、MVP から除外するか      |
| 機密情報            | バックアップファイルに API キー、認証情報、未公開アイデアが含まれるリスクをどう扱うか   |
| スキーマ変更          | DB スキーマとエクスポートスキーマのバージョン管理・マイグレーション方針           |
| UI              | ユーザーが 3 操作以内でバックアップ・復元に到達できる画面構成                |

# 選択肢比較

| 項目           | 選択肢A             | 選択肢B             | 選択肢C                   | 判断                                                                                 |
| ------------ | ---------------- | ---------------- | ---------------------- | ---------------------------------------------------------------------------------- |
| 可逆バックアップ形式   | JSON             | DB dump          | Markdown               | **JSON を採用**。DB 種別に依存せず、NextPatch のドメイン構造を保持できる。Markdown は可読性、DB dump は運用バックアップ向け。 |
| Markdown の扱い | 復元対象にする          | 読み物・Git 管理用にする   | 作らない                   | **読み物・Git 管理用**。Markdown から完全復元させると仕様が複雑化する。CommonMark 準拠を基本にし、独自メタデータは暫定案扱い。      |
| CSV の扱い      | 復元対象にする          | 一覧分析用にする         | 作らない                   | **一覧分析用**。構造・関連・本文 Markdown・履歴の復元には不向き。                                            |
| 復元方式         | 既存上書き            | 既存へマージ           | 新規ワークスペースへ復元           | **MVP は新規ワークスペース復元**。誤上書きと ID 衝突を避けられる。マージは後回し。                                    |
| 自動バックアップ     | 定期バックアップ UI まで作る | 重要操作前のみ          | 作らない                   | **MVP は重要操作前 + 運用手順としての DB 定期バックアップ**。UI 付きスケジューラは後回し。                             |
| 削除           | 即時物理削除           | 論理削除のみ           | アーカイブ + 論理削除 + 明示的物理削除 | **3層構造を採用**。通常はアーカイブ、誤操作はゴミ箱、完全削除は明示操作。                                            |
| 添付ファイル       | バイナリを保存          | URL のみ           | 外部ストレージ連携              | **MVP は URL のみ**。添付ファイルはバックアップ容量、権限、ウイルスチェック、ZIP 仕様が増えるため後回し。                      |
| GitHub 保存    | MVP で自動コミット      | 手動 Markdown 出力のみ | なし                     | **MVP は手動 Markdown 出力のみ**。リポジトリ履歴に機密情報が残るリスクが高い。                                   |

# 推奨案

推奨案は、**「JSON は復元用、Markdown は人間が読む用、CSV は一覧分析用、DB バックアップは運用保険」**という役割分担にすることです。

MVP のバックアップ方針は次のように明確化します。

1. **JSON エクスポートを NextPatch の唯一の可逆バックアップ形式にする。**
   DB が SQLite でも PostgreSQL でも、アプリのデータモデルを安定して移行できるようにするため。JSON は構造化データ交換形式として標準化されており、NextPatch 独自の `schemaVersion` と組み合わせやすい。([IETF Datatracker][1])

2. **Markdown エクスポートは、リポジトリごとの知識整理・レビュー・将来の Git 管理用にする。**
   Markdown は可読性に優れるが、完全復元に必要な関係性・履歴・削除状態・内部 ID を厳密に表すには不向き。CommonMark 準拠の本文を基本にし、YAML front matter など独自拡張を使う場合は「暫定案」とする。([CommonMark仕様][3])

3. **CSV エクスポートは、課題一覧・バグ一覧・進捗一覧の表計算分析用に限定する。**
   CSV はフラットな表形式に向いているが、ネストしたメモ、タグ、多対多関連、Markdown 本文、履歴復元には向かない。RFC 4180 に寄せ、ヘッダー行あり、UTF-8、引用符エスケープありで出力する。([RFC エディター][2])

4. **復元は、MVP では既存データ上書きではなく「新規ワークスペースとして復元」に限定する。**
   既存データへのマージは、ID 衝突、同名リポジトリ、重複タグ、削除済みデータの復活などが難しい。MVP では安全性を優先する。

5. **削除は、アーカイブ、論理削除、物理削除を分離する。**
   「使わないが残したい」はアーカイブ、「誤操作から戻したい」は論理削除、「将来のバックアップにも残したくない」は物理削除とする。

6. **添付ファイルは MVP では扱わず、URL だけを保存する。**
   NextPatch の本質は開発知識の整理であり、GitHub Issue、PR、設計資料、Figma、Notion、公式ドキュメントなどへの URL 参照で多くを満たせる。バイナリ添付は、ファイルアップロード UI、保管容量、バックアップ ZIP、権限管理、マルウェア対策が必要になるため後回しにする。

7. **GitHub リポジトリ内保存は将来案にする。**
   バックアップファイルや Markdown 出力を GitHub に置くと、API キー、社内情報、未公開構想、ChatGPT メモが履歴に残るリスクがある。GitHub Docs でも、機密情報を履歴から除去するには履歴書き換えや関係者調整、シークレットの失効・ローテーションが必要になるとされているため、MVP では自動コミットしない。([GitHub Docs][4])

DB 別の運用判断は条件付きです。

| DB             | 推奨運用                                                                                                                          |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| SQLite 採用時     | アプリ稼働中のファイルコピーではなく、SQLite Online Backup API などスナップショット性のある方法を使う。([SQLite][6])                                                 |
| PostgreSQL 採用時 | `pg_dump` / `pg_restore` を運用バックアップ・移行検証に使う。`pg_dump` はスナップショットとして一貫性があり、別アーキテクチャや新しい PostgreSQL への移行にも使いやすい。([PostgreSQL][7]) |

# MVPに含めるべき内容

MVP に含めるべきバックアップ方針は以下です。

1. **JSON フルバックアップ**

   * ワークスペース全体を JSON で出力する。
   * アクティブ、アーカイブ済み、論理削除済みを含める。
   * 物理削除済みデータは含めない。
   * `schemaVersion`、`exportedAt`、`appVersion`、件数、ハッシュを含める。
   * 復元可能な唯一の正本にする。

2. **Markdown エクスポート**

   * リポジトリ単位で出力する。
   * 課題、バグ、実装予定、将来構想、採用技術、参考サービス、ChatGPT メモを読みやすい文書にする。
   * 復元用途ではなく、ローカル保管・レビュー・将来の Git 管理用と明記する。

3. **CSV エクスポート**

   * 課題一覧・バグ一覧・実装予定一覧など、表形式にできるデータを出力する。
   * 本文の完全表現や関連復元は対象外。
   * 表計算ソフトでの確認・棚卸し用とする。

4. **JSON インポート検証**

   * ファイル選択後、即復元せずに検証する。
   * フォーマット、`schemaVersion`、必須フィールド、参照整合性、件数、作成元バージョンを確認する。
   * エラーは DADS 方針に沿って、何が悪く、どう直せるかを静的に表示する。

5. **JSON 復元**

   * MVP では「新規ワークスペースとして復元」のみ。
   * 復元後、元ワークスペースには影響しない。
   * 復元結果として、件数、スキップ件数、警告、復元先ワークスペース名を表示する。

6. **重要操作前の自動バックアップ**

   * インポート復元前。
   * 物理削除前。
   * DB マイグレーション前。
   * 大量アーカイブ・大量削除前。
   * ただし、ユーザーがスケジュールを細かく設定する UI は MVP から外す。

7. **アーカイブと論理削除**

   * アーカイブは通常一覧から隠すが検索・復元可能。
   * 論理削除はゴミ箱相当として扱い、復元可能。
   * 物理削除は明示的な最終操作に限定する。

8. **バックアップ機密情報警告**

   * JSON、Markdown、CSV の生成前に「バックアップには機密情報が含まれる可能性があります」と明示する。
   * GitHub に保存する場合の注意を表示する。
   * バックアップ保存場所の選択はユーザー責任であることを明記する。

# MVPでは後回しにする内容

MVP では以下を後回しにする。

| 後回し項目            | 理由                                             |
| ---------------- | ---------------------------------------------- |
| Markdown からの復元   | 可読文書と復元仕様を両立させると複雑化するため。                       |
| CSV からの復元        | データ構造が欠落しやすく、重複・関連・本文 Markdown の扱いが難しいため。      |
| 既存ワークスペースへのマージ復元 | ID 衝突、同名 repo、タグ統合、削除状態の衝突が発生するため。             |
| 差分バックアップ         | 初期はフル JSON の方が検証しやすい。                          |
| 世代管理 UI          | バックアップ運用画面が肥大化するため。                            |
| クラウド保存先連携        | Google Drive、Dropbox、S3 等は認証・権限・暗号化設計が必要。      |
| GitHub への自動コミット  | 機密情報が Git 履歴に残るリスクが高い。                         |
| 添付ファイル保存         | 容量、権限、ウイルス対策、ZIP 復元仕様が必要。                      |
| パスワード付き暗号化エクスポート | 将来は検討価値が高いが、MVP では保存先注意・機密警告・サーバー側アクセス制限を優先する。 |
| ポイントインタイムリカバリ    | 個人開発向け MVP としては過剰。                             |
| 復元進捗の詳細プログレス UI  | DADS でプログレスインジケーターは未整備のため、必要なら暫定案扱い。           |

# データ設計への影響

データ設計では、**復元可能性を後付けにしない**ことが重要です。

最低限、主要エンティティには以下を持たせる。

| フィールド                     | 目的                                                                               |
| ------------------------- | -------------------------------------------------------------------------------- |
| `id`                      | エクスポート・復元で使う安定 ID。UUID / ULID など DB 依存しない値が望ましい。                                 |
| `workspaceId`             | 復元先分離のため。                                                                        |
| `repositoryId`            | GitHub リポジトリ、ローカルプロジェクト、参考サービス単位の関連付け。                                           |
| `type`                    | `issue`, `bug`, `todo`, `idea`, `tech`, `reference`, `chat_note`, `decision` など。 |
| `title`                   | 一覧・Markdown 見出し・CSV の主キー的表示に使う。                                                  |
| `bodyMarkdown`            | 本文の正本。                                                                           |
| `status`                  | `open`, `in_progress`, `done`, `blocked` など。                                     |
| `priority`                | 優先度。                                                                             |
| `tags`                    | タグ ID またはタグ名配列。                                                                  |
| `externalLinks`           | GitHub Issue、PR、公式 docs、参考サービス等の URL。                                            |
| `createdAt` / `updatedAt` | 履歴・差分・並び替えに必要。                                                                   |
| `archivedAt`              | アーカイブ判定。                                                                         |
| `deletedAt`               | 論理削除判定。                                                                          |
| `purgedAt`                | 物理削除ログを残す場合のみ。MVP では監査ログ側でもよい。                                                   |

追加で必要な運用系テーブルは以下。

| テーブル                | MVP 必須度 | 内容                                             |
| ------------------- | ------: | ---------------------------------------------- |
| `export_logs`       |       中 | 形式、範囲、件数、ハッシュ、作成日時、成功/失敗。ファイル本体は保存しなくてもよい。     |
| `import_jobs`       |       高 | 検証結果、復元先 workspace、元 schemaVersion、件数、警告、失敗理由。 |
| `backup_snapshots`  |       中 | 重要操作前バックアップの保存場所、作成理由、保持期限。                    |
| `schema_migrations` |       高 | DB マイグレーション適用履歴。使用 ORM / DB に応じて既存機構を使ってよい。    |
| `audit_logs`        |       中 | 物理削除、復元、インポート、エクスポートの記録。                       |

JSON エクスポート仕様案は以下。

```json
{
  "format": "nextpatch.backup",
  "schemaVersion": 1,
  "exportedAt": "2026-04-20T00:00:00.000Z",
  "app": {
    "name": "NextPatch",
    "version": "0.1.0"
  },
  "scope": {
    "type": "workspace",
    "workspaceId": "wk_01H...",
    "repositoryIds": ["repo_01H..."]
  },
  "options": {
    "includeArchived": true,
    "includeDeleted": true,
    "includeAuditLogs": false,
    "includeAttachments": false,
    "redaction": "none"
  },
  "entities": {
    "workspaces": [],
    "repositories": [],
    "items": [],
    "tags": [],
    "itemTags": [],
    "externalLinks": [],
    "decisions": [],
    "notes": [],
    "relations": []
  },
  "integrity": {
    "counts": {
      "repositories": 3,
      "items": 120,
      "tags": 18
    },
    "contentHash": "sha256:..."
  }
}
```

JSON 仕様の方針。

* 文字コードは UTF-8。
* 日時は UTC の ISO 8601 文字列。
* 復元に必要な内部 ID を含める。
* DB の連番 ID に依存しない。
* `schemaVersion` は必須。
* 不明フィールドは原則無視できるようにする。
* 必須フィールド欠落は検証エラー。
* JSON は人間が読める整形版を基本にする。ただし大容量化した場合は将来 gzip 圧縮を検討する。
* 添付ファイルは MVP では含めない。
* シークレット検出・マスキングは将来案。ただし警告表示は MVP 必須。

Markdown エクスポート仕様案は以下。

```text
nextpatch-export-20260420-090000/
  README.md
  repositories/
    owner__repo-a/
      README.md
      backlog.md
      bugs.md
      roadmap.md
      decisions.md
      notes/
        2026-04-20-chatgpt-memo.md
      items/
        issue_np_001-title.md
        bug_np_014-title.md
```

Markdown ファイルの基本構成。

```markdown
# owner/repo-a

## 概要

## 今やるべきこと

## 未完了の課題

## バグ

## 実装予定

## 将来構想

## 採用技術

## 参考リンク

## ChatGPT 壁打ちメモ
```

個別アイテム Markdown の基本構成。

```markdown
# タイトル

- 種別: bug
- 状態: open
- 優先度: high
- リポジトリ: owner/repo-a
- タグ: ui, export
- 作成日: 2026-04-20
- 更新日: 2026-04-20
- NextPatch ID: item_01H...

## 本文

## 関連リンク

## メモ

## 履歴要約
```

Markdown 仕様の方針。

* CommonMark で安全に読める構文を基本にする。([CommonMark仕様][3])
* 復元用ではないため、完全な内部関係は JSON に任せる。
* YAML front matter を使う場合は CommonMark 外の独自仕様なので「暫定案」とする。
* Git 管理向けには、1 ファイルに詰め込みすぎず、リポジトリ単位・種別単位で分割する。
* ファイル名には内部 ID と slug を含め、タイトル変更で追跡不能にならないようにする。
* 機密情報警告を `README.md` 冒頭に入れる。

CSV エクスポート仕様案。

```text
items.csv
repositories.csv
tags.csv
links.csv
```

`items.csv` の列案。

| 列               | 内容                                                       |
| --------------- | -------------------------------------------------------- |
| `id`            | NextPatch 内部 ID                                          |
| `repository`    | `owner/name` またはローカル repo 名                              |
| `type`          | issue / bug / todo / idea / tech / reference / chat_note |
| `status`        | open / in_progress / done / blocked                      |
| `priority`      | high / medium / low                                      |
| `title`         | タイトル                                                     |
| `body_text`     | Markdown をプレーン寄りにした本文                                    |
| `tags`          | `;` 区切り                                                  |
| `external_urls` | `;` 区切り                                                  |
| `created_at`    | UTC                                                      |
| `updated_at`    | UTC                                                      |
| `archived_at`   | 空欄可                                                      |
| `deleted_at`    | 空欄可                                                      |

CSV 仕様の方針。

* ヘッダー行あり。
* UTF-8。
* カンマ、改行、ダブルクォートを含む値は RFC 4180 に沿って引用符で囲む。([RFC エディター][2])
* CSV からの復元は MVP では非対応。
* CSV は削除済みデータをデフォルトで含めない。JSON バックアップとの差を明記する。

# 画面設計への影響

画面は **設定 > データ管理** に集約する。目的まで 3 操作以内を目指す。

推奨する画面構成。

```text
設定
  └ データ管理
      ├ バックアップを作成
      ├ エクスポート
      ├ 復元
      └ アーカイブと削除
```

画面内の主操作は次のようにする。

| セクション     | 主ボタン                               | 副操作          |
| --------- | ---------------------------------- | ------------ |
| バックアップを作成 | `JSONバックアップを作成`                    | `含まれるデータを確認` |
| エクスポート    | `Markdownを書き出す`                    | `CSVを書き出す`   |
| 復元        | `JSONバックアップを選択` → `新規ワークスペースとして復元` | `検証結果を確認`    |
| アーカイブと削除  | `アーカイブ済みを表示`                       | `ゴミ箱を表示`     |

DADS への反映。

* プライマリーボタンは 1 画面 1 個を原則にする。バックアップ作成画面では `JSONバックアップを作成` を主操作にする。
* 復元結果、失敗、機密情報警告はページ上部または対象セクション内のノティフィケーションバナーで表示する。
* ファイル選択 UI は、ラベル、任意/必須、サポートテキスト、具体的なエラー文を持たせる。
* `disabled` ボタンで止めるのではなく、未選択のまま押された場合に「JSON バックアップファイルを選択してください」と具体的に案内する。
* 重要な注意、特に「バックアップには機密情報が含まれる可能性があります」はアコーディオンの中に隠さない。
* 物理削除の確認にモーダルを使う場合、DADS でモーダルダイアログは未整備のため暫定案扱いにする。安全側に寄せるなら、MVP ではモーダルではなくページ内確認パネルを推奨する。
* 復元中の進行表示にプログレスインジケーターを使う場合も DADS 未整備項目のため暫定案扱いにする。MVP では「処理中」「完了」「失敗」をバナーと状態表示で足りる範囲に絞る。

# API・機能設計への影響

機能設計は、復元可能性と安全性を優先する。

MVP API / 機能案。

| 機能                                     | 役割                                 |
| -------------------------------------- | ---------------------------------- |
| `createJsonExport(scope, options)`     | JSON バックアップ生成。                     |
| `createMarkdownExport(scope, options)` | Markdown ZIP 生成。                   |
| `createCsvExport(scope, options)`      | CSV 生成。                            |
| `validateImport(file)`                 | JSON 構造、schemaVersion、参照整合性、件数を検証。 |
| `restoreToNewWorkspace(importJobId)`   | 新規ワークスペースに復元。                      |
| `createPreOperationBackup(reason)`     | 重要操作前バックアップ。                       |
| `archiveItem(id)`                      | アーカイブ。                             |
| `unarchiveItem(id)`                    | アーカイブ解除。                           |
| `softDeleteItem(id)`                   | 論理削除。                              |
| `restoreDeletedItem(id)`               | ゴミ箱から復元。                           |
| `purgeItem(id)`                        | 物理削除。                              |
| `runSchemaMigration()`                 | DB スキーマ更新。事前バックアップ必須。              |

復元仕様案。

1. ユーザーが JSON ファイルを選択する。
2. `validateImport` で以下を確認する。

   * JSON として読めるか。
   * `format` が `nextpatch.backup` か。
   * `schemaVersion` が対応範囲内か。
   * 必須エンティティが存在するか。
   * 参照切れがないか。
   * 件数が異常に多くないか。
   * 添付ファイルを含んでいないか。
3. 検証結果を画面に表示する。
4. ユーザーが `新規ワークスペースとして復元` を実行する。
5. DB トランザクション内で復元する。
6. 復元に失敗したらロールバックする。
7. 完了後、復元先ワークスペースへのリンクを表示する。

復元の制約。

* MVP では既存ワークスペースを上書きしない。
* MVP では既存ワークスペースへマージしない。
* 復元元 JSON の `deletedAt` は保持する。つまり、バックアップ内で削除済みだったものは復元先でも削除済み扱いにする。
* `archivedAt` も保持する。
* 物理削除済みデータは復元できない。
* DB dump の直接アップロード復元は MVP では扱わない。DB dump は管理者運用の範囲に分ける。

自動バックアップ仕様案。

| タイミング          |            MVP 対応 | 内容                                      |
| -------------- | ----------------: | --------------------------------------- |
| 手動 JSON バックアップ |                必須 | ユーザーがいつでも作成。                            |
| 復元前            |                必須 | 現在状態の自動バックアップを作成。                       |
| 物理削除前          |                必須 | 削除前の自動バックアップを作成。                        |
| DB マイグレーション前   |                必須 | DB レベルバックアップ、または JSON バックアップを作成。        |
| 日次定期バックアップ     | 運用手順として必須、UI は後回し | 本番・自ホスト環境では cron / platform backup で実施。 |
| 世代管理           |               最小限 | 直近数世代だけ保持。詳細 UI は後回し。                   |

DB dump を復元に使う場合の注意。

* PostgreSQL の dump 復元は、復元元に由来する SQL を実行するため、信頼できない dump を復元しない。PostgreSQL 公式 docs でも dump 復元時の任意コード実行リスクに注意が示されています。([PostgreSQL][7])
* アプリ UI から扱う復元は JSON に限定し、DB dump は運用者の管理手順に分離する。

# リスクと対策

| リスク                          | 対策                                                                                                                              |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| JSON バックアップが壊れていて復元できない      | `contentHash`、件数、検証処理、定期的な復元テストを用意する。OWASP も復元手順のテストを推奨している。([OWASP Cheat Sheet Series][5])                                     |
| バックアップに API キーや未公開情報が含まれる    | 生成前に機密情報警告を出す。MVP では GitHub 自動保存をしない。将来はシークレット検出・マスキング・暗号化を検討する。                                                                |
| GitHub にバックアップを置いて機密情報が履歴に残る | MVP では自動コミットしない。将来実装時も `.gitignore`、redacted export、履歴リスク警告を必須にする。機密情報が履歴に入った場合は、削除だけでなくシークレットの失効・ローテーションが必要。([GitHub Docs][4]) |
| Markdown / CSV だけ保存して復元できない  | 画面上で「復元用は JSON」と明記する。Markdown / CSV には `README` で用途を明記する。                                                                       |
| スキーマ変更で古いバックアップが読めなくなる       | `schemaVersion` とマイグレーション関数を用意する。最低でも現在版と直前版の JSON を読めるようにする。                                                                   |
| 物理削除後に復旧できない                 | 物理削除前に自動バックアップを作る。物理削除は強い確認を挟む。                                                                                                 |
| 自動バックアップがサーバー容量を圧迫する         | 保持世代数・保持期間を決める。MVP は直近数世代に限定する。                                                                                                 |
| 添付ファイル未対応で資料が欠落する            | MVP では URL を必須級に使いやすくする。外部資料は URL と説明文で管理する。                                                                                    |
| 復元で既存データを壊す                  | MVP では新規ワークスペース復元のみにする。                                                                                                         |
| 大容量エクスポートでタイムアウトする           | MVP ではサイズ上限とエラー表示を用意し、将来は非同期ジョブ化する。                                                                                             |
| バックアップ保存場所のアクセス権が弱い          | サーバー側に保存する自動バックアップは暗号化・アクセス制限を運用要件にする。OWASP はバックアップの暗号化と制限された安全な保管を推奨している。([OWASP Cheat Sheet Series][5])                       |

# 未決定事項

以下は統括側または別担当と決める必要がある。

1. **採用 DB**

   * SQLite か PostgreSQL かで DB レベルバックアップ手順が変わる。
   * SQLite なら Online Backup API などを前提にする。
   * PostgreSQL なら `pg_dump` / `pg_restore` とホスティング側スナップショットを組み合わせる。

2. **ホスティング形態**

   * ローカルアプリ、個人サーバー、SaaS 風ホスティング、自ホストのどれか。
   * 自動バックアップの保存先、暗号化、保持期間に影響する。

3. **ユーザー認証・権限**

   * 個人専用か、複数ユーザー・チーム利用か。
   * 複数ユーザーなら、誰がエクスポート・復元・物理削除できるかを決める必要がある。

4. **バックアップ暗号化**

   * MVP では必須にしない案だが、サーバー側保存の自動バックアップは暗号化前提にすべき。
   * ユーザーがダウンロードする JSON をパスワード暗号化するかは未決定。

5. **保持期間**

   * 自動バックアップを何世代残すか。
   * 例: 直近 7 日分 + マイグレーション前 3 世代など。
   * MVP では簡素にする。

6. **削除済みデータのバックアップ含有**

   * 推奨は JSON フルバックアップには論理削除済みも含める。
   * ただし、ユーザーが「完全に消したい」情報は物理削除し、以後のバックアップに含めない。

7. **Markdown のメタデータ表現**

   * YAML front matter を使うか、本文内メタデータ表にするか。
   * CommonMark 外の拡張は暫定案扱いにする。

8. **GitHub 連携時の redacted export**

   * 将来、リポジトリ内に `.nextpatch/` を置く場合、何を除外するか。
   * ChatGPT メモ、非公開アイデア、API キーらしき文字列、ローカルファイルパスは特に注意。

# 統括担当へ渡す要点

バックアップ・エクスポート・運用の結論は次のとおり。

* **MVP の正本バックアップは JSON。**
* **Markdown は人間が読む・Git 管理するための出力。**
* **CSV は表計算で棚卸しするための出力。**
* **復元は MVP では新規ワークスペース作成のみ。既存データ上書き・マージは後回し。**
* **アーカイブ、論理削除、物理削除を明確に分ける。**
* **添付ファイルは MVP では扱わず、URL のみ保存する。**
* **GitHub リポジトリ内への自動保存は MVP では行わない。**
* **DB マイグレーション、物理削除、復元の前には自動バックアップを作る。**
* **バックアップファイルは機密情報を含む前提で扱う。**
* **DADS 未整備のモーダル・プログレス UI は暫定案。可能ならページ内確認とノティフィケーションバナーで設計する。**

## バックアップ・エクスポートの受け入れ条件

MVP 完了判定として、以下を満たすこと。

1. **JSON バックアップ**

   * ワークスペース全体を JSON でエクスポートできる。
   * `format`、`schemaVersion`、`exportedAt`、`appVersion`、`scope`、`entities`、`integrity.counts` を含む。
   * アクティブ、アーカイブ済み、論理削除済みデータが含まれる。
   * 物理削除済みデータは含まれない。
   * 生成前に機密情報警告が表示される。

2. **JSON 復元**

   * JSON ファイルを選択して検証できる。
   * 不正 JSON、未対応 `schemaVersion`、必須フィールド欠落、参照切れを具体的なエラーとして表示できる。
   * 検証済み JSON を新規ワークスペースとして復元できる。
   * 復元後、元データに変更がない。
   * 復元された件数が検証結果と一致する。
   * 復元に失敗した場合、部分データが残らない。

3. **Markdown エクスポート**

   * リポジトリ単位で Markdown 一式を出力できる。
   * README、課題、バグ、実装予定、将来構想、採用技術、参考リンク、ChatGPT メモが読みやすく整理される。
   * Markdown は復元用ではないことが README に明記される。
   * ファイル名に内部 ID または安定キーが含まれる。

4. **CSV エクスポート**

   * `items.csv` を出力できる。
   * ヘッダー行がある。
   * カンマ、改行、ダブルクォートを含む値が正しく引用符エスケープされる。
   * CSV は復元用ではないことが画面上で分かる。

5. **アーカイブ**

   * アイテムをアーカイブできる。
   * アーカイブ済みは通常一覧から隠れる。
   * アーカイブ済みを表示する導線がある。
   * アーカイブ解除できる。
   * JSON バックアップにはアーカイブ状態が保持される。

6. **削除**

   * 論理削除できる。
   * 論理削除済みを復元できる。
   * 物理削除は明示的な確認後にのみ実行される。
   * 物理削除前に自動バックアップが作られる、または作成できない場合は削除が中止される。
   * 物理削除済みデータは以後の JSON バックアップに含まれない。

7. **マイグレーション**

   * DB スキーマ変更前にバックアップを作る手順がある。
   * エクスポート `schemaVersion` と DB migration version を混同しない。
   * 少なくとも現行 `schemaVersion` の JSON を検証・復元できる。
   * 将来の `schemaVersion` 変更に備え、旧バージョン変換処理を追加できる構造になっている。

8. **UI**

   * `設定 > データ管理` から JSON バックアップ、Markdown 出力、CSV 出力、復元に到達できる。
   * 重要な警告がアコーディオン内に隠れていない。
   * 主操作ボタンが画面内で競合しない。
   * ファイル選択にはラベル、サポートテキスト、具体的エラーがある。
   * 成功・失敗・警告はノティフィケーションバナーまたは対象セクション内で明確に表示される。
