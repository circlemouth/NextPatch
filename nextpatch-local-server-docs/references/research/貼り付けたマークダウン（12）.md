# 担当領域

参考サービス・競合調査。

NextPatch が、GitHub リポジトリ単位で「課題・進捗・バグ・実装予定・将来構想・技術メモ・参考サービス・ChatGPT 壁打ちメモ」を整理し、個人開発者が「今やるべきこと」をすぐ把握できるアプリになるために、既存サービスから取り入れるべき設計、避けるべき複雑さ、MVP での取捨選択を整理する。

# 前提理解

NextPatch は、Jira や Linear のような汎用プロジェクト管理ツールをそのまま再現するものではなく、「GitHub リポジトリごとの開発判断と次アクションを集約する個人開発者向けの開発管理アプリ」と位置づけるべきである。

この回答では、会話内で提示された NextPatch の概要を要件前提として扱う。現時点でこのスレッド上から確認できた前提資料は `dads_app_ui_design_rules_20260411.md` であり、NextPatch 要件定義書および「開発着手前に決定すべき内容の検討メモ」の本文は確認できなかった。そのため、NextPatch 固有要件については、ユーザー文面で明示された範囲を前提にしている。

UI については DADS 参照ルールを優先する。特に、重要情報を隠さないこと、サービス内で一貫性を保つこと、フォームではラベル・サポートテキスト・具体的なエラー表示を基本にすること、ボタンは優先度と並び順を一貫させること、DADS で未整備のタブ、モーダル、テーブルコントロール、グローバルメニュー等は独自仕様を断定せず暫定案として扱うことを前提にする。

Web 情報の確認日は、すべて **2026年4月20日**。公式情報を優先して参照し、Height については公式サイト取得に失敗したため、二次情報を補助的に扱う。

# 決定すべき論点

第一に、NextPatch を「プロジェクト管理ツール」として設計するか、「リポジトリ単位の開発判断ダッシュボード」として設計するかを決める必要がある。GitHub Projects はテーブル、カンバン、ロードマップで issue、PR、メモを扱える柔軟な作業管理機能を持つため、NextPatch が同じ方向へ進むと差別化しにくい。GitHub Projects は GitHub データと同期する柔軟なコレクションとして設計されている。([GitHub Docs][1])

第二に、GitHub Issues を主データにするか、NextPatch 独自の「開発メモ／判断／次アクション」を主データにするかを決める必要がある。GitHub Issues はバグ、機能、アイデア、作業分解、ラベル、マイルストーン、依存関係、PR 連携まで対応しており、既に開発作業の一次情報として強い。([GitHub Docs][2])

第三に、スマホでのクイック登録を、通常の詳細フォームにするか、まず Inbox に放り込んで後から整理する方式にするかを決める必要がある。Todoist の Quick Add は日付、ラベル、リマインダー等を含むタスクを素早く追加する設計で、Trello もモバイルでホーム画面長押し、ウィジェット、アプリ内プラスボタンからカード追加できる。([Todoist][3])

第四に、優先度表示を「手入力の P0/P1/P2」中心にするか、「今やるべきこと」をアプリ側で集約表示するかを決める必要がある。Linear の My Issues は重要な issue を Focus グループで整理し、Urgent、SLA、ブロッカー、現在サイクル等を順に出す設計を持つ。NextPatch ではこれを個人開発向けに簡略化し、「今やる」「詰まり」「次に見る」「いつか」に落とすのが有効である。([Linear][4])

第五に、AI を中核機能にするか、整理補助に留めるかを決める必要がある。Height は自律的プロジェクト管理として AI バグトリアージ、バックログ整理、仕様更新を掲げていたが、2025年にサービス終了予定と報じられており、AI 自律化をプロダクトの根幹に置きすぎるリスクを示す事例として見るべきである。([Shortcut][5])

# 推奨案

NextPatch は **「リポジトリ別 Repo Action Hub」** として設計するのが最もよい。

つまり、GitHub Projects、Linear、Jira のような「チーム向け作業管理」ではなく、個人開発者がリポジトリごとに「今の状態」「次にやること」「詰まっていること」「後で見るアイデア」「技術判断」「参考サービス」「ChatGPT メモ」を短時間で把握するアプリにする。

理由は三つある。

一つ目は、GitHub Issues と GitHub Projects が既に issue/PR/プロジェクト管理の強い一次情報を持っているためである。NextPatch が同じ issue tracker になると、GitHub 側との二重管理が発生する。NextPatch は GitHub を置き換えるのではなく、GitHub に散らばる issue、PR、README、メモ、壁打ち結果を「リポジトリ単位で次アクション化する」位置を取るべきである。

二つ目は、個人開発者に必要なのは高度なワークフローよりも「迷わない画面」である。Jira や ClickUp のように機能を増やすほど、MVP の価値である「今やるべきことをすぐ把握」が弱くなる。

三つ目は、スマホでのクイック登録が差別化に直結するためである。Todoist、Trello、Linear は素早い登録や Inbox を重視している。NextPatch でも、詳細分類を入力時に強制せず、まず repo とタイトルだけで Inbox に保存し、後から整理できる構造にすべきである。

推奨する基本設計は次の通り。

| 項目           | 推奨案                                                            |
| ------------ | -------------------------------------------------------------- |
| 最上位単位        | Repository                                                     |
| 基本オブジェクト     | PatchItem                                                      |
| PatchItem 種別 | Task、Bug、Idea、Memo、TechDecision、Reference、Roadmap              |
| 標準状態         | Inbox、Next、Doing、Blocked、Waiting、Done、Later、Archived           |
| 優先度          | P0、P1、P2、P3 と `nextAction=true/false`                          |
| 主要ビュー        | Repo Dashboard、Next Queue、Inbox、Board、Roadmap、Notes/References |
| GitHub 連携    | Issue/PR/Project URL のリンク、後に読み取り同期                             |
| AI           | 自動実行ではなく、分類・要約・候補提示に限定                                         |
| スマホ登録        | 1画面・最小項目・保存後に Inbox へ                                          |

複数案が残る場合の判断条件は明確である。

GitHub 連携を重視するなら、GitHub Issues/Projects のミラーではなく、**GitHub item に NextPatch のメモ・判断・次アクションを付ける設計**を選ぶべきである。ローカルメモを重視するなら、Obsidian/Anytype 型に寄せて、Markdown エクスポートや repo 内 docs 連携を優先する。ただし MVP では、どちらの場合も「Repo Dashboard」と「Quick Capture」を優先する。

# MVPに含めるべき内容

MVP には次を含めるべきである。

| 機能                        | 理由                                                                       |
| ------------------------- | ------------------------------------------------------------------------ |
| リポジトリ登録                   | NextPatch の全情報を repo 単位に集約するため。                                          |
| Repo Dashboard            | 「今やる」「詰まり」「最近のメモ」「未整理 Inbox」「次リリース候補」を一画面で見るため。                          |
| Quick Capture             | スマホで思いついたバグ、アイデア、ChatGPT メモ、参考URLを即保存するため。                               |
| Inbox / Triage            | 入力時に分類を強制せず、後で整理できるようにするため。Linear の Triage に近いが、個人向けに簡略化する。([Linear][6]) |
| PatchItem 種別              | Task、Bug、Idea、Memo、TechDecision、Reference、Roadmap の最低限。                  |
| Status                    | Inbox、Next、Doing、Blocked、Done、Later。Waiting と Archived はあってもよい。          |
| Priority                  | P0〜P3。Todoist のように軽量に入力できることを重視する。([Todoist][20])                        |
| GitHub URL 紐づけ            | Issue、PR、Project、commit、discussion へのリンクを保持する。                           |
| タグ                        | 技術、領域、機能、参考サービスなど横断整理に必要。                                                |
| 簡易フィルター                   | repo、種別、状態、優先度、タグ、未整理、blocked。                                           |
| Now / Next / Later ロードマップ | Productboard 的な本格ロードマップではなく、個人開発向けの軽量表現。                                 |
| メモ本文                      | ChatGPT 壁打ち結果、技術判断、参考サービス調査を残すため。                                        |
| 出典リンク                     | AI 要約や外部情報の根拠を失わないため。                                                    |

MVP の画面到達は、主要目的を 3 操作以内にする。

例として、スマホでバグを登録する場合は「新規」→「タイトル入力」→「保存」。現在の repo 文脈がある場合、repo 選択は初期値で補う。詳細分類、優先度、GitHub URL、タグは保存後に任意で追加する。

# MVPでは後回しにする内容

MVP では次を後回しにする。

| 後回し項目                       | 理由                                                         |
| --------------------------- | ---------------------------------------------------------- |
| GitHub Projects の完全同期       | GitHub 側の柔軟な field/view/workflow を再現すると過剰。                 |
| GitHub Issues の作成・編集・コメント同期 | MVP では URL 紐づけと読み取り候補に留める方が安全。                             |
| Jira 風のカスタム workflow        | 個人開発では管理負荷が高い。                                             |
| JQL 的な高度検索構文                | 通常フィルターで足りる。                                               |
| Sprint / Cycle / Velocity   | 個人開発では必須ではない。Linear Cycles は参考になるが MVP では不要。([Linear][21]) |
| 複数ユーザー、権限、担当者、チーム           | NextPatch の初期価値は個人開発向け。                                    |
| ポートフォリオ・組織横断ダッシュボード         | Asana/ClickUp 領域であり MVP では過剰。                              |
| Sentry 連携による自動バグ生成          | 将来有望だが、まず手入力と URL リンクで足りる。                                 |
| 顧客フィードバックポータル               | Productboard 的機能であり、個人開発 MVP には重い。                         |
| AI による自動分類・自動実行             | 提案までに留め、ユーザー確認を必須にする。                                      |
| グラフビュー中心 UI                 | Obsidian 的な関連可視化は将来拡張でよい。                                  |
| P2P / E2E / セルフホスト同期        | Anytype 的な設計は魅力的だが MVP には重い。                               |
| 高度なテーブルコントロール               | DADS で未整備のため、初期は暫定案として簡易フィルターに留める。                         |
| タブ中心ナビゲーション                 | DADS で未整備のため、MVP では通常リンク・セクション・フィルターで代替する。                 |

# データ設計への影響

競合調査から見ると、NextPatch のデータ設計は「タスク管理」よりも「リポジトリ文脈付きメモ・課題・判断」のモデルにするべきである。

推奨する主要エンティティは次の通り。

| エンティティ        | 主な属性                                                                                      | 競合からの示唆                                                                       |
| ------------- | ----------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| Repository    | id、name、owner、githubUrl、description、status、lastReviewedAt                                 | GitHub を中心に置く。                                                                |
| PatchItem     | id、repositoryId、type、title、body、status、priority、nextAction、dueDate、sourceType、createdFrom | GitHub Issues、Trello card、Todoist task、Notion DB の中間。                         |
| ExternalLink  | itemId、kind、url、externalId、syncState                                                      | GitHub Issue/PR、参考記事、サービスURL、ChatGPTログへの接続。                                   |
| Tag           | name、category、color                                                                       | GitHub label、Todoist label、Notion property の簡略版。                              |
| Relation      | fromItemId、toItemId / externalUrl、relationType                                            | blocks、blocked_by、duplicates、related、implements。GitHub/Jira/Linear の依存関係を簡略化。 |
| RoadmapBucket | repositoryId、name、order                                                                   | Now、Next、Later、Shipped。                                                       |
| CaptureInbox  | rawText、defaultRepositoryId、suggestedType、suggestedTags、processedAt                       | Todoist Quick Add、Linear Triage、Mem 的な投入口。                                    |
| TechDecision  | itemId、decision、reason、alternatives、decidedAt                                             | 開発判断を残すため。PatchItem の type として実装してもよい。                                        |
| Reference     | itemId、url、serviceName、summary、takeaway                                                   | 参考サービス・技術記事・公式ドキュメント整理用。                                                      |

重要なのは、`PatchItem` を万能カードにしつつ、`type` と `relation` で構造化することである。Trello のように何でもカードにすると分類が弱くなり、Jira のように細かく型を分けすぎると入力が重くなる。その中間を取る。

また、AI 要約や自動分類を入れる場合は、`aiSuggestedType`、`aiSuggestedPriority`、`aiSummary`、`aiConfidence`、`aiSourceItemId` のように「AI が提案した値」と「ユーザーが確定した値」を分けるべきである。AI が誤分類しても元情報へ戻れるようにするためである。

# 画面設計への影響

画面設計では、競合の豊富な機能を並べるのではなく、NextPatch 固有の「リポジトリごとの今やること整理」を最上位に置く。

推奨する主要画面は次の通り。

| 画面                 | 目的                                    | 参考サービス                                             |
| ------------------ | ------------------------------------- | -------------------------------------------------- |
| Repository List    | 開発中 repo を一覧し、状態をすぐ見る                 | GitHub、Linear、Asana                                |
| Repo Dashboard     | その repo の「今やる」「詰まり」「未整理」「最近のメモ」を一画面表示 | Linear My Issues、Asana Dashboard、ClickUp Dashboard |
| Quick Capture      | スマホでメモ・バグ・アイデアを即保存                    | Todoist Quick Add、Trello mobile add、Mem            |
| Inbox / Triage     | 未整理メモを分類・優先付け                         | Linear Triage、Todoist Inbox                        |
| Board              | 状態別に PatchItem を移動                    | Trello、GitHub Projects、Jira                        |
| Roadmap            | Now / Next / Later / Shipped          | Productboard、GitHub Projects roadmap               |
| Item Detail        | 本文、GitHubリンク、関連、判断理由、次アクション           | GitHub Issue、Notion page、Linear issue              |
| Notes / References | 技術メモ、参考サービス、ChatGPT 壁打ちメモを検索          | Notion、Obsidian、Mem                                |

UI の基本判断は DADS を優先する。

重要情報である「今やる」「詰まり」「未整理 Inbox 数」「次リリース候補」はアコーディオンに隠さず、Repo Dashboard の初期表示に置く。DADS は重要情報を隠さないことを重視している。

スマホでは、ボトムナビゲーションを安易に使わない。DADS ではボトムナビゲーションは非推奨とされているため、MVP ではヘッダー、メニュー、明確な「新規」ボタンを基本にする。画面下固定のクイック追加ボタンを使う場合は、DADS に明確な定義がないため **暫定案**として扱う。

フォームは、Quick Capture でもラベルを省略しない。プレースホルダーだけで説明せず、ラベルとサポートテキストを使う。入力項目は「タイトル」「本文」「repo」「種別」程度に絞り、未入力で保存ボタンを disabled にするのではなく、押された後に具体的なエラーを出す設計にする。

ボタンは 1 画面 1 プライマリーを原則にする。Quick Capture 画面では「保存」をプライマリー、詳細追加はセカンダリーまたは保存後の導線にする。

# API・機能設計への影響

API・機能設計では、GitHub との境界を明確にする必要がある。

MVP では、NextPatch は GitHub の完全同期システムではなく、GitHub の issue、PR、Project、commit、discussion へのリンクを保持し、必要に応じて読み取り同期に進むのがよい。GitHub Issues は Web UI、GitHub CLI、GraphQL、REST API、GitHub Mobile など多様な作成経路を持つため、NextPatch が初期から作成経路まで抱える必要はない。([GitHub Docs][2])

推奨する機能 API の分け方は次の通り。

| 機能領域          | MVP                                      | 将来                                                   |
| ------------- | ---------------------------------------- | ---------------------------------------------------- |
| Repository    | 登録、編集、アーカイブ、GitHub URL 保持                | GitHub OAuth、repo 自動取得                               |
| PatchItem     | CRUD、status 更新、priority 更新、nextAction 切替 | GitHub Issue 作成、双方向同期                                |
| Quick Capture | 最小入力で Inbox 作成                           | Share Extension、音声入力、画像添付                            |
| Triage        | 種別・repo・priority・status の確定              | AI 分類候補、重複候補                                         |
| GitHub 連携     | URL/ID 保存、手動リンク                          | issue/PR 状態の読み取り、ラベル同期                               |
| Dashboard     | repo ごとの集計                               | AI 週次サマリー、進捗レポート                                     |
| Search/Filter | type、status、priority、tag、repo            | 自然言語検索、ベクトル検索                                        |
| AI            | なし、または手動要約                               | ChatGPT ログ取り込み、TODO抽出、技術判断抽出                         |
| 外部連携          | なし                                       | Sentry、Notion、Todoist、Slack、Obsidian Markdown export |

機能設計上の重要点は、AI や同期処理を「ユーザーの意思決定の補助」に留めることである。Mem や Notion AI はメモの整理・要約に有用だが、NextPatch では「このメモから Task/Bug/TechDecision を作りますか？」という確認ステップを入れるべきである。([Mem][19])

# リスクと対策

| リスク                   | 内容                                   | 対策                                                  |
| --------------------- | ------------------------------------ | --------------------------------------------------- |
| GitHub Projects と競合する | Projects の簡易版に見えると価値が薄い。             | GitHub の代替ではなく「repo ごとの判断・次アクション整理」に集中する。           |
| Jira/ClickUp 化する      | 機能を増やしすぎて個人開発者が使い始めにくい。              | MVP では item type、status、priority、tag を固定に近い形で提供する。  |
| Quick Capture がゴミ箱化する | 何でも Inbox に入り、整理されない。                | Dashboard に未整理数を出し、Triage を主要導線にする。                 |
| AI 誤分類                | ChatGPT メモやバグを誤って分類する。               | AI は提案のみ。確定値と提案値を分け、元メモリンクを残す。                      |
| GitHub との二重管理         | issue 状態が NextPatch と GitHub でずれる。   | MVP はリンク中心。双方向同期は後回し。syncState を持つ。                 |
| UI が複雑化する             | Board、Roadmap、Notes、Dashboard が乱立する。 | 初期表示は Repo Dashboard、Quick Capture、Inbox、Board に絞る。 |
| DADS 未整備コンポーネントの独自拡張  | タブ、テーブルコントロール、モーダル等の仕様が曖昧になる。        | MVP では通常リンク、セクション、シンプルなリストで代替し、未整備部品は暫定案として明記する。    |
| Height 的な AI 中心依存     | 自律 PM を売りにしすぎると、精度・信頼・運用負荷が重くなる。     | コア価値は手動でも成立させ、AI は補助機能にする。                          |
| 個人開発者にチーム機能を押し付ける     | 担当者、承認、権限、SLA は使われない可能性が高い。          | single-user first。共有・チームは将来拡張。                      |

# 統括担当へ渡す要点

**NextPatch が取り入れるべき設計パターン**

| パターン                        | 由来                        | NextPatch での形                              |
| --------------------------- | ------------------------- | ------------------------------------------ |
| Repo-first dashboard        | GitHub / Linear           | リポジトリを開いたら「今やる」「詰まり」「未整理」「最近の判断」が見える。      |
| Inbox / Triage              | Linear / Todoist / Trello | スマホ・PC から雑に登録し、後で分類する。                     |
| Quick Capture               | Todoist / Trello / Mem    | タイトルだけでも保存できる。repo は現在文脈から初期設定。            |
| Focus / Next Queue          | Linear My Issues          | 優先度、blocked、status、最近更新から「今見るべき item」を出す。  |
| Lightweight Board           | Trello / GitHub Projects  | Inbox、Next、Doing、Blocked、Done、Later の状態移動。 |
| Now / Next / Later Roadmap  | Productboard 系            | 個人開発向けに日付より意思決定を重視する。                      |
| Notes beside tasks          | Notion / Obsidian         | 技術判断、参考URL、ChatGPT メモを item に紐づける。         |
| Source-linked AI suggestion | Mem / Notion AI           | 要約・分類候補には必ず元メモへのリンクを持たせる。                  |

**NextPatch が避けるべき設計パターン**

| 避けるもの                    | 理由                                |
| ------------------------ | --------------------------------- |
| GitHub Projects の再実装     | 既に強い公式機能がある。NextPatch の差別化にならない。  |
| Jira 風の workflow builder | 個人開発者には設定負荷が高い。                   |
| ClickUp 的 everything app | 機能過多で「今やること」が埋もれる。                |
| Notion 的 blank canvas    | 使い始めに設計が必要になり、NextPatch の即効性が落ちる。 |
| AI 自律実行                  | 誤分類・誤更新時の信頼低下が大きい。                |
| ボトムナビ前提のスマホ UI           | DADS では非推奨。使う場合は暫定案として慎重に扱う。      |
| 重要情報をアコーディオンに隠す UI       | DADS 方針と NextPatch の目的に反する。       |

**MVP時点で真似しなくてよい機能**

GitHub Projects のカスタムフィールド、チャート、複数ビュー、テンプレート。Jira の JQL、Advanced Roadmaps、複雑な workflow、管理者設定。Linear の Cycles、SLA、チーム Triage 担当ローテーション。Asana の Portfolios、Workload、承認、フォーム運用。ClickUp の Docs/Chat/Goals まで含む all-in-one 化。Productboard の顧客ポータルとフィードバックスコアリング。Sentry のランタイム監視。Obsidian のグラフ中心 UI。Anytype の P2P / E2E / セルフホスト同期。Height 的な完全自律 PM。

**将来拡張として有望な機能**

GitHub Issues / PR の読み取り同期、GitHub issue 作成、Sentry からのバグ候補取り込み、ChatGPT 壁打ちログの貼り付け要約、AI による Task/Bug/TechDecision 抽出、Markdown export、Obsidian vault 連携、Notion import、Todoist 連携、週次レビュー、リリースノート生成、重複バグ候補、関連 item 推薦、個人開発ポートフォリオ全体の「止まっている repo」表示。

統括担当への最終推奨は、**MVP の競合比較軸を「機能数」ではなく「リポジトリごとの次アクション到達速度」に置くこと**である。GitHub、Linear、Jira、Notion、Todoist の良い点を取り入れるが、NextPatch の初期価値は「repo を開いたら、今やることが 10 秒で分かる」に絞るべきである。
