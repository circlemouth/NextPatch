# 担当領域

認証・セキュリティ・プライバシー。

NextPatch に保存される情報は、未公開アプリの構想、リポジトリ URL、バグ内容、技術選定、将来ロードマップ、ChatGPT との壁打ちメモなどであり、通常のタスク管理情報よりも漏えい時の事業影響が大きい前提で扱う。

# 前提理解

この会話で本文を確認できた前提資料は `dads_app_ui_design_rules_20260411.md` です。NextPatch 要件定義書と「開発着手前に決定すべき内容の検討メモ」は、現時点では本文を直接確認できないため、依頼文に含まれる NextPatch の説明を前提に設計判断します。

DADS については、アクセシビリティ、一貫性、重要情報を隠さないこと、フォームではラベル・サポートテキスト・具体的なエラーテキストを重視すること、ボタンの無効化を安易に使わないこと、ノティフィケーションバナーを重要情報の提示に使うことを UI 設計の優先ルールとして扱う。モーダルダイアログ、タブ、グローバルメニューなどは DADS 側で詳細ガイドライン未整備とされているため、使う場合は「暫定案」として扱う。

Web 参照元は、OWASP Cheat Sheet Series、GitHub Docs、OpenAI Platform Docs、FIDO Alliance、W3C WebAuthn 仕様を確認した。確認日は 2026-04-20。OWASP は CSRF ではフレームワークの組み込み保護を使い、なければ状態変更リクエストに CSRF トークンを付けることを推奨している。セッション Cookie では `Secure` と `HttpOnly` の重要性が示されているが、`HttpOnly` だけでは CSRF を防げない点も明記されている。([OWASP Cheat Sheet Series][1])

GitHub 連携については、GitHub Docs が GitHub Apps を OAuth Apps より一般に推奨しており、理由として fine-grained permissions、リポジトリ単位の制御、短命トークンを挙げている。OAuth Apps を使う場合も、scope は必要最小限にし、`repo` scope は公開・非公開リポジトリへの広いアクセスを含むため MVP では避けるべきである。([GitHub Docs][2])

OpenAI API については、API に送信されたデータは明示的に opt-in しない限りモデル改善・訓練に使われない一方、 abuse monitoring logs にはプロンプトや応答などの customer content が含まれる可能性があり、通常は最大 30 日保持される。Zero Data Retention や Modified Abuse Monitoring は承認が必要な制御として扱われている。([OpenAI開発者][3])

# 決定すべき論点

最重要の設計判断は、「NextPatch を最初から private-by-default な認証必須アプリとして作るか」です。結論として、MVP から認証は必須にする。匿名利用、ローカルだけの仮運用、または後付け認証は推奨しない。理由は、MVP の初期データモデルに `userId` がないまま実装すると、後から共有・チーム・GitHub 連携を入れる際に、アクセス制御の抜けやデータ移行リスクが大きくなるため。

決めるべき論点は次の通り。

| 論点                     | MVPでの判断                               |
| ---------------------- | ------------------------------------- |
| 認証を MVP から入れるか         | 入れる。必須。                               |
| GitHub OAuth を使うか      | MVP の主認証にはしない。入れる場合は副ログインか将来連携の入口に限定。 |
| メールログインを使うか            | 主認証として採用。メールリンクまたはワンタイムコード方式を推奨。      |
| パスキーを使うか               | MVP では後回し。ただし将来追加できるデータ構造にしておく。       |
| `userId` によるデータ分離      | MVP から全テーブル・全 API で必須。                |
| 複数ユーザー対応               | UI 上のチーム共有は後回し。ただし DB は複数ユーザー前提。      |
| 共有リンク                  | MVP では作らない。                           |
| 公開ページ                  | MVP では作らない。                           |
| GitHub トークン保存          | MVP では保存しない。                          |
| ChatGPT API への本文送信     | MVP では原則後回し。入れるなら明示操作・最小送信・ログ非保存が必須。  |
| エラーログに本文を含めるか          | 含めない。                                 |
| バックアップ                 | 暗号化・短期有効・所有者限定。                       |
| 削除・アーカイブ・復元            | `ownerUserId` チェック、監査イベント、復元可能期間を設計。  |
| CSRF / XSS / 権限 / 入力検証 | MVP から必須。                             |

# 選択肢比較

## 認証方式の比較

| 方式                       | 長所                                                                                                         | 短所・リスク                                             | MVP判断                 |
| ------------------------ | ---------------------------------------------------------------------------------------------------------- | -------------------------------------------------- | --------------------- |
| 認証なし                     | 実装が最速                                                                                                      | 機密メモ、未公開構想、リポジトリ情報を保護できない。後付け認証でデータ分離ミスが起きやすい。     | 不採用                   |
| メールリンク / ワンタイムコード        | パスワードを NextPatch 側で持たない。GitHub アカウントに依存しない。個人開発者にも導入しやすい。                                                  | メール配送、リンク盗難、メールボックス乗っ取りへの依存。                       | 採用                    |
| メール + パスワード              | 利用者に馴染みがある。                                                                                                | パスワード保管、リセット、総当たり対策、漏えい時対応が必要。                     | 原則不採用。使うなら認証基盤に任せる    |
| GitHub OAuth             | 開発者向けアプリと相性がよい。GitHub identity と紐付けやすい。                                                                    | GitHub 依存、scope 過剰要求、トークン保存リスク、GitHub を使わない利用者の排除。 | MVP 主認証にはしない。副ログインなら可 |
| パスキー                     | フィッシング耐性が高く、サーバーに盗まれるパスワードを置かない。FIDO はパスキーをフィッシングや credential stuffing に強い方式として説明している。([FIDO Alliance][4]) | 実装・アカウント復旧・端末移行 UX の検討が必要。                         | MVP 後に追加              |
| SSO / SAML / OIDC 組織ログイン | 将来のチーム・企業利用に向く。                                                                                            | MVP の個人開発者向けには重い。                                  | 後回し                   |

メールリンク方式を採る場合、メールに含めるトークンは十分な長さ、暗号学的に安全な乱数、単回使用、期限付きにする。OWASP はメール所有確認用リンクのトークンについて、少なくとも 32 文字、安全な乱数、単回使用、期限付きとする考え方を示している。([OWASP Cheat Sheet Series][5])

## GitHub OAuth / GitHub App / トークン保存の比較

| 方式                              | できること                | セキュリティ評価                                                             | MVP判断  |
| ------------------------------- | -------------------- | -------------------------------------------------------------------- | ------ |
| GitHub OAuth をログインだけに使う         | GitHub アカウントで本人確認    | `read:user` / `user:email` 程度なら比較的限定的。ただし token を保存しない設計が必要。         | 条件付きで可 |
| GitHub OAuth で `repo` scope を要求 | private repo の情報取得など | `repo` は公開・非公開リポジトリへの広い read/write を含むため、MVP では過剰。([GitHub Docs][6]) | 不採用    |
| GitHub App                      | リポジトリ単位・権限単位で制御しやすい  | GitHub 公式も一般に OAuth Apps より GitHub Apps を推奨。([GitHub Docs][2])       | 将来採用   |
| ユーザーに PAT を貼らせる                 | 早く実装できる              | ユーザー体験が悪く、漏えいリスクが高い。組織ポリシーにも左右される。                                   | 不採用    |

GitHub OAuth を使う場合、`state` は CSRF 対策として使い、PKCE も使う。GitHub Docs は OAuth web application flow で `state` と PKCE を strongly recommended として説明している。([GitHub Docs][7])

## 共有・公開機能の比較

| 機能         | 長所                              | リスク                                              | MVP判断 |
| ---------- | ------------------------------- | ------------------------------------------------ | ----- |
| 共有なし       | 最も安全。private-by-default を守りやすい。 | 他者共有のニーズには応えられない。                                | 採用    |
| 期限付き共有リンク  | レビュー依頼に便利。                      | URL 自体が bearer token になる。転送・スクショ・検索インデックス誤設定リスク。 | 後回し   |
| 公開ページ      | ポートフォリオやロードマップ公開に便利。            | 未公開構想や脆弱性情報の誤公開リスクが高い。                           | 後回し   |
| チームワークスペース | 複数人開発に必要。                       | RBAC、招待、退会、権限棚卸し、監査が必要。                          | 後回し   |

# 推奨案

推奨案は次の通り。

MVP は **メールリンクまたはワンタイムコードによるメールログインを主認証** とし、サーバー管理セッションを `Secure`、`HttpOnly`、`SameSite=Lax` 以上の Cookie で保持する。NextPatch 本体はパスワードを保存しない。セッション Cookie だけに頼らず、状態変更 API には CSRF 対策を入れる。OWASP は状態変更リクエストに CSRF トークンを追加・検証すること、また Cookie の `Secure` / `HttpOnly` 属性の重要性を示している。([OWASP Cheat Sheet Series][1])

GitHub OAuth は MVP の主認証にはしない。MVP で「GitHubで続ける」を入れる場合は、ログイン補助に限定し、`read:user` と `user:email` 相当の最小 scope だけを明示指定し、取得した access token はプロフィール取得後に保存しない。GitHub Docs では `read:user` は profile data の読み取り、`user:email` は email addresses の読み取りとして説明されている。([GitHub Docs][6])

GitHub の private repository 連携、Issue 連携、PR 連携、リポジトリ一覧取得などを将来入れる場合は、OAuth App ではなく GitHub App を第一候補にする。GitHub App の installation token は短命かつ権限・対象リポジトリを細かく制御できるため、NextPatch の機密性と相性がよい。([GitHub Docs][2])

パスキーは MVP では実装しない。ただし将来追加を見越して、`users` と `auth_accounts` を分け、後から WebAuthn credential を追加できる設計にする。W3C WebAuthn は、Relying Party の origin にスコープされた public key credential を使う強い認証 API として定義されている。([W3C][8])

共有リンク、公開ページ、チーム機能、GitHub token 保存、ChatGPT API 自動連携は MVP では後回しにする。理由は、NextPatch の初期価値は「自分の開発リポジトリごとに今やるべきことを安全に整理すること」であり、共有・外部連携は漏えい面を大きく広げるため。

# MVPに含めるべき内容

## 1. MVPで必須のセキュリティ要件

MVP に必ず含めるべき内容は次の通り。

| 項目        | 必須要件                                                                                |
| --------- | ----------------------------------------------------------------------------------- |
| 認証        | 未ログイン状態ではアプリ本体に入れない。ログイン前に見えるのはランディング、利用規約、プライバシー説明、ログイン画面まで。                       |
| セッション     | Cookie は `Secure`、`HttpOnly`、`SameSite=Lax` 以上。本番は HTTPS 前提。                        |
| CSRF      | Cookie セッションを使うため、POST / PATCH / DELETE など状態変更には CSRF 対策を入れる。                       |
| userId 分離 | すべてのプロジェクト、リポジトリ、メモ、課題、バグ、ロードマップ項目に `ownerUserId` または `workspaceId` を持たせる。          |
| 権限チェック    | すべての API で「ログインユーザーが対象リソースの所有者か」をサーバー側で確認する。                                        |
| 入力検証      | すべての入力をサーバー側で検証。URL、ステータス、種別、日付、文字数、配列長を制限する。                                       |
| XSS 対策    | ユーザー入力は出力時にエスケープ。Markdown を許可する場合も raw HTML は無効化またはサニタイズ。                           |
| ログ        | 本文、ChatGPT メモ、GitHub token、API key、セッション ID、DB 接続文字列、個人情報をログに出さない。                  |
| エラー収集     | エラー本文ではなく、エラーコード、resource type、resource id、user id、request id、発生箇所、スタックトレースを最小限で収集。 |
| 監査イベント    | ログイン、ログアウト、削除、復元、エクスポート、外部連携追加・解除を記録。ただし本文は記録しない。                                   |
| バックアップ    | バックアップファイルは暗号化、アクセス制限、保持期限、復元時の userId 検証を必須にする。                                    |
| 削除・復元     | `deletedAt` によるソフト削除、所有者限定の復元、一定期間後の完全削除方針を持つ。                                      |
| レート制限     | ログインメール送信、ワンタイムコード検証、OAuth callback、エクスポートにレート制限を入れる。                               |
| シークレット管理  | API key、OAuth client secret、DB credential は環境変数または secret manager で管理し、リポジトリに置かない。  |

権限については「deny by default」と「毎リクエストでの権限検証」を基本にする。OWASP は、アクセス制御では明示的に許可されない限り拒否し、すべてのリクエストで権限を検証することを推奨している。([OWASP Cheat Sheet Series][9])

XSS は単一の対策で防げるものではないため、入力検証、出力エスケープ、Markdown サニタイズ、CSP、Cookie の `HttpOnly` を組み合わせる。OWASP XSS Prevention Cheat Sheet も、XSS 対策には複数の防御手法の組み合わせが必要だとしている。([OWASP Cheat Sheet Series][10])

入力検証は UX 用のクライアント検証だけでなく、必ずサーバー側で行う。OWASP は、入力検証は構文的・意味的な両面で行い、クライアント側 JavaScript の検証は回避され得るため、サーバー側検証が必要だとしている。([OWASP Cheat Sheet Series][5])

# MVPでは後回しにする内容

## 2. MVPでは後回しにできる要件

次は MVP では入れない。

| 後回し項目                          | 理由                                   |
| ------------------------------ | ------------------------------------ |
| パスキー本実装                        | 高セキュリティだが、アカウント復旧と端末移行 UX の設計が必要。    |
| メール + パスワード                    | パスワード保管リスクを避ける。                      |
| GitHub private repo 自動取得       | token 保存、権限 scope、組織ポリシーの考慮が必要。      |
| GitHub token 永続保存              | 漏えい時の被害が大きい。GitHub App 設計後に判断。       |
| チーム・組織ワークスペース                  | RBAC、招待、退会、権限変更、監査 UI が必要。           |
| 共有リンク                          | bearer URL の漏えいリスクが高い。               |
| 公開ページ                          | 誤公開リスクが高い。                           |
| ChatGPT API 自動要約               | 本文送信、ログ、保持、利用者同意、コスト管理が必要。           |
| 詳細な監査ログ UI                     | 監査イベントの保存は MVP から行うが、UI は後回し可。       |
| エンタープライズ SSO                   | 初期ユーザーが個人開発者なら過剰。                    |
| per-user encryption key / BYOK | 将来の企業向け。MVP ではインフラ暗号化と secret 管理を優先。 |
| IP 制限・デバイス管理                   | 企業利用段階で検討。                           |

ただし、後回しにするのは UI・高度機能であり、DB 設計上は将来追加できる余地を残す。特に `workspaceId`、`auth_accounts`、`audit_events`、`external_integrations` は MVP の時点で拡張可能な形にしておく。

# データ設計への影響

## 5. データアクセス制御方針

基本方針は **private-by-default / owner-based access control / deny-by-default**。

MVP の最小データモデルは次の方針にする。

| テーブル                    | 主なセキュリティ項目                                                                                                     |
| ----------------------- | -------------------------------------------------------------------------------------------------------------- |
| `users`                 | `id`, `email`, `emailVerifiedAt`, `createdAt`, `disabledAt`                                                    |
| `auth_accounts`         | `userId`, `provider`, `providerAccountId`, `email`, `createdAt`。OAuth token は MVP では保存しない。                     |
| `sessions`              | `userId`, `sessionHash`, `expiresAt`, `rotatedAt`, `createdIpHash`, `userAgentHash`                            |
| `workspaces`            | MVP では 1 user = 1 personal workspace。将来チーム対応。                                                                  |
| `workspace_members`     | MVP では本人のみ。将来 `role` を追加。                                                                                      |
| `projects`              | `ownerUserId` または `workspaceId`, `visibility='private'`, `archivedAt`, `deletedAt`                             |
| `repositories`          | `ownerUserId` / `workspaceId`, `projectId`, `url`, `provider`, `externalRepoId nullable`                       |
| `items`                 | 課題、進捗、バグ、実装予定、将来構想などを統一的に持つ場合も `ownerUserId` / `workspaceId` 必須                                                |
| `notes`                 | ChatGPT 壁打ちメモ、自由記述。本文はログに出さない。                                                                                 |
| `tags`                  | `ownerUserId` / `workspaceId` を持たせ、他ユーザーのタグと混ざらないようにする。                                                        |
| `audit_events`          | `actorUserId`, `action`, `resourceType`, `resourceId`, `occurredAt`, `requestId`, `metadataRedacted`。本文は保存しない。 |
| `backup_exports`        | `ownerUserId`, `fileKey`, `expiresAt`, `downloadedAt`, `sha256`, `status`                                      |
| `external_integrations` | 将来用。`provider`, `installationId`, `scopes`, `tokenEncrypted nullable`, `revokedAt`                             |

すべてのリソース取得は、単に `id` で引かず、必ず `id + ownerUserId` または `id + workspaceId + membership` で引く。URL に UUID を使っても、それはアクセス制御の代替にはならない。ID 推測を防ぐだけでなく、推測されても権限チェックで拒否する。

削除・アーカイブ・復元は次のように分ける。

| 操作          | セキュリティ上の意味                                    |
| ----------- | --------------------------------------------- |
| アーカイブ       | 表示上の整理。機密性は変わらない。復元可。                         |
| ソフト削除       | `deletedAt` を付け、通常一覧から除外。所有者だけ復元可。            |
| 完全削除        | 復元不可。関連メモ・添付・バックアップ反映方針を明確にする。                |
| バックアップからの復元 | 復元先 userId / workspaceId を再付与し、他ユーザー領域へ復元しない。 |

# 画面設計への影響

ログイン画面は、DADS のフォームルールに従い、メールアドレス欄には明示的なラベル、サポートテキスト、具体的なエラー文を置く。プレースホルダーだけで説明しない。ログインボタンは基本 1 画面 1 つのプライマリー CTA にする。未入力時にボタンを disabled にせず、押された後に「メールアドレスを入力してください」のように具体的に案内する。これは DADS の「disabled を安易に使わない」方針に合わせる。

GitHub ログインを副導線として置く場合は、ボタン付近に「リポジトリ権限は要求しません」「ログイン確認にのみ使います」と明記する。GitHub OAuth は scope の見え方が利用者の信頼に直結するため、重要情報をアコーディオン内に隠さない。

AI 連携を将来入れる場合、送信前画面に「送信される対象」を表示する。例えば「このプロジェクトのタイトル、選択したメモ本文、選択した課題だけを送信します」と明記する。送信されないもの、たとえば GitHub token、API key、非選択メモも併記する。DADS のノティフィケーションバナー方針に従い、AI 送信やエクスポートなど重要な注意はページ上部または対象セクション内に表示する。

削除確認は、MVP ではモーダルよりも専用確認ページまたはインライン確認を優先する。DADS ではモーダルダイアログの詳細ガイドラインが未整備とされているため、モーダルを使う場合は NextPatch 独自の「暫定案」として扱い、フォーカストラップ、Esc、キーボード操作、読み上げ、閉じる操作を別途検証する。

3 操作以内の導線としては、ログイン後に「今日やること」へ自動遷移し、プロジェクト詳細、設定、エクスポート、アーカイブ/削除に 3 操作以内で到達できる構成にする。ただし、削除・外部送信・エクスポートは誤操作防止の確認を挟む。

# API・機能設計への影響

API は次の順序で処理する。

1. セッション確認。
2. CSRF 検証。ただし GET など安全メソッドは対象外。
3. 入力スキーマ検証。
4. 対象リソースを `id + ownerUserId` または `workspaceId + membership` で取得。
5. 操作権限を判定。
6. 実行。
7. 監査イベントを本文なしで記録。
8. レスポンスには必要最小限の情報だけを返す。

API 例として、`PATCH /api/projects/:projectId/items/:itemId` では、`itemId` だけで更新してはいけない。必ず `project.ownerUserId === session.user.id` と `item.projectId === project.id` を確認する。クライアントから送られた `ownerUserId` は無視する。

## 6. ログ・監査・エラー収集方針

ログに残すものは次に限定する。

| 残す                        | 残さない                 |
| ------------------------- | -------------------- |
| request id                | メモ本文                 |
| user id                   | ChatGPT 壁打ち全文        |
| resource type             | GitHub token         |
| resource id               | OpenAI API key       |
| action                    | セッション ID 生値          |
| status code               | DB 接続文字列             |
| error code                | パスワード、ワンタイムコード       |
| 発生ファイル・関数                 | private repo のファイル本文 |
| redacted stack trace      | Authorization ヘッダー   |
| IP hash / user agent hash | Cookie 生値            |

OWASP Logging Cheat Sheet は、access tokens、session identification values、authentication passwords、database connection strings、encryption keys、commercially-sensitive information などをログへ直接記録せず、削除・マスク・サニタイズ・ハッシュ化・暗号化すべき対象としている。NextPatch の本文メモは commercial-sensitive information に該当し得るため、エラーログに含めない。([OWASP Cheat Sheet Series][11])

監査イベントは、ユーザー本人の安全確認と将来のチーム機能に備えて MVP から保存する。ただし MVP では監査ログ UI は簡易でよい。保存するイベントは、ログイン成功/失敗、ログアウト、プロジェクト作成、削除、復元、エクスポート、外部連携追加/解除、AI 送信実行、共有設定変更に限定する。

## 7. 将来 GitHub / ChatGPT 連携時の注意点

GitHub 連携では、最初に「ユーザーが手入力したリポジトリ URL を保存するだけ」に留める。private repo の一覧取得、Issue 取り込み、PR 取り込み、README 取得などは GitHub App 導入後に行う。OAuth App を使う場合でも、GitHub Docs が示す通り scope は必要な種類を指定し、ユーザーが許可 scope を変更する可能性もハンドリングする。([GitHub Docs][6])

ChatGPT API 連携では、送信対象をユーザーが明示選択する。プロジェクト全体を丸ごと送る、全メモを自動送信する、エラー時に prompt をログへ残す、といった設計は禁止する。OpenAI API ではデータは opt-in しない限りモデル訓練に使われないが、abuse monitoring logs に customer content が含まれ得て、通常最大 30 日保持されるため、NextPatch 側でも送信前の同意と最小化が必要である。([OpenAI開発者][3])

OpenAI API を使う場合、API key はクライアントへ出さず、サーバー側の secret として管理する。OpenAI の production best practices でも、API keys をコードや公開リポジトリに露出せず、環境変数や secret management service を使うことが示されている。([OpenAI開発者][12])

Responses API などで保存設定がある場合は、用途に応じて `store=false` を明示する方針にする。OpenAI Docs は Responses API の application state がデフォルトまたは `store=true` で 30 日保持されること、Zero Data Retention では `store` が `false` 扱いになることを説明している。([OpenAI開発者][3])

# リスクと対策

| リスク                   | 影響                    | 対策                                        |
| --------------------- | --------------------- | ----------------------------------------- |
| 認証なし MVP で機密メモが保存される  | 未公開アイデア・脆弱性情報の漏えい     | MVP から認証必須                                |
| 後付け `userId`          | データ移行ミス、権限漏れ          | 初期 DB から `ownerUserId` / `workspaceId` 必須 |
| GitHub OAuth scope 過剰 | private repo への過剰アクセス | MVP では repo scope 不使用。将来は GitHub App      |
| GitHub token 保存       | token 漏えい時の被害拡大       | MVP では保存しない。将来は暗号化・短期化・失効導線               |
| ChatGPT API に全文送信     | 外部サービスへの機密送信          | 明示操作、送信対象プレビュー、最小化、redaction              |
| エラーログ本文混入             | ログ閲覧者・外部監視サービスへの漏えい   | ログ redaction、body logging 禁止              |
| XSS                   | メモ本文からセッション悪用・画面改ざん   | Markdown サニタイズ、出力エスケープ、CSP、HttpOnly       |
| CSRF                  | ログイン中ユーザーの意図しない更新・削除  | CSRF token、SameSite、Origin/Referer 検証     |
| IDOR                  | 他ユーザーのプロジェクト閲覧・更新     | `id + ownerUserId` で DB 取得、権限テスト          |
| バックアップ漏えい             | 全データ流出                | 暗号化、署名付き短期 URL、アクセスログ、期限削除                |
| 誤削除                   | 重要メモ喪失                | ソフト削除、復元、完全削除前確認                          |
| 共有リンク漏えい              | URL を知る第三者が閲覧         | MVP では共有リンクなし                             |
| 公開ページ誤公開              | ロードマップ・バグ情報の露出        | MVP では公開ページなし                             |
| メールログイン悪用             | メール爆撃、コード総当たり         | レート制限、単回使用、短期期限、同一メッセージ                   |

# 未決定事項

次は統括側で追加決定が必要。

| 未決定事項                        | 推奨する決め方                                                          |
| ---------------------------- | ---------------------------------------------------------------- |
| 認証基盤を自前にするか、外部 Auth サービスにするか | MVP では自前実装を避け、成熟した Auth ライブラリまたはサービスを採用する方針が安全。                  |
| メールリンクか、6 桁コードか              | メールクライアントからのリンククリック UX を重視するならリンク。コピペしやすさを重視するならコード。どちらも単回・短期期限。 |
| セッション有効期限                    | 個人開発者向けなら 7〜30 日程度。高機密運用なら短め。                                    |
| バックアップ保持期間                   | 自動バックアップ、ユーザーエクスポート、削除済みデータで保持期間を分ける。                            |
| 完全削除までの猶予                    | 7 日、14 日、30 日のいずれか。MVP は 30 日が安全寄り。                              |
| ChatGPT API 連携を MVP に含めるか    | 含めない推奨。含めるなら、AI 送信確認 UI とログ非保存設計を MVP 必須に格上げ。                    |
| GitHub ログインを MVP に含めるか       | 含めない推奨。マーケティング上必要なら副ログインとして token 非保存で実装。                        |
| 添付ファイルを MVP に含めるか            | 含めるとアップロード検証・マルウェア・容量制限が必要になるため後回し推奨。                            |
| 利用規約・プライバシーポリシー              | MVP 公開前に必須。特に AI 連携、ログ、バックアップ、削除、外部送信を記載。                        |

# 統括担当へ渡す要点

統括担当には、次の方針で渡す。

MVP は **認証必須・private-by-default・メールログイン主認証・GitHub token 非保存・共有/公開なし** で進める。DB は初期から複数ユーザー前提にし、全リソースに `ownerUserId` または `workspaceId` を持たせる。UI 上のチーム機能は後回しでよいが、後から追加できるよう `workspaces` と `workspace_members` の概念は最初から入れる。

GitHub は MVP ではリポジトリ URL の手入力保存までに留める。GitHub ログインを入れる場合も、ログイン補助に限定し、scope は最小、token は保存しない。将来の private repo 連携は GitHub App で再設計する。

ChatGPT API 連携は MVP では後回しが望ましい。入れる場合は、送信対象を明示し、本文をログに残さず、API key をクライアントへ出さず、OpenAI 側のデータ保持設定と `store=false` 方針を確認してから実装する。

## 実装時のセキュリティチェックリスト

### 認証・セッション

* [ ] 未ログインではアプリ本体へアクセスできない。
* [ ] メールログイン token は安全な乱数、単回使用、短期期限。
* [ ] ログイン試行・メール送信にレート制限がある。
* [ ] セッション Cookie は `Secure`、`HttpOnly`、`SameSite=Lax` 以上。
* [ ] 本番は HTTPS 強制。
* [ ] ログアウト時にセッションが無効化される。
* [ ] セッション更新・失効処理がある。
* [ ] 認証エラー文で「そのメールが登録済みか」を漏らさない。

### データ分離・権限

* [ ] 全リソースに `ownerUserId` または `workspaceId` がある。
* [ ] API は `id` 単体でリソースを取得しない。
* [ ] クライアントから渡された `ownerUserId` を信用しない。
* [ ] すべての読み取り・更新・削除でサーバー側権限チェックを行う。
* [ ] 権限なしは deny-by-default。
* [ ] IDOR テストがある。
* [ ] 将来の workspace / role 追加を壊さない設計になっている。

### CSRF / XSS / 入力検証

* [ ] POST / PATCH / DELETE に CSRF 対策がある。
* [ ] Origin / Referer 検証を補助的に使う。
* [ ] すべての入力にサーバー側スキーマ検証がある。
* [ ] GitHub URL は許可形式に正規化・検証する。
* [ ] 本文・メモは表示時にエスケープする。
* [ ] Markdown を使う場合、raw HTML を無効化またはサニタイズする。
* [ ] CSP を設定する。
* [ ] `dangerouslySetInnerHTML` 相当の使用箇所をレビューする。

### GitHub 連携

* [ ] MVP では GitHub token を保存しない。
* [ ] GitHub OAuth を入れる場合、`state` と PKCE を使う。
* [ ] OAuth scope は明示的に最小指定する。
* [ ] `repo` scope を MVP で要求しない。
* [ ] provider account id と NextPatch user id の紐付けを一意にする。
* [ ] 将来の private repo 連携は GitHub App 前提で設計メモを残す。

### ChatGPT / OpenAI 連携

* [ ] MVP で AI 連携を入れるか明確に決めている。
* [ ] 送信対象をユーザーが明示選択できる。
* [ ] 送信前に「送られる内容」と「送られない内容」を表示する。
* [ ] prompt / response 本文をアプリログに残さない。
* [ ] OpenAI API key をクライアントへ露出しない。
* [ ] `store=false` など保持設定を確認している。
* [ ] 外部送信についてプライバシーポリシーに記載している。

### ログ・監査・エラー収集

* [ ] request body を標準ログに出していない。
* [ ] メモ本文、ChatGPT 本文、GitHub token、API key、Cookie、Authorization ヘッダーを redact している。
* [ ] エラー収集サービスへ送る情報を制限している。
* [ ] 監査イベントに本文を含めていない。
* [ ] 削除、復元、エクスポート、外部連携、AI 送信を監査イベント化している。
* [ ] ログ閲覧権限を制限している。

### バックアップ・エクスポート・削除

* [ ] バックアップは暗号化されている。
* [ ] バックアップ復元時に `ownerUserId` / `workspaceId` を再検証する。
* [ ] ユーザーエクスポートは短期有効 URL にする。
* [ ] エクスポートファイル名に機密名を含めない。
* [ ] 削除はソフト削除から始める。
* [ ] 復元は所有者だけができる。
* [ ] 完全削除の猶予期間と対象範囲が決まっている。
* [ ] バックアップ内の削除済みデータの扱いがプライバシーポリシーと一致している。

### UI

* [ ] ログイン画面にラベル、サポートテキスト、具体的エラーがある。
* [ ] プレースホルダーだけで入力条件を説明していない。
* [ ] ボタン disabled に頼らず、押下後に具体的なエラーを出す。
* [ ] 外部送信、削除、エクスポートは重要情報を隠さず表示している。
* [ ] モーダルを使う場合は DADS 未整備項目として暫定仕様レビューを行う。
* [ ] 主要操作は 3 操作以内で到達できる。
* [ ] 共有・公開機能は MVP UI に出さない、または「未対応」と明確に扱う。
