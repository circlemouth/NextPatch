# 担当領域

UIデザインシステム・DADS適用。

NextPatch の UI を、DADS参照ルールに基づいて「開発管理アプリ向けの初期スタイルガイド」として整理する。実装コードではなく、開発着手前に決めるべき設計判断を示す。

# 前提理解

NextPatch は、GitHub リポジトリや開発中アプリごとに、課題、進捗、バグ、実装予定、将来構想、採用技術、参考サービス、アイデア、ChatGPT との壁打ちメモを整理し、「今やるべきこと」をすぐ把握するための開発管理アプリである。

本回答で直接参照できた資料は、アップロード済みの `dads_app_ui_design_rules_20260411.md` と、DADS公式ページである。NextPatch 要件定義書・検討メモの詳細本文は本セッション上では確認できないため、NextPatch の機能像はユーザー提示文を前提にする。DADS参照ルールは「完成済みの見た目集」ではなく、個別サービスのスタイルガイドを作るための基準として扱う。

DADS公式情報は、2026年4月20日に確認した。DADS公式では、本文・UI文字は原則 16 CSS px 以上、本文行高は少なくとも 1.5 倍を推奨し、管理画面では用途に応じて 120〜130% 程度の Dense な行高もあり得るとしている。([デジタル庁デザインシステムβ版][1]) また、ブレークポイントは 768px 未満をモバイル・タブレット、768px 以上をデスクトップとする例が示されている。([デジタル庁デザインシステムβ版][2])

DADS公式では、プレースホルダーを原則使わず、入力条件や例はサポートテキストに書くこと、`disabled` フィールドを使わないことが明記されている。([デジタル庁デザインシステムβ版][3]) ボタンについても、作業完了前に無効化するのではなく、押された後に必要な作業をエラーで案内する方針が示されている。([デジタル庁デザインシステムβ版][4])

DADS公式では、ボトムナビゲーションは deprecated であり、アクセシビリティまたはユーザビリティの観点から現在は使用が推奨されない。NextPatch でも、モバイルでボトムナビゲーションは使わない方針とする。([デジタル庁デザインシステムβ版][5])

# 決定すべき論点

| 論点          | NextPatchで決めるべきこと      | 初期判断                             |
| ----------- | ---------------------- | -------------------------------- |
| カラースキーム     | ブランド色、機能色、状態色、リンク色     | 青系をプライマリー、ニュートラルを基調、状態色は意味に合わせる  |
| 文字サイズ・見出し階層 | H1〜H4、本文、メタ情報、テーブル密度   | 16px 未満を原則使わず、管理画面でも可読性優先        |
| 余白スケール      | コンポーネント内、カード間、セクション間   | 8px 基準の 5 段階を採用                  |
| 角丸          | カード、ボタン、入力欄、チップの丸み     | 0 / 4 / 8 / 12 / full の段階制       |
| エレベーション     | カード、ドロワー、モーダル、通知の高さ    | 原則フラット、重なりが必要な時だけ使う              |
| アイコン        | 状態、外部リンク、ナビゲーション、操作補助  | 意味補助に限定し、色だけで状態を伝えない             |
| ボタン優先度      | プライマリー、セカンダリー、ターシャリー   | プライマリーは 1 画面 1 個を基本              |
| ボタン配置       | PC / モバイルでの並び順         | PCは進む操作を右、戻る・中止を左。モバイルは縦積み       |
| フォーム設計      | ラベル、サポート、エラー、入力制限      | ラベル必須、プレースホルダー非使用、disabled 原則回避  |
| エラー表示       | フィールドエラー、画面エラー、同期エラー   | 静的に表示し、原因と修正方法を書く                |
| リンク表示       | GitHubリンク、参考サービス、外部リンク | 下線あり。外部リンクはアイコン併用                |
| ブレークポイント    | PC / スマホの切り替え          | 768px 未満 / 以上で切り替え               |
| ボトムナビ       | モバイル主要ナビとして使うか         | 使わない                             |
| 重要情報の表示     | 今やるべきこと、ブロッカー、期限、同期失敗  | 折りたたまず初期表示                       |
| アコーディオン     | 補足情報の折りたたみ             | 重要情報には使わない                       |
| タブ          | リポジトリ詳細の切り替え           | DADS未整備のため、MVPでは原則避ける。使う場合は暫定案   |
| モーダル        | 作成・編集・削除確認             | DADS未整備のため、複雑な入力には使わない。削除確認のみ暫定案 |

| レイアウト        | ダッシュボード全面カード     | 左ナビ + 主要作業エリア + 補助カラム | カンバン中心           | 案B                      |
| 理由           | 情報が散る            | 「今やるべきこと」と詳細を両立できる    | MVPでは実装・情報設計が重い  | PCは 2 カラム中心、モバイルは 1 カラム |

| リポジトリ詳細の情報切替 | タブ               | 見出し + セクション縦積み        | アコーディオン          | MVPは案B                  |

# 推奨案

NextPatch の UI は、**DADS準拠のアクセシブルな管理画面スタイル**として設計する。

最優先する画面構造は、次の通り。

1. 画面上部またはメイン先頭に「今やるべきこと」を常時表示する。
2. ブロッカー、期限超過、GitHub同期失敗、未分類メモなど、判断に必要な重要情報は折りたたまない。
3. PCでは左サイドナビ + メイン領域 + 補助カラム、モバイルでは 1 カラム + モバイルメニューに切り替える。
4. ボトムナビゲーションは使わない。
5. フォームはラベル必須、プレースホルダー非使用、disabled 原則回避とする。
6. プライマリーボタンは原則 1 画面 1 個に絞る。
7. タブ、モーダル、テーブルコントロールなど DADS 側で詳細ガイドライン未整備の部品は、MVPでは極力避ける。使う場合は「暫定案」として仕様書に明記する。

カラースキームは、**NextPatch Blue** をプライマリーにする。理由は、開発管理アプリとしての信頼感、GitHub連携・情報整理との親和性、CTAの視認性を両立しやすいためである。アクセント色は多用せず、状態色は成功・エラー・警告・情報の意味に限定する。DADS公式では、テキストは背景とのコントラスト比 4.5:1 以上、非テキストは 3:1 以上が必要であり、状態色も意味を保ったまま明度・彩度を調整できるとされている。([デジタル庁デザインシステムβ版][6])

# MVPに含めるべき内容

MVPでは、見た目の完成度よりも「迷わず把握できる」「入力で詰まらない」「重要情報が隠れない」ことを優先する。

含めるべき内容は以下。

| 項目      | MVPで含める内容                                          |
| ------- | -------------------------------------------------- |
| カラートークン | プライマリー、リンク、背景、テキスト、枠線、成功、エラー、警告、情報                 |
| タイポグラフィ | H1〜H4、本文、メタ情報、ボタン、テーブル用 Dense 表示                   |
| 余白      | 8px 基準の余白スケール                                      |
| ボタン     | Primary / Secondary / Tertiary / Destructive の使い分け |
| フォーム    | ラベル、要否ラベル、サポートテキスト、エラーテキスト                         |
| 通知      | ページ上部のノティフィケーションバナー、セクション内通知                       |
| ナビゲーション | PC左サイドナビ、モバイルメニュー、パンくず                             |
| ダッシュボード | 「今やるべきこと」カード、期限超過、ブロッカー、未整理メモ                      |
| リポジトリ一覧 | PCはテーブル、モバイルはカード                                   |
| 課題・メモ一覧 | フィルター、検索、状態ラベル、優先度、次アクション                          |
| 詳細画面    | 重要情報、次アクション、関連メモ、GitHubリンクを初期表示                    |
| エラー表示   | 入力エラー、保存失敗、GitHub同期失敗、認証切れ                         |
| レスポンシブ  | 768px 未満 / 以上のレイアウト切り替え                            |

通知は、DADS公式の通り、通常はページ上部のファーストビューに置く。特定セクションに関する通知はそのセクション内に置く。固定・追従表示にする場合は閉じるパーツを付ける。([デジタル庁デザインシステムβ版][7])

# MVPでは後回しにする内容

| 項目             | 後回しにする理由                              |
| -------------- | ------------------------------------- |
| ボトムナビゲーション     | DADSで非推奨。NextPatchでも使わない              |
| ダークモード         | カラー検証・コントラスト検証が別途必要                   |
| 複雑なタブUI        | DADSで詳細ガイドライン未整備。重要情報を隠しやすい           |
| 複雑なモーダル編集      | DADSで詳細ガイドライン未整備。長文入力・エラー表示に不向き       |
| 高機能テーブルコントロール  | DADSで未整備扱い。MVPは基本テーブルに留める             |
| ドラッグ＆ドロップのカンバン | 操作性・アクセシビリティ検証が重い                     |
| 高度なグラフ・分析画面    | MVPの「今やるべきこと把握」から優先度が下がる              |
| カスタム日付ピッカー     | DADSの扱いに沿い、まずは直接入力 + 必要最小限にする         |
| リッチテキストエディタ    | ChatGPTメモの保存には便利だが、MVPではプレーンテキスト中心で十分 |

# データ設計への影響

UI方針から、データ設計では「画面に常時出すべき判断材料」を明確に持つ必要がある。

| データ項目                     | 必要理由                           |
| ------------------------- | ------------------------------ |
| `repository_id`           | リポジトリ単位で整理するため                 |
| `item_type`               | 課題、バグ、実装予定、メモ、アイデア、参考サービスなどの区別 |
| `title`                   | 一覧・カードで最初に読む情報                 |
| `summary`                 | 詳細を開かずに内容を把握するため               |
| `next_action`             | 「今やるべきこと」の中心項目                 |
| `status`                  | 未着手、進行中、確認待ち、完了、保留など           |
| `priority`                | 表示順と強調に使う                      |
| `due_at`                  | 期限超過・近日対応の判定に使う                |
| `is_blocked`              | ブロッカーを初期表示するため                 |
| `blocker_reason`          | なぜ止まっているかを隠さず表示するため            |
| `source_type`             | GitHub、ChatGPT、手入力、Web参考など     |
| `source_url`              | GitHub Issue、PR、参考サービスへのリンク    |
| `tags`                    | 絞り込み・分類用                       |
| `updated_at`              | 古い情報の見落とし防止                    |
| `github_sync_state`       | 同期成功、失敗、認証切れ、未連携               |
| `severity`                | 通知やエラーの意味を色以外でも扱うため            |
| `dismissed_notifications` | 閉じた通知の再表示制御に必要                 |
| `display_order`           | ダッシュボードの表示順調整                  |
| `is_supplemental`         | 補足情報だけを折りたたむ判断に使う              |

重要なのは、UI上の色名をデータに直接保存しないこと。データには `status`、`severity`、`priority` のような意味を保存し、表示側で色・アイコン・ラベルへ変換する。これにより、色だけに依存しない UI にできる。

# 画面設計への影響

PC、モバイルともに、目的到達を原則 3 操作以内にする。

## PC / 768px以上

| 画面      | 推奨構成                                        |
| ------- | ------------------------------------------- |
| ダッシュボード | 左ナビ、H1、リポジトリ切替、「今やるべきこと」、ブロッカー、期限超過、最近のメモ   |
| リポジトリ詳細 | 上部に概要、中央に課題・メモ一覧、右カラムに同期状態・技術メモ・関連リンク       |
| 課題詳細    | タイトル、状態、次アクション、期限、ブロッカー、本文、関連GitHubリンクを初期表示 |
| 作成・編集   | 専用ページ。入力欄は縦積み。保存ボタンは右下またはフォーム末尾             |
| 一覧      | PCではテーブル。状態、優先度、期限、次アクションは列として見せる           |

## モバイル / 768px未満

| 画面      | 推奨構成                                |
| ------- | ----------------------------------- |
| ダッシュボード | 1カラム。最上部に「今やるべきこと」。次にブロッカー、期限、未整理メモ |
| ナビゲーション | ハンバーガーメニュー + ドロワー。ボトムナビは使わない        |
| 一覧      | テーブルではなくカードリスト。タイトル、状態、次アクション、期限を表示 |
| 詳細      | 重要情報を先頭に集約。補足メモ・履歴だけ下部に置く           |
| フォーム    | ラベル、サポート、入力欄、エラーを縦積み。ボタンは縦積み        |

DADS公式では、グリッドレイアウトにおいてカラム、ガター、マージンを基準にし、ガターは原則として本文文字サイズの 2 倍を確保するとされている。PCでは左サイドナビの右端からグリッドを組み、モバイルでは 1 カラムへ落とす。([デジタル庁デザインシステムβ版][2])

# API・機能設計への影響

UI方針により、APIは単なるCRUDだけでなく、画面で重要情報をすぐ出せる形にする必要がある。

| API / 機能  | 必要な設計                                                                |
| --------- | -------------------------------------------------------------------- |
| ダッシュボード取得 | `next_actions`、`blocked_items`、`overdue_items`、`sync_errors` をまとめて返す |
| 一覧取得      | status、priority、due_at、repository、tag で絞り込み可能にする                     |
| 詳細取得      | 重要情報と補足情報を分けて返す                                                      |
| 保存API     | `disabled` 前提にせず、未入力でも送信後にバリデーションエラーを返す                              |
| エラー形式     | フィールド単位の `field`、`code`、`message` を返す                                |
| 通知API     | severity、message、action_label、action_url、dismissible を返す             |
| GitHub同期  | 同期状態、最終同期時刻、失敗理由、再接続導線を返す                                            |
| リンク管理     | 外部リンク、GitHubリンク、参考資料リンクの種別を持つ                                        |
| モバイル表示    | 一覧APIはカード表示用の要約フィールドを返す                                              |
| アクセシビリティ  | 色名ではなく意味値を返し、UI側でラベル・アイコン・色に変換する                                     |

エラー文言は、たとえば `＊タイトルを入力してください。` のように、何が悪いか、どう直すかを具体的に返す。DADS参照ルールに従い、エラーテキストは静的に表示し、読み上げ強制のための `aria-live` や `role="alert"` は原則使わない。

# リスクと対策

| リスク            | 内容                        | 対策                                 |
| -------------- | ------------------------- | ---------------------------------- |
| 情報過多           | 課題、バグ、メモ、将来構想が同じ画面に並ぶ     | 「今やるべきこと」「ブロッカー」「期限」の優先表示に絞る       |
| 重要情報の隠蔽        | タブやアコーディオンで重要情報が見えなくなる    | 重要情報は初期表示。折りたたむのは補足・履歴のみ           |
| 色依存            | ステータスを色だけで伝えてしまう          | ラベル、アイコン、テキストを併用                   |
| disabledによる停止  | 保存ボタンが押せず理由が分からない         | ボタンは有効のまま、押下後にエラー案内                |
| プレースホルダー依存     | 入力中に条件が消える                | サポートテキストに条件・例を書く                   |
| モバイルの表崩れ       | テーブルが横にはみ出す               | モバイルはカードリスト化                       |
| DADS未整備部品の独自実装 | タブ、モーダル、テーブルコントロールで仕様がぶれる | MVPでは避ける。使う場合は暫定案として記録             |
| 通知の乱発          | すべてが警告に見える                | severity と表示条件を整理し、重要度の低いものは一覧内に出す |
| 長いリポジトリ名       | カードやテーブルで崩れる              | 折り返し、2行表示、詳細では全文表示                 |
| 固定UIが内容を隠す     | sticky footer や固定通知が本文を覆う | 原則固定しない。固定する場合は閉じる操作と余白確保          |

DADS公式では、エレベーションを持つ要素は背景との 3:1 以上のコントラストを確保し、ドロップシャドウだけで境界を担保しないように求めている。NextPatch でも、シャドウは控えめにし、必要な場合はボーダーを併用する。([デジタル庁デザインシステムβ版][8])

# 未決定事項

| 未決定事項            | 判断が必要な条件                         |
| ---------------- | -------------------------------- |
| 正式ロゴ・ブランド色       | NextPatch のブランド方針が確定した時点で再調整     |
| フォントファミリー        | Webフォントを使うか、OS標準フォントにするか         |
| ダークモード           | MVP後に必要性とコントラスト検証を行う             |
| タブの採用            | リポジトリ詳細で縦積みが長くなりすぎる場合のみ暫定採用を検討   |
| モーダルの採用範囲        | 削除確認など短い確認だけに限定するか               |
| 右ドロワー編集          | PC専用で採用するか、専用ページに統一するか           |
| テーブルのソート・一括操作    | MVPで必要な操作量を見て判断                  |
| GitHub同期エラーの通知粒度 | リポジトリ単位、アイテム単位、全体通知のどれにするか       |
| ChatGPTメモの構造化    | プレーンテキスト、Markdown、ブロック化のどこまで入れるか |
| 参考サービスの表示        | リンク集、カード、比較表のどれを標準にするか           |

# 統括担当へ渡す要点

NextPatch の UI は、DADSを固定デザインとして流用するのではなく、DADSの原則に基づく独自スタイルガイドとして起こす。

最重要方針は以下。

| 項目           | 統括判断                            |
| ------------ | ------------------------------- |
| ラベル          | 必ず表示する                          |
| プレースホルダー     | 原則使わない                          |
| disabled ボタン | 原則避ける                           |
| 重要情報         | 折りたたまない                         |
| プライマリーボタン    | 1画面1個を基本                        |
| ボトムナビ        | モバイルでも使わない                      |
| ブレークポイント     | 768px 未満 / 以上で切り替え              |
| タブ           | DADS未整備のためMVPでは避ける。使う場合は暫定案     |
| モーダル         | DADS未整備のため複雑な入力には使わない。削除確認のみ暫定案 |
| アコーディオン      | 補足・履歴だけ。重要情報には使わない              |
| PCレイアウト      | 左ナビ + メイン + 補助カラム               |
| モバイルレイアウト    | 1カラム + ハンバーガーメニュー               |
| 一覧           | PCはテーブル、モバイルはカード                |
| エラー          | 上部サマリー + フィールド直下の具体的エラー         |
